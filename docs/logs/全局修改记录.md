# 全局修改记录

## 2026-05-14

本次围绕 `PiecewiseLinearFunction` 的右端定义域约束和 `mergeMinimum` 输入契约做了一次整理。主要改动是把 `mergeMinimum/mergeMinimum2` 接到统一的 `Utility.debugCheckPWLFMergeContract(...)`，并在 `Utility` 中补充说明：`tail.end` 表示 segment 链表真实右端，`domainEnd` 表示对象元数据，二者不一致时通常说明某个操作实际把函数裁短了。该检查由 `Configure.debugPWLFDomainCheck` 控制，默认关闭，不改变现有求解流程。

同时补充了两类记录：分段函数细节写入 `docs/plans/2026-05-13-PiecewiseLinearFunction测试分析记录.md`；全局层面从本文件开始记录每次实质性修改。`AGENTS.md` 也补充了全局记录要求，后续代码、文档、测试、分析和文件整理完成后，都要在这里写一条简短记录。

验证上，已重新运行 PWLF 专项测试：`merge-find-contract` 模式结果为 `passed=5, warnings=0, failed=0`，完整测试结果为 `passed=10, warnings=3, failed=16`。这说明本次调试检查接入没有改变已知测试格局；完整测试里剩余失败仍是既有风险，主要集中在 `minimizeSuffixInPlace`、`dominates` 和 partial dominance 的严格语义上，不能直接作为 BPC pricing 的最终实现依据。

同日又对文档结构做了整理：原 `2026-05-13-PiecewiseLinearFunction测试分析记录.md` 和 `2026-05-14-BPC定价中分段函数使用约束.md` 内容高度重叠，因此合并为一份专题文档 `docs/plans/2026-05-13-PiecewiseLinearFunction测试分析记录.md`，并删除 5-14 那份重复文档。合并时同时清理了原文档中残留的编码异常段落，使后续只维护一份分段函数与 BPC 定价函数约束记录。

同日补充了启发式 forward 函数初始化处的注释：序列首任务最早完工时间应为 `s[0][job] + p[job]`，其中 `s[0][job]` 表示从虚拟起点 0 到任务 `job` 的首任务 setup，`p[job]` 是加工时间。因此初始化 `setDomain(...)` 时裁掉更早时间点，是为了避免选择物理不可行的 completion。

同日补充了 `tail.end` 与 `domainEnd` 关系的专题记录。准确结论是：如果输入函数都满足实际右端和元数据右端同为 `T`，则 `add`、有效输入下的 `mergeMinimum/mergeMinimum2`、当前版本的 `normalize` 和正向 `shiftX` 不会主动破坏这个关系；真正会主动制造 `tail.end != domainEnd` 的，是 `setDomain(start,end)` 且 `end < 原 domainEnd`，因为它会裁短真实链表右端后再恢复原元数据定义域。

同日进一步明确了该 debug 判断的意义：它不是为了检查 `domainEnd` 字段是否变化，而是为了识别函数真实 segment 链表右端是否已经不再到全局 `T`。在 forward pricing 语义下，若 `tail.end != domainEnd`，说明实际定义域被裁短，后续仍按 `[a,T]` 做 `add/merge/dominance` 就可能出错，因此该检查能够定位这类右端定义域风险。

同日更正了对 `updateDominatedIntervals` 和 partial dominance 的判断。当前“被支配区间打成 `big_M`，再允许 `normalize()/minimizePrefixInPlace()` 做 forward 闭包”的做法，在 forward frontier 语义下可以解释通，不应再作为第一阶段主要问题反复分析。真正需要保证的是 pricing 前把 `Utility.curUpperBound` 重置为 `Utility.big_M`，避免启发式上界截断 reduced cost 函数；同时在 dominance 判定层严格控制哪些 label 有资格支配。

同日分析了 `minimizeSuffixInPlace()` 和 backward 函数定义域。结论是 backward 构造阶段基本不提前裁窄 domain，末任务直接复制原惩罚函数，递推时用负向 `shiftX`、`add` 和 `minimizeSuffixInPlace()`；当前主要风险在 suffix 最小化实现会漏掉应保留的尾部常数最小值段，而不是 backward 定义域设置本身。详细分析已补充到 `docs/plans/2026-05-13-PiecewiseLinearFunction测试分析记录.md`。

同日继续分析了 `minimizeSuffixInPlace()` 中边界判断的原始动机。该判断应是从 prefix 最小化迁移过来的保护逻辑，用来避免在函数边界插入由 `Utility.curUpperBound` 形成的人工上界段；但 suffix 版本里右侧最小值可能来自真实尾部函数，直接套用 `lastT != tail.end` 会误跳过真实尾部常数最小值段。该判断的动机和问题已补充到分段函数专题文档。
