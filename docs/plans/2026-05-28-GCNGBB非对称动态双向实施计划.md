# GCNGBB 非对称动态双向实施计划

本文记录基于 `docs/literature/dynamic_halfway_soft_tw.pdf` 和当前讨论形成的实现方案。目标不是修改现有正式 `GCNGBBStyleBidirectional`，而是在已有 full-domain 复制版基础上再复制一份实验版本，用动态 half-way 边界控制 forward/backward 扩展，观察它是否能更好平衡 dominance graph 与 final join 的工作量。

## 1. 背景和目标

当前正式版本使用静态 `Tmid`，forward label 存储在 `[ell,Tmid]`，backward label 存储在 `[Tmid,rho]`，最后统一 crossing-arc join。此前 full-domain 对照实验把函数定义域放宽到 `[0,pricingHorizon]` 后，发现它不是单纯增加函数段操作，而是把复杂度从 dominance graph 转移到 join：`superset visited` 下降，但 join candidates 和 function eval 大幅上升。时间上 full-domain 和 half-domain 差距不大，且 full-domain 不是单调改进。

PDF 中的 Tilk-style dynamic half-way point 提供了另一个思路：不固定一个静态 `Tmid`，而是维护两个动态边界 `HB` 和 `HF`。`HB` 表示 backward 侧已经推进到的左边界，`HF` 表示 forward 侧已经推进到的右边界，并始终保持 `HB <= HF`。在本问题里，这两个边界只用于控制 label 是否继续扩展，不进入 dominance key、visited set、reachable set 的语义定义，也不改变最终 join 的 reduced-cost 函数计算方式。

本次计划中的新版本暂定名为 `GCNGBBDynamicBidirectional`。它从 `GCNGBBStyleBidirectionalFullDomain` 复制出来，保留 full-domain 函数定义域，但用动态 `HB/HF` 替代静态 `Tmid` 作为扩展边界。因此它更准确地说是“full-domain function labels + dynamic halfway expansion boundary”的实验版本。

## 2. 动态资源和边界定义

PDF 中的 `[L,U]` 不能直接理解为 `[0,pricingHorizon]`。在当前 TWET pricing 里，动态 half-way 的资源区间应沿用现有 midpoint 逻辑中的有效左端：

```java
dynamicHB = Math.max(dynamicMinHStart, earliestSourceCompletion);
dynamicHF = pricingHorizon;
```

其中 `dynamicMinHStart` 是当前 pricing 轮的有效任务窗口最左端，`earliestSourceCompletion` 是从 source 到任一任务的最早物理完成时间，`pricingHorizon` 是当前 pricing 轮的有效右端。这样初始化后，`HB/HF` 的资源含义和当前动态 midpoint 的左、右端保持一致，不会把 `HB` 放在没有实际任务完成意义的 0 上。

forward label 的单调资源使用 `earliestForwardCompletion(label)`，即当前 label frontier 的 `head.start`。backward label 的单调资源使用 `latestBackwardCompletion(label)`，即当前 label frontier 的 `tail.end`。这两个值和当前 `TIME` 队列排序已经一致。

## 3. 队列排序和出队语义

动态版本必须强制使用当前 `TIME` 主排序，不再允许 reduced-cost 作为主排序。当前 `TIME` 排序已经满足需求：forward 按最早完成时间升序，backward 按最迟完成时间降序；tie-break 仍保留 reachable set 基数和 reduced cost。这样 forward 出队顺序从左向右推进，backward 出队顺序从右向左推进，`HB/HF` 才能解释成已经处理过的资源前沿。

出队时需要处理 stale label。因为 `HF` 会随 backward 处理不断缩小，之前生成的 forward label 可能在弹出时已经位于当前 `HF` 右侧；同理，`HB` 会随 forward 处理不断增大，之前生成的 backward label 可能在弹出时已经位于当前 `HB` 左侧。由于队列按资源单调排序，一旦队首已经越界，剩余同侧 label 也全部越界，可以直接清空该队列。

forward 出队逻辑为：

```java
ForwardLabel label = FWUL.peek();
if (earliestForwardCompletion(label) > dynamicHF) {
    FWUL.clear();
    return;
}
label = FWUL.poll();
forwardExtend(label);
dynamicHB = Math.max(dynamicHB, earliestForwardCompletion(label));
```

backward 出队逻辑为：

```java
BackwardLabel label = BWUL.peek();
if (latestBackwardCompletion(label) < dynamicHB) {
    BWUL.clear();
    return;
}
label = BWUL.poll();
backwardExtend(label);
dynamicHF = Math.min(dynamicHF, latestBackwardCompletion(label));
```

因为越界 label 会在出队前直接剪掉，更新 `dynamicHB/dynamicHF` 时不再需要 `min(..., dynamicHF)` 或 `max(..., dynamicHB)` 兜底。`HB <= HF` 由出队检查和更新顺序自然维持；若出现数值误差，可在实现中用 `Utility.compare...` 做容差判断。

## 4. 外层处理流程

当前 `GCNGBBStyleBidirectionalFullDomain.solve()` 是先完整耗尽 forward 队列，再完整耗尽 backward 队列，最后统一 join。动态 half-way 版本需要改成 forward/backward 交错处理，但 final join 仍保留现在的统一 join，不采用 streaming join。这样可以最大限度复用现有 label table、candidate top-K 和 join 逻辑，也更容易和当前 full-domain 对照版本做公平比较。

外层循环的基本结构为：

```java
while (canContinue() && (!FWUL.isEmpty() || !BWUL.isEmpty())) {
    if (shouldProcessForward()) {
        dynamicForwardExtend(lp);
    } else {
        dynamicBackwardExtend(lp);
    }
}
compactAndSortActiveLabelListsForJoin();
joinAllForwardTerminalGroups(lp);
finalizeGeneratedColumns();
```

选边策略先做两个实验配置：

1. `MORE_LABELS_FIRST`：哪个队列剩余 label 多，就优先处理哪个队列。这个策略偏向控制 backlog，避免一侧积压过多。
2. `FEWER_LABELS_FIRST`：哪个队列剩余 label 少，就优先处理哪个队列。这个策略可能更快推进某一侧边界，从而收缩另一侧的可扩展范围，但也可能造成不平衡。

如果某一侧队列为空，则处理另一侧。如果某一侧队首越界并被清空，则下一轮自然转向另一侧。选边策略只影响效率和平衡，不应影响正确性。

## 5. 扩展可行性判断

新版本仍保留 full-domain 函数定义域 `[0,pricingHorizon]`。动态边界只替代当前扩展停止判断里的静态 `tMid`。

forward direct feasibility 当前为：

```java
earliestCompletion <= tMid
```

动态版本改为：

```java
earliestCompletion <= dynamicHF
```

backward direct feasibility 当前为：

```java
rhoPrime >= Math.max(tMid, HStart)
```

动态版本改为：

```java
rhoPrime >= Math.max(dynamicHB, HStart)
```

`buildForwardReachableSet`、`buildForwardReachableSetFromParent`、`buildBackwardReachableSet` 和 `buildBackwardReachableSetFromParent` 都会通过上述 direct feasibility 使用当前 `dynamicHF/dynamicHB`。需要注意的是，label 创建时的 reachable set 是当时边界下的结果；边界后续变化后，队列里可能残留 stale label，因此出队时的队首越界检查仍然必须保留。

## 6. 保持不变的语义

该实验版本不改变以下内容：

1. 不改变 `PiecewiseLinearFunction` 的 full-domain 函数定义域。
2. 不把 `dynamicHB/dynamicHF` 写入 dominance key、visited set、reachable set signature 或 column signature。
3. 不改变 zero-dual job 过滤、active column 去重、candidate top-K、debug evaluator 复核逻辑。
4. 不改变 final join 的 crossing-arc 语义，仍然用 forward full extension、backward full extension 和 crossing arc reduced cost 生成完整列。
5. 不改变 source/sink root 的 full-domain 初始化。

这几个“不变”是正确性边界。`HB/HF` 只是扩展控制参数，不是 label 状态的一部分。

## 7. 正确性判断

正确性直觉来自资源单调处理。forward 队列按最早完成时间升序，如果队首已经大于当前 `dynamicHF`，剩余 forward label 的最早完成时间只会更大，因此它们不应再由 forward 侧扩展；backward 队列按最迟完成时间降序，如果队首已经小于当前 `dynamicHB`，剩余 backward label 的最迟完成时间只会更小，因此它们不应再由 backward 侧扩展。

这种剪枝不会把 `HB/HF` 当成 dominance 条件，也不删除已经存入 label table 的 live label；它只停止未处理队列中越过当前动态边界的同侧扩展。已经生成的 forward/backward labels 最后仍会统一参与 join，因此不会因为选边策略不同而改变完整列拼接语义。

需要特别避免的是 reduced-cost 主排序。如果 forward 不是按 earliest 升序、backward 不是按 latest 降序，则队首越界不能推出后续同侧 label 全部越界，清空队列就可能不安全。因此动态版本必须强制 `TIME` 主排序，reachable size 和 reduced cost 只能作为 tie-break。

## 8. 实施步骤

第一步，复制类和 engine。以 `GCNGBBStyleBidirectionalFullDomain` 为基准复制出 `GCNGBBDynamicBidirectional`，并新增对应 pricing engine。类名不采用过长的 `GCNGBBStyleBidirectionalDynamicHalfwayFullDomain`，避免后续使用和日志过长。

第二步，新增配置。配置至少包括：

```java
public boolean useGCNGBBDynamicBidirectionalPricing = false;
public String dynamicBidirectionalSideSelection = "moreLabels";
```

其中 `dynamicBidirectionalSideSelection` 支持 `moreLabels` 和 `fewerLabels`。动态版本启用时应强制内部使用 `TIME` 主排序，不依赖外部 `bidirectionalLabelQueueOrdering`。

第三步，改外层 solve loop。删除“先 forward 后 backward”的两段 while，改成交错处理。新增 `dynamicForwardExtend` 和 `dynamicBackwardExtend`，内部先做队首越界检查，必要时清空队列；真正 poll 并扩展后更新 `dynamicHB/dynamicHF`。

第四步，替换扩展边界。新增 `currentForwardBoundary()` 和 `currentBackwardBoundary()` 之类的小方法，避免到处直接读字段。forward direct feasibility 使用 `dynamicHF`，backward direct feasibility 使用 `dynamicHB`。如果保留原 `tMid` 字段用于统计或兼容复制代码中的旧方法，需要明确它不再是扩展边界；`dynamicHB/dynamicHF` 也不用于 join 常数延拓。第一版应保留 full-domain 版本现有 join extension 行为，避免同时修改 join 语义。

第五步，补统计。`statisticsSummary()` 需要记录 `dynamicHB/dynamicHF` 的最终值、forward/backward 队列被 boundary clear 的次数、被 clear 的 label 数量、选边策略、forward/backward 实际 pop 次数。这样才能判断动态策略到底是在减少 dominance graph 还是减少 join。

第六步，新增 comparison runner。可以复用 `GCNGBBFullDomainComparisonTest` 的结构，新增 dynamic-more 和 dynamic-fewer 两种模式，与 half-domain、full-domain static 进行对照。第一批测试仍使用 no-outsourcing 15、20、21、22、25，记录 objective、bound、valid、exact time、join candidates、function eval、superset visited。

## 9. 验证标准

基础验证包括 focused `javac`、`PaperDominanceGraphConsistencyTest` 和带 CPLEX 的小样例 BPC。功能验证要求：

1. 与当前 half-domain/full-domain 对照一样，所有测试 case 的 incumbent、bound 和 `valid=true` 保持一致。
2. 对 15、20、21、22、25 这些已有 no-outsourcing 样例，dynamic-more 和 dynamic-fewer 至少不能出现漏列导致 root bound 变差或 validation 失败。
3. 日志中 `dynamicHB <= dynamicHF` 始终成立；如果最终两者交叉，说明出队检查或更新时机有问题。
4. 如果某侧队列被 clear，必须能从队首资源和当前边界解释清楚，不能因为非 TIME 主排序误清。

性能验证不以单个 case 的 wall time 为唯一依据，而同时看 `join candidates`、`join function eval`、`superset visited`、`envelopeMerges`、`dominanceChecks`。前面的 full-domain 对照已经说明，单看 wall time 很容易被两个热点此消彼长误导。

## 10. 暂不做事项

第一版暂不做 streaming join。PDF 中提到可以在 backward label 处理时做 join，但当前 GCNGBB-style 已经统一到最终 join，保留这一点更利于验证 correctness。

第一版暂不把 `HB/HF` 纳入 dominance key，也不改 NG/DSSR 或 reachable-set 签名。动态边界只是扩展控制参数，不能混入路径状态。

第一版暂不实现 workload 复杂代理，比如根据 label 可扩展 job 数、frontier 段数或 graph node 数选边。先只比较“剩余队列多的一边优先”和“剩余队列少的一边优先”。如果这两个简单策略都没有明显收益，再考虑更复杂的 workload 指标。

第一版暂不把当前 full-domain 对照改成正式版本。已有统计显示 full-domain 不是单调更好；动态版本只是一个新的实验分支。
