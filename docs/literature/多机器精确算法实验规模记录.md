# 多机器精确算法实验规模记录

这份记录只看多机器场景，而且只保留精确算法相关论文。单机 exact 的 Tanaka 2009、Tanaka 2012 不放在这里。非精确算法的论文也不放进来，例如 Şen-Bülbül 2015 本身是 preemptive relaxation + primal-dual / heuristic 框架，不算严格意义上的 exact；Kramer 2015 也不是 exact。当前和我们后续工作最相关、同时又属于精确算法的，主要就是 Bulhões et al. (2020) 这篇统一的 branch-cut-and-price 框架论文。

## 结论

如果只看多机器精确算法，并且只看和我们现在这条 TWET 线最相关的文章，那么当前最值得作为参考上限的是 Bulhões et al. (2020)。这篇文章不是只做一个具体问题，而是给了一个统一的 branch-cut-and-price 框架，并在几类并行机调度 benchmark 上做了实验。就 earliness/tardiness 这条线来说，文中最核心的两组数据分别是 Şen-Bülbül 2015 的 `R|| Σ(w'E + wT)` 实例，以及 Kramer 2015 的 `R|r,s| Σ(w'E + wT)` 实例。前者最大做到 120 jobs、5 machines，后者最大做到 80 jobs、4 machines。再往外扩一点，这篇文章在其他并行机精确问题上还测到了 100 jobs、20 machines，但那已经不是 TWET 目标了。

## 对我们最有用的判断

如果只盯着和当前课题最接近的多机器 TWET 精确算法，那么现阶段最值得参考的就是 Bulhões 2020 这一条线。它对应的两个直接相关结论很明确。第一，不带 release/setup 的并行机 TWET，精确算法已经能做到大约 120 jobs、5 machines。第二，带 release dates 和 sequence-dependent setup times 后，问题难度明显上升，实验规模会回落到大约 80 jobs、4 machines，而且并不是所有组都能稳稳解完。

这对我们后面设计 TWET 的精确算法有两个直接启发。一个是如果我们先做不带 setup 或 setup 较弱的版本，那么把目标放在百级 jobs、少量机器上是合理的。另一个是如果一开始就把 release、setup、非规则目标一起压进去，那么即使沿着 branch-price-cut 这条路走，规模也大概率会很快卡住，实验设计需要更保守。
