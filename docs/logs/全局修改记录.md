同日逐个检查 `PiecewiseLinearFunction` 操作函数的 backward 适用性。结论是底层函数操作大多方向无关；不适用于 backward 的主要是 `normalize()`，以及内部依赖 `normalize()` 的 `updateDominatedIntervals()`、`mergeMinimum()`、`mergeMinimum2()`。这些当前都是 forward 集合操作，后续若做 backward label 合并或 partial dominance，需要单独实现 backward 版本。

同日补充方向化接口取舍：`mergeMinimum2()` 不参与正式方向化改造，后续可注释或标记为实验函数；正式接口应集中在 `normalize(direction)`、`mergeMinimum(g,direction)` 和 `updateDominatedIntervals(g,direction)`。旧无方向接口不再作为主接口使用，pricing 中应显式传入 forward/backward 方向。

同日进一步检查了启发式局部搜索当前实现是否“逻辑正确”。结论是单候选邻域构造和分段函数拼接评价框架基本成立，但跨机器算子存在实质风险：`savedOperations` 只保存扫描过程中的全局 record-best，随后 `commit()` 又把未保存的机器对写入 `noImprovedSeqPair`，可能跳过本来有改进的机器对；`OptCost.compareTo()` 仍有 double 差值强转 int 的排序问题；`PathInsertOperator` 中 `if(len==1) break` 很可能导致长度为 2 的反向插入候选漏枚举；`TwoOptStarOperator` 和 `CrossExchangeOperator` 用 `Set.of(seq1, seq2)` 做 pair cache，在两条序列相等时存在异常风险；非零 setup 下 `PathInsertOperator.mergeCmaxValidation()` 中“m1 只会变小”的假设也不能作为严格依据。这些已补充到启发式对应关系文档，后续应优先修复并做小规模全候选对拍。

同日根据进一步讨论修正了上述风险优先级：`savedOperations` 只保留全局最优或 record-best 候选本身可以作为设计选择，真正风险是不能据此把未保存的机器对直接写入 `noImprovedSeqPair`；跨机器批量提交不是一定错误，而是需要明确它是互不重叠 move 的贪婪批处理，不是论文式单 move best-improvement；`compareTo()` 在当前整数数据下影响较小，可先补注释或后续顺手改；`Set.of(seq1, seq2)` 在当前每台机器都有任务且序列不可能相同的前提下不太会触发；比较确定需要关注的是 `PathInsertOperator` 的 `if(len==1) break` 会漏掉两个任务片段的反向插入候选，以及非零 setup 下 Cmax 剪枝仍未完善。

同日修正跨机器局部搜索算子的候选保存语义。`TwoOptStarOperator`、`PathInsertOperator` 和 `CrossExchangeOperator` 的 `searchBest()` 不再只保存刷新全局 `bestCost` 的候选，而是保存所有 `deltaCost < 0` 的改进候选；`commit()` 仍按原逻辑排序并贪婪提交互不重叠机器上的 move。这样 `savedOperations` 与 `noImprovedSeqPair` 的语义一致，避免“有改进但没刷新全局 best 的机器对”被误写入无改进缓存。验证：`javac -encoding UTF-8 -cp target/classes -d target/classes src/HEU/Move.java` 通过。

同日继续排查启发式局部搜索剩余风险。排除非零 setup 导致的 Cmax 剪枝后，当前主要剩两个较明确问题：`PathInsertOperator` 的 `if(len==1) break` 仍会漏掉两个任务片段的反向插入候选；跨机器算子在批量 commit 后写 `noImprovedSeqPair` 时使用的是提交后的新序列，因此如果某台机器已被本轮其他 move 改动，它和第三台机器形成的新 pair 其实没有被搜索过，不应写入 no-improved 缓存。低优先级问题包括 `OptCost.compareTo()` 的 double 转 int、`Set.of(seq1, seq2)` 的极端重复序列鲁棒性，以及 VND 固定随机顺序/批量提交带来的策略差异。

同日复查前述两个暂不修改的问题。插入算子的风险在于：ALNS repair 当前不显式过滤 `curUpperBound/big_M` 插入位置；在 `Greedy/Regret` 中若某个任务所有位置都不可行，仍会选一个上界位置插入，`RandomizedInsertion` 在可行位置少于随机候选数时也可能抽到上界位置。该问题在“删除任务来自原可行解、扰动比例不大、CmaxH 宽松”的当前设定下不一定触发，暂不修改。`Set.of(seq1,seq2)` 的风险在于两条机器序列完全相同，尤其多台空机器时会抛异常；若始终保证每台机器非空且任务序列不会重复，则当前可接受，暂不修改。后续若允许空机器或更激进 ALNS repair，应优先补这两处兜底。

同日补充确认 ALNS 插入算子暂不修改的理由：当前 repair 允许较差甚至被上界截断的位置参与选择，可以理解为保留更大的扰动空间；在当前 `CmaxH` 较宽、删除比例受控、每台机器保持非空的设定下，暂不把它作为主正确性修复点。后续如果发现 repair 后频繁出现 `curUpperBound/big_M` 成本或不可行序列，再考虑改成“优先真实可行位置；全部不可行时再兜底并记录 warning”的软约束方式。另确认当前 ALNS+LS 的解更新策略：全局最优解保存在 `data.configure.bestSolution`，`EngineVND.commit()` 每次 LS move 成功后都会尝试更新全局最优；`EngineALNS.search()` 每轮 ALNS repair + VND 完成后再次比较并更新全局最优。ALNS 下一轮扰动基于 `current`，而不是直接基于全局最优；`current` 只有在 `tmpSol.curCost <= current.curCost` 时才更新为本轮局部搜索后的解。
