# 函数单点与 Gap 分析记录

这份记录只针对当前 TWET 代码里的函数表示与操作实现，尤其是 `PiecewiseLinearFunction`、`shift`、`add`、下包络/占优处理，以及启发式里实际会调用到的两段拼接和三段拼接。这里记录的是当前阶段的工程判断，用来帮助后续继续改代码时缩小排查范围，不把注意力耗在不太可能发生的异常情况上。它不是形式化证明，但基于前面对代码和操作链条的逐段分析，当前结论已经足够指导实现。

当前最重要的判断是：在现在这个问题设定下，函数里出现单点的来源基本只有一种，就是函数在参与操作之前，已经因为全局 makespan 上界 `T` 的裁剪而只剩下最右端那个可行点。换句话说，单点只可能出现在最大时间 `T` 处，而不应该在中间某个非 `T` 的位置孤立出现。只要输入函数本身是连续的、没有单点，那么 `add` 和 `shift` 这类操作本身不会凭空制造一个新的内部单点；它们最多只是把原来已经存在的边界单点继续传递下去。类似两个函数取下包络、一个函数在某区间上占优另一个函数、或者 dominate 产生某段不可行区间，这些操作在当前实现下也只会生成正常连续段，不会平白制造出一个夹在连续区间内部的突变单点。dominate 即使把一段替换成 `M`，本质上也是一整段水平不可行区间，不是离散的孤立点。

因此，如果某一步真的出现了单点，更合理的解释不是“这一步操作把普通连续函数加工坏了”，而是“参与这一步操作的某个函数本身已经被裁到了 `T`，只剩下最后那个点”。这和前面 `add` 那里专门允许 `cur <= nxt` 的处理是一致的：代码要保的不是任意位置的奇怪单点，而主要是上界裁剪后在末端留下的那个合法边界点。按这个理解，当前实现里不太可能出现“函数内部某个非最大时间位置有一个单点，而左右两边仍然是连续段”的情况，更不太可能恶化成一个函数里有多个相互分离的单点。如果真存在这种多单点结构，那么经过 `shift`、`add`、下包络、拼接这些操作后，单点会很快扩散到更多函数对象里，整个函数库的实现都必须额外支持“多个离散单点单独成段、单点之间也要逐点比较”的复杂逻辑；而前面的分析说明，当前问题设定下基本不会走到这一步。

这也意味着，当前单点问题的工程影响相对可控。因为它只会卡在最大时间 `T` 的最右边界，所以相关兼容处理主要集中在函数最右端的保留，例如 `add` 时允许 `[T,T]` 这样的公共区间继续生成一段，`evaluate` 时允许 `t == tail.end` 单独取值，`trimToDomain` 和 prefix/suffix minimize 时允许尾部被裁成单点而不是直接清空。只要这些地方一致地把最右端边界点当成合法点保住，就不需要再为“内部离散单点”额外设计复杂机制。再往实际求解上看，真正大规模实例里，标签或函数恰好卡到全局 makespan 上界 `T` 的情况本身也不会太多，所以这类边界兼容更像是防止小规模调试和极端边界情况下的伪空函数，不会成为主导性瓶颈。

关于 2011 和 2014 两篇文章里提到的竖直 gap 与水平 gap，也可以顺着这个判断一起理解。竖直 gap 出现在两个函数 merge 时，更多是端点左右函数值不完全连续的问题，本质上只是“边界左右值不一样”；只要后续操作里这些地方仍然按正常段处理，而不是变成某个独立单点，工程上影响不大。水平 gap 则更简单，它对应某一段没有有限函数值，直接视为一段不可行区间即可；在现在的实现里，这等价于那一段是极大值或者 `M`，并不会额外引入离散点结构。之前真正麻烦、也最值得担心的，其实一直不是这种正常的竖直或水平 gap，而是“两个连续段之间突然夹了一个必须单独处理的突变单点”。因为一旦存在这种结构，函数比较、相加、下包络、dominance 乃至拼接时，都必须把这个点单独拆出来处理，复杂度和实现负担都会明显上升。基于当前分析，这种情况在本问题设定下基本不会出现，所以这部分担忧可以先放下。

当前可以据此采用一个更收敛的实现判断：后续继续检查函数操作时，优先把注意力放在“最右端 `T` 附近的边界点有没有被误删、误判成空函数、或者在 `tail.end` 取值时被漏掉”这类问题上，而不需要预先为内部多单点、分散单点、单点链式传播等复杂情形做额外机制。如果后续实验里真的观察到内部非 `T` 单点，再单独回头扩展函数表示与操作规则即可；在那之前，按当前这一套判断收住范围是更合理的。
