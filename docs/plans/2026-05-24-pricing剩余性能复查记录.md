# 2026-05-24 pricing 剩余性能复查记录

这轮只做静态复查，不改代码。复查范围按当前默认主路径来定：`HeuristicPricingEngine -> GCBidirectional -> PaperDominanceGraph`，不把已经退居 fallback 的旧 `ExactPricingEngine + DominanceGraph` 当成当前主要矛盾。

先说结论。当前 pricing 主链上那些最显眼、最低级的冗余已经基本清掉了：启发式 seed 选择已经改成 top-K heap，accepted move 后的 route profile 也已经改成局部重建；双向 exact pricing 已经有 terminal-job bitset、group 级时间/成本下界、组内 lazy compaction、以及 normal label 的端点 O(1) 最小值读取。现在剩下的热点主要有两类：一类是启发式邻域枚举中“先做重计算、后做便宜过滤”的顺序问题；另一类是 `PaperDominanceGraph` 里仍然存在的结构性图维护开销。

1. `HeuristicPricingEngine` 当前最值得继续动的点，不是再改 `apply()->rebuild()`，而是把 cheap check 提前到 expensive merge 前面。现在 `evaluateRemove()`、`evaluateAdd()`、`evaluateExchange()` 都是先算 `candidateCost`，再判断 `isRemoveCompatible()` / `isInsertCompatible()`。但后者只是看分支禁弧，是 O(1) 的便宜判定；前者会走 `removeCost()` / `insertOrReplaceCost()`，其中 `insertOrReplaceCost()` 还会调 `merge3Segments()` 做函数拼接。也就是说，当前大量本来会被禁弧直接剪掉的候选，仍然先付了一次分段函数拼接成本。这是现在 heuristic pricing 里最清晰、最稳妥的下一刀。

2. `HeuristicPricingEngine` 里 `tryAddNegative()` 仍然保留了 `hasDuplicateJobs(sequence)`。从当前状态不变量看，这层大概率已经退化成防御性检查：seed 来自当前列池，tabu move 生成时又已经靠 `state.used[job]` 防止 add/exchange 产生重复 job。因此这部分不是大热点，但严格说仍然有一点重复扫描序列的冗余。是否删除可以放在禁弧 cheap check 前移之后再看。

3. `GCBidirectional` 主链已经没有明显傻瓜级冗余。当前 `joinFromBackward()` 不再扫 `0..n` 全量 terminal，而是只扫 `activeForwardTerminalJobs`；组内也会把已支配的 forward label 物理压掉，同时重算当前 live label 的 `minForwardReducedCostByLastJob` 和 `minForwardEllByLastJob`。这条线上剩下的性能问题更偏“结构性不可避免”：每个 backward label 仍然需要按 terminal group 做下界筛选，过筛后仍然要对 surviving pairs 做 join 函数最小值评估。这里我暂时没有再看到一个像前几轮那样明显、低风险、立刻该动的点。

4. `PaperDominanceGraph` 仍然是 exact pricing 里最像“还能继续提速”的地方。虽然 `roots`、邻接表和 subset candidate 过滤都已经收紧过，但 `deleteNode()` 里仍然会对每个 predecessor 收集一遍 compatible successors，再做一次 maximal frontier 过滤。它比原来的全量前驱×后继笛卡尔积已经好很多，但图规模大时，这里依然是结构性开销。另外，`findTerminalSupersetNodes()` / `findImmediateSubsetNodes()` 依旧是基于图遍历加 visited set 的在线搜索，是否还能再降，要么得进一步收紧图的 cover 关系，要么得引入更重的索引结构，这一层就不是“轻量优化”了。

5. 当前默认主路径是 `enableHeuristicPricing=true` 且 `enableBidirectionalPricing=true`。因此真正该盯的是 `HeuristicPricingEngine + GCBidirectional + PaperDominanceGraph`。旧的 `ExactPricingEngine + DominanceGraph` 还保留在配置 fallback 里，但不应再作为当前主要性能判断依据。

当前判断是：如果下一步继续做 pricing 性能优化，优先级应当是 `HeuristicPricingEngine` 里把禁弧兼容性判断前移，再看是否删掉重复 job 防御性检查；exact 侧则继续盯 `PaperDominanceGraph.deleteNode()` 的重连成本。双向 join 本身暂时没有发现新的明显冗余。

同日已先落地第一步低风险优化：`HeuristicPricingEngine.TabuRouteState` 的 `evaluateRemove()`、`evaluateAdd()`、`evaluateExchange()` 现在先做 `isRemoveCompatible()` / `isInsertCompatible()` 这类 O(1) 的禁弧兼容性检查，再决定是否进入 `removeCost()` / `insertOrReplaceCost()` 的 `merge2Segments/merge3Segments` 评估。这样可以避免本来就会被分支禁弧直接剪掉的候选先付一次分段函数拼接成本。此次不改 tabu 邻域、接受规则和 reduced-cost 增量公式，只调整 cheap check 与 expensive merge 的顺序。

验证方面，`git diff --check` 通过，`HeuristicPricingEngine.java` focused `javac` 编译通过。额外尝试运行 `HEU.SmallBPCBatchTest` 做一轮真实 heuristic pricing 路径回归，但命令行环境缺少 CPLEX runtime `ilog.concert.IloNumExpr`，因此本次停在编译验证，没有做成端到端运行。

继续清理后，`tryAddNegative()` 里的 `hasDuplicateJobs(sequence)` 也已删除。当前 heuristic pricing 的 seed 来自现有列池，而 tabu move 生成阶段又通过 `state.used[job]` 保证 add/exchange 不会把已在序列中的 job 再插入一次，因此这层重复 job 扫描已经退化成防御性检查。删除后每次尝试把负列候选放入本地池时，少做一次整序列线性扫描。当前仍保留的序列合法性检查主要是 `isSequenceCompatible()` 的分支禁弧判定和后续的 signature 去重。
