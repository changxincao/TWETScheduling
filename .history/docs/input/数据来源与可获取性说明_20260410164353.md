# TWETScheduling 数据来源与可获取性说明

这份说明记录本项目的输入数据来自哪里，以及其他可能的来源等信息。结论为：当前仓库里真正已经落到本地、格式清楚、代码已经接上、可以直接持续使用的数据，核心就是 Tanaka 体系下的单机 total weighted earliness-tardiness benchmark，也就是 `wet40.zip`、`wet50.zip` 和 `wet100.zip` 这一组。围绕 parallel machine TWET 相关文献，我们另外追过几条常见数据线，但结果基本都是“知道它存在，知道它在论文里被用过，甚至能找到历史下载痕迹”，却拿不到稳定可用的原始文件。因此，从工程推进和后续论文写作两个角度看，现阶段先基于 Tanaka 数据继续做，是最稳妥也最现实的选择。

## 当前项目里实际在用什么数据

仓库里的数据组织方式已经能说明很多问题。`data` 目录下既有原始压缩包 `wet40.zip`、`wet50.zip`、`wet100.zip`，也有已经展开或进一步处理后的目录 `40-1`、`50-1`、`100-1`、`40-2`、`40-4`、`50-2`、`50-4`、`100-2`、`100-4`。其中 `40-1 / 50-1 / 100-1` 对应的更像原始单机实例，`40-2 / 40-4 / 50-2 / 50-4 / 100-2 / 100-4` 则明显是项目内部继续加工后的并行机版本。从代码入口也能确认这一点。当前启发式和几套精确模型都在直接读取这些 `.dat` 文件，例如 `src/HEU/Solution.java`、`src/Basic/ArcFlowModel.java`、`src/Basic/TimeIndexModel.java` 和 `src/Basic/ATIParallel.java`。换句话说，当前项目运行时并不是临时随机造实例，而是实打实地在读取本地 benchmark 文件。

不过，不能把这套数据简单理解成“原样下载后直接使用”。项目内部已经对原始实例做了两层处理。第一层在 `src/Common/ETConverter.java`，这里的逻辑很清楚，就是把单机 E/T 实例转换成并行机版本，同时按机器数对 due date 做缩放，新的输出文件名还会带 `_2m`、`_4m` 这类标记。第二层在 `src/Basic/Data.java`，当前读入时 setup time 采用的是零 setup 设定，原先类似随机 setup 的写法并没有启用。因此，严格来说，当前项目最终使用的是“基于公开 benchmark 转换并再设定参数后的并行机测试集”，而不是未经处理的原始数据。这种做法本身并不特殊，很多 parallel machine 调度文章都会在公开单机数据上继续派生自己的实验集。

## Tanaka 这套数据的来源

关于原始数据来源，这部分现在已经可以基本坐实。项目里的 `wet40.zip`、`wet50.zip` 和 `wet100.zip` 与 Tanaka 维护的 benchmark 页面是直接对上的，最直接的入口是 `https://sites.google.com/site/shunjitanaka/sips/benchmark-results-sips`。这个页面列出了 `TWT`、`TWET` 等多类单机调度 benchmark，其中 `TWET` 正好就是 `wet40 / wet50 / wet100` 这一组。除了文件名一致，文件格式也完全对得上。项目中的原始单机 `.dat` 文件第一行是作业数 `n`，后面每行四列分别是 `p d wE wT`，这和 Tanaka 页面上的数据说明一致。因此，当前项目最准确的表述应该是：原始数据来源于 Tanaka 的单机 TWET benchmark，项目在其基础上再构造了并行机版本。

## 其他文献数据的追查结果

为了确认是否存在更贴近 parallel machine TWET 文献、并且可以直接替代当前数据的公开实例，我们重点追了 Bulhões et al. (2020) 这条线。原因很直接，这篇文章本身就是一篇统一的 branch-cut-and-price 框架论文，而且实验部分用了两组与我们目标最接近的 earliness/tardiness 数据，一组来自 Şen and Bülbül (2015)，另一组来自 Kramer (2015)。顺着这条线往下看，Şen and Bülbül 2015 并不是沿用别人的现成 benchmark，而是作者自己生成的一套 unrelated parallel machine TWT/TWET 实例。论文里把生成规则写得很细，包括作业规模的组织方式、不同机器上的加工时间如何派生、earliness 和 tardiness 权重如何生成，以及 due date 如何按经典的 `TF / RDD` 方案构造。更重要的是，论文里还明确提到 source code 和 test instances 可以向通讯作者索取，或者从 `bulbul/publications.html` 获取。我们后来确实追到了一条历史压缩包链接，即 `Sen_Bulbul_Rm_TWT-TWET_source-data-results_IJOC_2015.rar`，但现在访问只返回 `403 Forbidden`。也就是说，这组数据过去确实公开过，但当前已经无法稳定获取。

Kramer (2015) 这条线更难处理。Bulhões 2020 已经说明，这组实例是在 Şen-Bülbül 实例基础上继续派生出来的，额外加入了 release dates 和两类 sequence-dependent setup times，也就是 small 和 large 两种设置。但问题在于，我们没有找到这组数据自己独立的公开下载包。和 Şen-Bülbül 相比，后者至少还留有一条能定位到文件名的历史链接，而 Kramer 这组连同等级别的公开入口都没能确认下来。所以目前对 Kramer 数据最稳的结论只能是：知道它的来源关系，知道它大致是怎么构造出来的，但原始实例文件当前没有拿到。

我们还查了 Kramer and Subramanian (2019) 的综述，因为这篇文章整理过一个“公开可下载实例”的列表，理论上最适合做反查。结果是，这篇综述确实给出了几条和我们最相关的线索，包括 Tanaka 的 SiPS、Şen-Bülbül 2015 的历史压缩包入口、`Sourd / Kedad-Sidhoum` 的 ET 数据以及 UFAM/ALGOX 的 weighted earliness-tardiness 数据。但进一步核对后可以看到，真正还稳定可用的几乎只剩 Tanaka 的 SiPS 页面。Sabancı 那条压缩包链接已经返回 `403`，`www-poleia.lip6.fr/~safia/et/` 这一类老站点基本失效，UFAM/ALGOX 对应页面当前也无法稳定访问。也就是说，文献里出现过的数据来源不少，但真正还能顺利落地到本地、供当前项目直接使用的数据源非常有限。

## 还能补什么数据

如果把范围再稍微放宽一点，Tanaka 自己维护的 `SiPSi` 页面仍然值得保留为后续扩展方向，因为这里至少还能看到两类和 TWET 直接相关的数据家族，即 `SKS` 和 `BKY`。这说明 Tanaka 体系本身并不只有当前这套 `wet40 / wet50 / wet100`。但这件事的意义更像是“后续可扩展的公开来源还在”，而不是“现在马上就有另一套并行机 TWET 数据可以无缝替换进来”。从当前工程目标出发，最重要的判断仍然没有变：现阶段不存在一套比 Tanaka 更明确、更稳定、更方便接入的可用公开数据。

## 当前建议

因此，当前最合适的口径应该统一为：项目原始数据来源于 Tanaka 维护的单机 TWET benchmark，即 `wet40 / wet50 / wet100` 系列实例；项目在其基础上利用内部转换程序构造并行机版本，并在读入阶段采用零 setup 的设定。我们同时检索了文献中常见的其他 TWET 数据来源，特别是 Şen-Bülbül 2015 及其后续派生的 Kramer 2015，但这些数据目前只能确认历史公开线索或实例生成规则，尚无法稳定获得原始文件。在这种情况下，先基于 Tanaka 体系继续完成算法开发、框架验证和第一轮实验，是完全合理的安排。等后续算法真正跑起来，若有必要再补第二套数据，一条路是继续沿着 Tanaka 的 `SiPSi` 体系扩展，另一条路是依据 Şen-Bülbül 2015 已公开的生成规则自行重建一套更贴近文献的实例集。现阶段没有必要为了几条高概率已经失效的老链接继续停滞项目进度。
