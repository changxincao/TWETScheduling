# 2026-06-02 node join 半域拼接语义补充

本记录补充 `GCBBStyleBidirectionalFullDomainNodeJoin` 的 one-sided node join 语义。当前判断是：用户下面这段描述在“当前 one-sided 半域加速变体”下基本成立，但不能解释成严格 full-domain node join 已经实现；严格 full-domain 对照应保持两侧函数定义域为 `[0, pricingHorizon]`，只通过入队和 terminal 规则控制哪一侧第一次越过 `Tmid`。

用户原文如下，作为后续讨论背景保留：

> OK，那也就是说，现在这个node join这个采用半域的拼接是OK的。把这一串分析都详细注释和日志一下，即node join这里，应该只需要采用one-side的超出Tmid就好了，本来是打算做不超过那侧其实只需要控制如果超过Tmid以后不生成这个label就好了，定义域还是全域的。但现在做成了半域的，也可以，这个能够加强占优和减少扩展的耗时等等。以及这里边，由于node join的时候不需要再做函数偏移了，可以直接拼接，此时不需要做像arc join的那个函数延拓了。（其实arc join那里感觉上也不需要常数延拓的，之后分析），本质就在于，现在一侧定义域是全域，一侧是半域，直接拼接肯定有公共区间的，就类似arc join但是full domain的一样。 然后由于做了半域裁剪，这一侧可能导致Tmid处的单点，这个也不需要做常数延拓了，add是可以直接加一个单点函数从而产生一个单点函数的，就OK了这部分。
>
> 然后的话，这里关于node join比arc join效果要差的原因，感觉上可能还是在于join多一点消耗的时间不是很多，但从arc join要做node join的时候，那多扩展的一侧要多做的扩展，占优等可能时间要更多。

确认后的结论分三层。

第一，one-sided node join 只需要一侧承担“第一次越过 `Tmid`”这个判断是合理的。原本更干净的 full-domain 方案是两侧函数都保持 `[0, pricingHorizon]`，未承担 crossing 的一侧只是在生成越过 `Tmid` 的 child 后不再保存或不再继续扩展；这样定义域仍是 full-domain，只是扩展规则控制 crossing side。当前代码进一步把非 crossing 侧的 root/penalty 裁成对应 half-domain。这不是严格 full-domain ablation，但作为加速变体是自洽的：函数定义域变短后，dominance 只需要覆盖保留下来的半域，比较和 envelope 更新都更容易成功，保留 label 和后续扩展量也会下降。

第二，node join 的 final join 不需要 arc join 那种 crossing arc 的 `delta = setup(i,r)+p_r` 平移。node join 已经把 join job 物化到同一个 node 上，forward 侧保存的是 `preNodeFrontier`，用于避免 join node 的 job penalty 和 job dual 被两侧重复计算；backward 侧保存的是从同一 job 开始的 suffix-min frontier。因此 final join 可以直接做 `forward.preNodeFrontier.add(backward.frontier)`。当前代码也确实没有在这里做 `getForwardJoinExtension()` / `getBackwardJoinExtension()` 常数延拓。若 one-sided 模式保留了半域裁剪，`add()` 的语义就是只在两侧物理定义域公共区间上相加，不会临时补回被裁掉的另一半。

第三，半域裁剪带来的 `Tmid` 单点在当前实现里有专门保护。`insertForward()` / `insertBackward()` 进入普通 `PaperDominanceGraph` 前会先调用 `isSinglePointFrontier()`，只要 frontier 恰好退化为 `head.start == tail.end == tMid`，就进入 `SinglePointStore`，并通过 `dominatesSinglePoint(..., tMid, ...)` 做点支配查询。final join 扫描 single-point store 时，底层 `add()` 对普通 `[Tmid,Tmid]` 单点接触可以生成零长度结果，因此这部分不需要额外常数延拓。需要保留的限制是：`add()` 不能被当成内部跳变点左右极限的严格点值处理；同时 single-point store 目前只参数化到 `Tmid`，如果以后出现非 `Tmid` 的单点边界，需要另行泛化。

node join 慢于 arc join 的主要原因，当前也更倾向于用户的判断：不是 final join 多一次同点拼接本身最重，而是 node join 为了把 crossing arc 前移到 crossing-side label，会让承担 crossing 的一侧多生成、插入和占优更多 label。即使 one-sided 后 join function evaluation 已经大幅下降，仍然比 full-domain arc join 慢，说明额外扩展、dominance graph 更新、boundary terminal 保存和由此带来的候选传播成本仍然占主要位置。换句话说，arc join 把很多 crossing 组合留到 final join 阶段做扫描和剪枝；node join 则提前把一部分组合物化为 label 状态。前者的 join 开销未必比后者的 label table 膨胀更贵。

因此当前工程判断是：如果继续保留 node join 分支，one-sided half-domain 变体可以作为有效加速口径继续观察；但命名和论文/实验解释中必须说明它不是纯 full-domain node join。后续若要严格比较 full-domain node join 与 full-domain arc join，应把函数定义域恢复为全域，只保留 one-sided crossing 的入队和 terminal 控制。arc join 是否也能取消常数延拓属于另一个问题，不能由本次 node join 结论直接推出，需要单独按 crossing arc 平移后的定义域对齐再分析。
