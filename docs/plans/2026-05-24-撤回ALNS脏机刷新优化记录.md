# 2026-05-24 撤回 ALNS 脏机刷新优化记录

这次把前一轮针对 `ALNS -> VND` 交接层做的脏机局部刷新和按机器 copy-on-write 全部撤回，恢复成原来的全量 `updateInformationM(m)` 流程。

撤回原因很直接。那套改动虽然方向上成立，但实现复杂度明显高于当前收益：一方面，初始解启发式不是当前整套 BPC 的主要时间瓶颈；另一方面，`Solution.copy()`、机器级函数缓存和 VND 入口刷新之间原本耦合较深，强行引入共享态和延迟断开以后，后续维护和排查成本会上升，也更容易埋下状态污染或脏缓存边界错误。现阶段没有必要为了这部分潜在收益承担额外复杂度。

本次撤回后，保留前面已经完成的 pricing 侧优化，包括 exact pricing 的 reachable set / join 剪枝、`PaperDominanceGraph` 数据结构收紧，以及 heuristic pricing 的 top-K seed 选择和 accepted move 局部 profile 重建。这些改动仍然是当前最值得保留的性能优化。

验证方面，撤回后重新做了 `HEU` 相关 focused `javac` 编译，并重跑 `HEU.OutsourcingMoveConsistencyTest`，结果通过，`checked=10974`。当前判断是先维持启发式初始解主流程的简单性，把后续性能工作继续集中在 pricing 链路上。
