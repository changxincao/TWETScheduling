# double 比较检查记录

本次检查的目标是找出代码里直接用 `< / <= / > / >= / == / !=` 比较 `double` 的位置。这里不把所有符号都当成问题：循环下标、数组索引、泛型、测试 oracle、绘图坐标循环、已经显式写了 `Math.abs(... ) <= eps` 的校验，以及 `Utility.compareLt/compareLe/...` 自身内部实现，都不属于需要优先修改的范围。

当前真正需要优先处理的是主流程里的时间、成本、上界、reduced cost、LP 变量值等比较。总体看，`PiecewiseLinearFunction` 主体大部分已经使用 `Utility.compare...`，剩下的主要问题分散在启发式邻域、旧 BPC 参考代码、新 TWETBPC 框架和少量数据/模型工具里。

## 需要优先改的地方

1. `src/HEU/Solution.java`

这里有几处成本和上界比较仍然直接用符号。例如 `formerShift + p + s > CmaxH`，以及三段拼接里 `cost12 + b3_LB >= Utility.curUpperBound`、`cost23 >= Utility.curUpperBound` 等。这些都是启发式剪枝或可行性判断，建议统一改成 `Utility.compareGt(...)` 或 `Utility.compareGe(...)`。尤其是剪枝判断，裸比较可能在边界处把本来应保留的候选误剪掉。

2. `src/HEU/Move.java`

`totalTime > data.CmaxH` 这种总时长可行性判断仍然是裸比较，建议改成 `Utility.compareGt(totalTime, data.CmaxH)`。同文件里很多 `==` 实际是整数下标判断，不需要改；重点只看时间、成本和 bound。

3. `src/HEU/EngineALNS.java`

`dist < bestDist`、`delta < bestDelta`、`regret > bestRegret` 属于 move 选择和插入评价，变量是 double。建议改成 `Utility.compareLt(...)` / `Utility.compareGt(...)`。这类地方不一定导致不可行，但会影响 tie-breaking 和细微改进判断。

4. `src/HEU/SchedulerForReleaseNoWait.java`

这里有 `currentCompletionTime < minCompletionTime`、`currentCompletionTime == minCompletionTime`、`earliestPossibleStartTime < bestStartTimeForSelected`、`earliestPossibleStartTime == bestStartTimeForSelected`、`candidateMakespan < bestMakespanFound` 等。等号比较尤其需要改成 `Utility.compareEq(...)`，大小比较也建议统一用 `Utility.compareLt(...)`。如果这个启发式后续还会用，建议作为一组单独修。

5. `src/Basic/Data.java`

`if (improvedCmax < CmaxH)` 建议改成 `Utility.compareLt(improvedCmax, CmaxH)`。这是全局时间上界更新，虽然数据大多是整数，但类型是 double。

6. `src/Basic/TimeIndexModel.java`

`if (cost != 0)` 建议改成 `!Utility.compareEq(cost, 0)`。`last.completion <= task.start + Utility.EPS` 已经手动带 EPS，语义上可以保留；如果追求统一，也可以换成 `Utility.compareLe(last.completion, task.start)`。

7. `src/Common/PiecewiseLinearFunction.java`

主体已经基本正确。当前还看到 `while (head != null && head.intercept >= Utility.big_M)` 这类比较，建议改成 `Utility.compareGe(head.intercept, Utility.big_M)`，保持风格一致。其他 `cur != null`、`cur.next != null` 是对象空判断，不是 double 问题。

8. `src/Common/ETConverter.java`

`stdDev <= 0.0`、`value < lowerBound || value > upperBound` 是数据生成/截断逻辑里的 double 比较，建议改成 `Utility.compareLe(...)`、`Utility.compareLt(...)`、`Utility.compareGt(...)`。这不是核心求解逻辑，但改动简单。

## BPC 相关需要注意

旧 `src/BPC/...` 参考代码里还有大量 reduced cost、LP 变量值和阈值比较，例如 `reduce_cost < node.reduce_cost`、`lb.m_reduced_cost < tolerance`、`vars_value[i] > var_threshold`、`value < -zero_tolerance` 等。这部分如果只是作为参考代码暂时不运行，可以先不动；如果后续要复用到 TWET 的 BPC 里，应统一改成带容差的比较。

新 `src/TWETBPC/...` 也有一些需要改的地方：`ArcBrancher` 里的 `frac <= tolerance`、`score < bestFrac`，`Node.compareTo` 里的 `pseudoCost < other.pseudoCost`，`Tree` 里的 `objectiveValue < incumbentCost` 和 `objectiveValue >= incumbentCost - tolerance`，以及 `TWETMasterSolution.getArcValue(...) > 0.0`。这些位置虽然目前多是框架/placeholder，但后续接真实 LP 后最好统一用容差比较，避免分支、剪枝和整数性判断受浮点误差影响。

## 可以暂时不管的地方

测试类、绘图类、`saved` 备份代码、`PiecewiseLinearFunctionArray` 旧数组版本，以及已经显式用 `Math.abs(...) <= 1e-6` 或 `+ Utility.EPS` 的输出验证代码，可以暂时不作为第一批修改对象。后续如果要做全仓库风格统一，再单独处理。

当前建议的修改顺序是：先改 `HEU/Solution.java`、`HEU/Move.java`、`HEU/EngineALNS.java` 和 `Common/PiecewiseLinearFunction.java` 这些会影响当前启发式与分段函数判断的地方；再改 `SchedulerForReleaseNoWait.java`、`Data.java` 和 `TimeIndexModel.java`；最后处理 BPC/TWETBPC 框架里的 LP/reduced cost 比较。
