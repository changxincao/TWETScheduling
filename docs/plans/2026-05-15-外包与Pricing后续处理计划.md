# 外包、setup cost 与 pricing 函数语义后续处理记录

## 1. 背景

当前启发式主流程已经基本接入非零 setup time，分段线性函数这一层也已经围绕 forward/backward labeling 做了方向化处理。但后续如果要继续靠近 `parallel_machine_scheduling_with_due_window.pdf` 里的问题设定，还需要处理三类新内容：一是 pricing 中带 `M` 段函数的比较和运算语义；二是外包决策；三是 setup cost 和任务硬时间窗。这里先不直接改代码，只把后续要做的事和当前判断记录下来。

文档里的基本设定是：任务要么在内部同质并行机上加工，要么外包。外包任务不进入任何内部机器序列，因此不产生内部 setup cost，也不产生内部 due-window penalty；外包成本由外包任务集合决定，典型写法是先累加任务的 baseline outsourcing cost，再通过一个非降凹函数 `G(B(O))` 得到总外包成本。内部加工时，setup time 影响可行完工时间，setup cost 则直接进入目标函数。

## 2. Pricing 中 `M` 段的语义

后续做 pricing 时，可能会遇到函数中存在 `M` 段的情况。这里的 `M` 本质上表示某段时间不可行或已经被剪掉，而不是一个普通可比较的真实成本。如果两个函数都有 `M` 段，再经过 `add`、`merge` 或 dominance 比较，可能出现 `M + a`、`M + b` 这种数值差异。数学语义上它们都应当是“不可行状态”，但如果代码只按普通 double 值比较，就可能出现一个本来应当占优的 label，因为某段是 `M + 100` 而另一个是 `M + 10`，导致完整占优判断失败。

当前判断是：如果后续先不启用 partial dominance，只做完整 label dominance，那么 label 函数中一般不应该主动产生大面积 `M` 段。完整 dominance 只是判断一个 label 是否整体支配另一个 label，不需要把被支配的部分函数区间挖掉，因此对 `M` 段的依赖较弱。真正容易引出这个问题的是 partial dominance：某些时间区间被另一个 label 覆盖后，会被打成 `M` 段，然后这些 `M` 段继续参与后续扩展和比较。

因此后续如果启用 partial dominance，需要额外加一个稳定处理。简单做法是把足够大的数统一视为 `M` 状态，而不是继续保留 `M+a` 的细微差别。例如在函数操作或 dominance 后做一次 post-process：如果某段函数值全程大于某个阈值，如 `M/2`，就统一压回标准 `M` 段；如果一个 label 的剩余所有段都已经是这种 `M` 状态，就可以直接判定该 label 没有继续扩展价值。这个处理不一定第一版就要实现，但在 exact pricing 中不能长期依赖“`M+a` 还能像普通成本一样比较”的假设。

## 3. 外包决策的接入思路

外包可以直观理解成一台特殊机器，但它和普通机器有明显区别。普通机器的成本依赖任务顺序，因为 setup time、setup cost 和完工时间惩罚都和顺序相关；外包任务不进入内部序列，同一个外包集合的不同排列没有意义，成本只依赖集合本身。因此从算法实现看，把外包强行塞进现有 `machines[]` 序列数组虽然表面统一，但会让很多邻域算子和分段函数评价背上不必要的分支判断。

更稳的做法是单独维护外包集合，而不是把它当作普通机器数组里的最后一台机器。这样 ALNS 的 destroy/repair 可以允许任务在“内部机器”和“外包集合”之间移动；VND 如果暂时只优化内部序列，也可以先把外包集合作为当前解的一部分固定下来。外包集合的增删成本可以做到比普通机器更简单，因为不需要重新计算序列函数，只需要维护外包 baseline cost 的累计值和外包成本函数值。

第一版可以先把外包成本函数实现成一个单调递增的连续分段线性函数。这样既能表达外包成本随外包规模增加而增加，也为后续加入凹折扣留接口。需要提前设计的数据包括：每个任务的 baseline outsourcing cost、外包成本函数 `G` 的分段表示、当前解中的外包任务集合、当前外包累计 baseline cost，以及支持 O(1) 或近似 O(1) 的加入/移除增量计算。后续如果要精确支持非线性凹函数，只要外包集合成本仍然只依赖累计 `B(O)`，增量评价仍然比较直接。

## 4. setup cost 的接入位置

当前启发式已经处理 setup time，但文档设定里还有 setup cost `kappa_ij`。两者作用不同：setup time 进入时间递推，影响完工时间和可行性；setup cost 不改变时间，只是序列中每条弧的固定成本，需要直接加到目标值里。

后续要改的地方主要有三类。第一，数据结构需要增加 setup cost 矩阵，包括虚拟起点到任务的 `kappa_0j` 和任务间的 `kappa_ij`。第二，机器序列成本计算时，除了分段函数算出来的时间惩罚，还要加上该序列所有相邻弧的 setup cost。第三，所有邻域增量评价都要同步更新 setup cost 的差值；这部分可以类比现在 setup time 的弧替换逻辑，但它不需要参与 `shiftX` 或 `Cmax` 验证，只进入 `deltaCost`。

这里需要注意封装。setup time 和 setup cost 都是弧属性，但前者影响 duration，后者影响 objective。后续最好单独提供弧时间和弧成本的工具函数，避免在每个算子里散落 `s[i][j] + p[j]`、`kappa[i][j]` 之类的手写逻辑。这样后续如果外包、虚拟起点或特殊任务规则变化，也更容易统一修改。

## 5. 任务硬时间窗与 `setDomain`

文档中每个 job 的完成时间可能有一个有效时间窗，窗外可以视为不可行。当前启发式里主要使用软时间窗惩罚函数；如果后续要加入硬时间窗，可以先把它作为一种加强约束，而不是一开始就改变所有邻域逻辑。

直接风险在于 `add` 目前按公共定义域相加。如果 job 成本函数被 `setDomain(a,b)` 物理裁成 `[a,b]`，而 label 函数仍然是 `[x,T]`，那么相加后的函数定义域可能被截短，后续 `mergeMinimum`、dominance 或 forward/backward 约束就可能不再满足我们前面定下的右端到 `T` 或左端到 `0` 的结构要求。因此不能简单把 job 函数窗口外的部分删掉以后就直接参与 `add`。

更合适的做法是使用我们已经讨论过的 `setDomain(start, end, true)` 语义：窗口内保留原函数，窗口外不物理删除，而是填成 `big_M`。对于 forward label，要特别保证右侧仍然能延伸到全局 `T`，即 `[b,T]` 这段用 `big_M` 表达不可行；对于 backward label，则要保证左侧仍然能覆盖到 `0`，即 `[0,a]` 这段用 `big_M` 表达不可行。这样函数结构仍然连续，`add` 可以继续按当前公共区间扫描，后续 normalize 再按 forward/backward 方向把不可达或无效段处理掉。

关于 `H_ij` 的问题，第一版不建议让硬时间窗依赖序列位置或前驱后继，否则会把 job 函数本身变成“弧相关函数”，后续 `add` 和缓存会复杂很多。可以先取一个与 job 相关的统一硬时间窗，例如对所有可能前驱/外包设定取一个安全的聚合窗口。你提到的“取 max 的 j”可以理解为先用保守窗口预设 job 函数域，避免因为过细的 `H_ij` 依赖把函数操作层搞复杂。外包相关的窗口只需要用于外包成本或外包可行性判断，不需要引入 dual 项来定义这个硬窗口。

## 6. 后续实现顺序建议

当前更稳的顺序是先做数据结构，再做成本评价，最后再接入复杂 dominance。第一步先补外包参数、setup cost 参数和硬时间窗参数的数据结构，不急着改所有算法。第二步在 `Solution` 层把“内部机器序列成本 = 时间惩罚 + setup cost”封装清楚，并给外包集合单独建成本计算入口。第三步再让 ALNS 的删除/插入支持任务在内部机器和外包集合之间转移。第四步才考虑 BPC pricing 中 partial dominance 的 `M` 段标准化问题。

当前暂定结论是：不采用 partial dominance 的第一版 pricing，可以先尽量避免 label 函数内部出现大量 `M` 段；如果后续为了效率启用 partial dominance，就必须把 `M` 状态标准化作为一个明确步骤。外包不要急着塞进普通机器数组，先按单独任务集合实现更干净。setup cost 不影响时间函数，只影响目标增量。硬时间窗不要物理裁断函数定义域，应该用 `big_M` 填充窗外以保持 forward/backward 函数结构。

## 7. 当前优先级判断

从实现顺序看，第一步建议先做 setup cost，而不是先做外包或 pricing 的 partial dominance。原因是 setup cost 的语义最清楚，改动范围也最可控：它只是在现有弧 `i -> j` 上多一个固定成本，不改变时间递推、不改变分段函数定义域，也不改变机器序列结构。先把这件事做好，可以顺手把“弧时间”和“弧成本”封装出来，后面外包、BPC column cost、输出验证都会用到这个基础。

第二步建议做硬时间窗的函数域处理，但先只做函数层能力和数据入口，不急着强行打开所有约束。这里的关键不是窗口公式本身，而是把 `setDomain(start,end,true)` 这类“窗外填 `big_M`、不破坏 forward/backward 定义域结构”的操作稳定下来。这个能力对后续 exact pricing 更重要，因为 pricing 里一旦 job 函数被物理裁短，`add/merge/dominance` 的前提就会被破坏。

第三步再做外包。外包看起来只是多一种任务去向，但它会改变 `Solution` 的状态表示、ALNS 的 removal/repair、目标函数计算、输出验证和后续 column 解释。它不是一两个公式能补完的，最好等 setup cost 和硬时间窗这两个基础成本层都清楚以后，再单独设计“内部机器序列 + 外包集合”的解结构。

第四步才处理 pricing 中 `M` 段标准化和 partial dominance。这个问题非常重要，但它依赖真正的 label 结构、dominance 策略和 reduced cost 计算。现在提前写太细容易变成空设计。短期只需要保留原则：第一版 pricing 尽量先用完整 dominance，避免过早启用 partial dominance；如果后续启用 partial dominance，就必须把 `M+a` 统一回不可行状态，并检测全 `M` label 是否应直接丢弃。

因此当前最合理的执行顺序可以写成：`setup cost -> 硬时间窗函数域能力 -> 外包集合结构 -> pricing 的 M 状态与 partial dominance`。如果中途优先级变化，例如先要跑带外包的启发式结果，也可以把外包提前，但那会同时牵动 ALNS 和解验证，工作量会明显更大。

## 8. setup cost 的具体修改面

参考 Ibaraki et al. 2008 的 VRP 设定，`d_ij` 是固定 traveling cost，进入 `dsum(sigma)`；`t_ij` 是 travel time，只进入时间约束和函数平移。对应到当前 TWET 项目，setup cost 应当对应 `d_ij`，setup time 对应 `t_ij`。因此 setup cost 不能混进 `shiftX(s_ij+p_j)` 这条时间递推里，而应该作为弧上的固定目标成本，和时间惩罚函数算出来的最优 penalty 相加。

第一处是 `Data`。需要增加一个与 `s` 同维度的 setup cost 矩阵，例如 `setupCost` 或 `kappa`，默认值为 0，从而兼容现有数据。读取格式上建议追加一个可选块，例如 `SETUP_COST` 后跟 `(n+1)*(n+1)` 矩阵；如果没有这个块，则保持全 0。后续数据生成器也应同步生成该矩阵。为了避免后面到处手写，`Data` 里最好补两个清晰接口：`getArcDuration(i,j)` 返回 `s[i][j] + p[j]`，`getSetupCost(i,j)` 返回 `kappa[i][j]`。其中前者用于时间递推，后者只用于目标成本。

第二处是 `Solution` 的机器成本定义。当前 `cost[m]` 表示机器 `m` 的时间惩罚成本，`curCost` 是所有机器时间惩罚之和。接入 setup cost 后，`cost[m]` 最好继续表示完整机器成本，即“最优时间惩罚 + 序列 setup cost”。这样外部 ALNS、VND、BPC 初始列都不需要再额外理解两个成本字段。对应地，`calCost(m)` 在用 forward 函数得到最优时间惩罚后，还要加上该机器序列的 setup cost；空机器仍为 0。

第三处是固定弧成本的高效计算。最直接的方案是不要把 setup cost 写进 `PiecewiseLinearFunction`，而是在 `Solution` 里单独维护 setup cost 前缀缓存，类似现在的 `cumDurationNormal/cumDurationReverse`。这样分段函数仍然只负责时间惩罚，setup cost 作为弧固定成本在拼接结果外面补上。需要的接口包括：整条机器序列的 setup cost、正常方向某段内部 setup cost、反向某段内部 setup cost，以及给定几段拼接时的新桥接弧 setup cost。这样就能在邻域评价中 O(1) 或 O(片段长度) 地得到新序列固定成本，避免每个候选都重扫整条机器。

第四处是 `merge2Segments` 和 `merge3Segments` 的使用语义。当前它们返回的是分段函数拼接后的最优时间惩罚，不包含固定弧成本。接入 setup cost 后，不建议直接把 setup cost 塞进这两个函数内部，除非同时重写所有 segment 函数的定义。更稳的做法是保留它们的含义不变，然后在调用处计算“新时间惩罚 + 新 setup cost - 原完整成本”。也就是说，`merge2Segments` 仍然只管时间，调用它的 `evalDelta` 才负责把桥接弧的 setup cost 和内部段 setup cost 加进去。

第五处是 `Move.java` 的五类邻域。`IOptOperator`、`TwoOptOperator`、`TwoOptStarOperator`、`PathInsertOperator`、`CrossExchangeOperator` 现在的 `evalDelta` 都是用 `merge2Segments/merge3Segments` 得到新机器成本，再减去 `s.cost[m]` 或两台机器的 `originCost`。接入 setup cost 后，这些 `costM` 变量必须明确变成完整成本，而不能继续只接收 merge 返回值。每个算子的改法本质一样：先用现有分段函数拼接得到新时间惩罚，再用 setup cost 缓存计算新序列的固定弧成本，最后二者相加。

第六处是 `EngineALNS`。`evaluateInsertionCost(...)` 当前计算的是插入后机器时间惩罚减去原 `s.cost[m]`；接入后应改成插入后完整机器成本减去原完整成本。`WorstRemoval.computeIndividualPenalties(...)` 同理，删除某个任务后用两段拼接得到的是删除后时间惩罚，还要加删除后序列的 setup cost。`ShawRemoval.taskDistance(...)` 可以选择把 setup cost 也加入相似度，但这不是正确性必需项，只是搜索策略问题，可以第二阶段再调。

第七处是校验和列评价。`Move.testSequence(...)` 现在是直接重算一条序列的时间惩罚，用于对拍；接入 setup cost 后，它也应该返回完整成本，否则测试会把正确的新实现误判为不一致。`TWETColumnEvaluator` 通过 `scratch.calCost(0)` 计算列成本，如果 `calCost` 已经改成完整成本，这里会自然继承。输出验证器后续也要把 setup cost 加入重算目标，否则模型/启发式输出的目标值会和验证目标不一致。

当前建议的实现路线是：先加 `Data.setupCost` 和读取默认值，再在 `Solution` 里加 setup cost 缓存与序列成本 helper，最后逐个改 `EngineALNS` 和 `Move` 的 `evalDelta`。这条路线改动比“把 setup cost 直接写进所有分段函数”更安全，因为它不改变 `PiecewiseLinearFunction` 的语义，也不会影响我们前面已经稳定下来的 forward/backward 函数操作。后续真正实现时，需要特别写测试：当 setup cost 全为 0 时，所有结果应与当前版本一致；当 setup cost 非零、setup time 为 0 时，算法应只因为弧成本改变搜索选择，而不改变时间可行性。

2026-05-15 继续讨论后，对“setup cost 是否直接加进函数”做一个修正判断。严格说，setup cost 是时间无关的固定弧成本，完全可以通过函数纵向平移加入累计函数。比如从任务 `i` 扩展到任务 `j` 时，时间上仍然做 `shiftX(s_ij+p_j)`，目标上再把 `kappa_ij` 作为常数加到函数值里。这样序列内部的 setup cost 会自然跟着 forward/backward 函数传播，整体形式也更接近 Ibaraki 2008 中 `d_ij + time penalty` 的统一评价函数。

但需要注意，不能简单理解成“把 setup cost 全部塞进 `PiecewiseLinearFunction` 后就不用管了”。当前局部搜索大量使用的是两段、三段预处理函数拼接。每个预处理段内部的 setup cost 可以在构造该段函数时加入；但是段与段之间新连出来的桥接弧，例如 `pre -> firstOfMiddle`、`lastOfMiddle -> post`，并不属于原来任何一个预处理段的内部弧。这个桥接弧的 setup cost 仍然必须在拼接时显式加入，除非把 `merge2Segments/merge3Segments` 的接口改成同时接收桥接弧成本。

因此更合适的实现不是“完全外置 setup cost 缓存”，也不是“完全藏进函数里”，而是折中：函数本身表示“该段内部的时间惩罚 + 段内部 setup cost”；跨段拼接时，`merge2Segments/merge3Segments` 或其调用处额外加桥接弧的 setup cost。这样优点是单个序列的 `calCost(m)` 和预处理函数更统一，缺点是 merge 接口或调用处需要更明确地区分 `shift` 和 `bridgeCost`。后续实现时，优先考虑给 `merge2Segments` 增加一个 `bridgeCost` 参数，给 `merge3Segments` 增加 `bridgeCost1/bridgeCost2` 参数，保留旧接口作为默认桥接成本 0，避免一次性破坏所有调用。
2026-05-15 进一步明确 setup cost 的实现规则：跨段拼接时确实应当和 setup time 一一对应处理。也就是说，凡是当前代码中因为新弧 `i -> j` 使用 `setup time s_ij` 做 `shiftX(s_ij + p_j)`、`Cmax` 验证或拼接时间修正的地方，后续都应该同步考虑同一条弧上的 `setup cost kappa_ij`。区别只在于，`s_ij` 改变时间轴，进入横向平移和可行性判断；`kappa_ij` 不改变时间，只进入函数值的纵向常数偏移。这样理解后，setup cost 不需要单独设计一套完全独立的逻辑，而是作为“弧属性的另一部分”跟随现有 setup time 的弧替换逻辑一起接入。

因此后续实现时可以按这条规则检查所有位置：如果某个地方新建、删除或替换了一条 setup time 弧，就必须同步更新对应 setup cost；如果某个地方只是处理任务自身 processing time 或 penalty function，则不应该混入 setup cost。这个规则比“单独维护 setup cost 增量”更清楚，也更符合当前代码的拼接结构。

2026-05-15 对 setup cost 在函数内部的表示再做一个实现取舍。理论上 setup cost 是固定弧成本，可以在 `PiecewiseLinearFunction` 外面额外维护一个函数级常数项，从而避免每次都遍历所有 segment 修改截距。但这样会让函数比较、占优、merge、输出和调试都多一个隐含状态，后续做 label dominance 时也容易漏掉这个常数项。当前先不引入这层优化，采用更直接的做法：当某条弧的 setup cost 需要加入函数时，就把这个常数加到函数所有 segment 的截距上。这个操作是 O(段数)，但语义最直观，所有后续函数操作仍然只需要比较显式函数值。等后续 profiling 证明这一步成为瓶颈，再考虑增加函数级 lazy constant。

2026-05-15 已按上述规则完成第一版接入。数据层在 `Data` 中增加 `setupCost[i][j]`，旧数据没有 `SETUP_COST` 块时默认全 0；如果数据文件在 `SETUP` 后追加 `SETUP_COST`，则按同样的 `(n+1)*(n+1)` 矩阵读入。分段函数层增加 `shiftYInPlace(delta)`，用于把固定弧成本直接加到所有 segment 的截距上。启发式层中，forward/backward 函数递推、二维/三维预处理函数以及 `Move.testSequence` 都在每次使用弧 `i -> j` 的 setup time 时同步加入 `setupCost[i][j]`。跨段拼接处通过 `merge2Segments(..., bridgeCost)` 和 `merge3Segments(..., bridgeCost1, bridgeCost2)` 显式补入新桥接弧成本，旧接口保留并默认桥接成本为 0。

同时，`EngineALNS` 的插入评价和最坏移除评价也同步改成完整成本口径；`ArcFlowModel` 和 `ATIParallel` 的目标函数把被选中弧的 setup cost 加入目标；模型解验证器在重算目标时也加入每个任务前驱弧的 setup cost。需要注意，当前 `TimeIndexModel` 本身没有显式顺序弧，原来就不严格表达 sequence-dependent setup time，因此这次没有把 setup cost 强行塞进去。后续如果要比较带 setup cost 的精确模型，优先使用 arc-flow 或 ATI 这类带弧的模型。

## 9. 硬时间窗和 H_ij 的实现分层

2026-05-15 重新阅读 `parallel_machine_scheduling_with_due_window.pdf` 后，需要把“任务预设硬时间窗”和 pricing 里的 `H_ij` 区分开。文档中的 internal pricing 标签函数定义为 `Dom(f_L)=[ell_L,T]`，扩展 `i -> j` 时先有最早完成时间 `ell_L+s_ij+p_j`，然后利用 profitable completion window `H_ij=[h_ij,\bar h_ij]` 进一步缩小可选完成时间。这里的 `H_ij` 不是一个纯 job 常量，它依赖前驱 `i`，还依赖 `Gamma_ij = B_ij + min{pi_j,b_j}`，其中 `pi_j` 是主问题覆盖约束的 dual，`b_j` 是外包 baseline cost，`B_ij` 是删除 job `j` 后可能带来的 setup-cost advantage 上界。因此它更适合放在 pricing 扩展阶段动态使用，而不是在数据预处理阶段一次性写死到 job 成本函数里。

当前更稳的实现分层是：数据/启发式层只维护一个 job 级的粗硬窗，表示这个任务物理上或业务上允许的 completion 范围。这个粗窗口如果要作用到分段函数，不能用两参数 `setDomain(start,end)` 物理裁掉窗口外定义域，而应使用 `setDomain(start,end,true)` 把窗口外写成 `big_M` 段，并保留累计 label 函数的固定端点结构。这样 forward 仍能保持右端到 `T`，backward 仍能保持左端到 0，`add()` 后也不会因为公共定义域被截短而破坏后续 merge/dominance 的输入契约。

pricing 阶段再单独计算更窄的 `H_ij`。扩展 `i -> j` 时，可以先根据 `ell_L+s_ij+p_j` 和 `h_ij` 得到新 label 的 `ell`，如果 `ell > \bar h_ij` 就直接丢弃扩展；否则在构造本次 job `j` 的成本函数时，用 `setDomain(max(ell, h_ij), \bar h_ij, true)` 或等价窗口化操作把 `H_ij` 外的完成时间写成 `big_M`。随后再做 `shiftX + add + minimizePrefixInPlace()`。这里不建议把 `H_ij` 提前预处理进 `Data.penaltyFunction[j]`，因为 dual 每轮变、`B_ij` 依赖前驱，预处理会要么过宽没有效果，要么过窄误删可行定价状态。

如果要在 pricing 之前先做一层预处理，可以只做“不含 dual、不含给定前驱 i 的保守窗口”。具体说，对每个任务 `j` 可以预先对所有可能前驱 `i` 取一个 `B_j^{max}=max_i B_ij`，用这个最大 setup-cost advantage 给出一个不会误删的粗窗口。这个窗口的含义不是论文里的精确 `H_ij`，而是一个安全的上层过滤：它假设 `j` 前面可能接任何 `i`，因此只能排除那些无论前驱是谁都明显不值得完成的时间段。真正进入 pricing 扩展时，仍然要用当前前驱 `i` 和当前 dual `pi_j` 重新计算 `H_ij`，得到更窄的窗口。这样分层以后，预处理只负责减少明显无效的函数区域，不会把依赖 dual 和前驱的精细剪枝提前写死。

外包成本在这个逻辑里主要用于 `min{pi_j,b_j}` 的阈值。第一版如果还没有完整外包 pricing，可以先把 `b_j` 作为每个 job 的 baseline outsourcing cost 存起来，用它参与 `Gamma_ij` 的计算；不要把外包成本直接加进内部加工的 job 惩罚函数。内部序列列的 reduced cost 仍由内部时间惩罚、setup cost 和 dual 项组成；外包集合成本 `G(B(O))` 是另一类列/决策的成本，应该在外包 pricing 或外包集合评价里处理。

因此，这一块的推荐实现顺序是：先给 `Data` 增加 job 级粗窗口字段，默认 `[0,T]`，并保留 `penaltyFunction[j]` 的原始完整定义域；再提供一个 helper，在需要时返回“应用窗口后的 job 函数副本”。启发式如果暂时不启用硬窗，可以不调用这个 helper。BPC pricing 第一版扩展时，再根据当前 dual、前驱 `i`、外包 baseline `b_j` 和 setup-cost advantage `B_ij` 计算 `H_ij`，只对本次扩展的 job 函数副本做窗口化。这样既保留了论文中的 `H_ij` 精细剪枝，也不会提前把底层函数定义域结构搞坏。

2026-05-15 后续实现时先采用更直接的做法：粗硬窗直接作用到 `Data.penaltyFunction[j]`，但必须使用 `setDomain(left,right,true)`，即窗外写成 `big_M`，不物理删除定义域。这样当前启发式仍然可以直接使用 `data.penaltyFunction[j]`，不会额外引入 helper 或调用分支；同时因为右端仍保持到 `CmaxH/T`，不会破坏前面已经稳定下来的 forward/backward 函数契约。真正的 `H_ij` 仍不在当前阶段接入，后续 pricing 扩展时再按当前前驱和 dual 动态计算。

代码层现在在 `Data` 中增加了 `outsourcingCost`、`maxSetupCostAdvantage`、`hardWindowStart/hardWindowEnd`。`outsourcingCost[j]` 默认设为 `big_M`，表示当前没有真实外包 baseline cost 时不主动收缩窗口；如果后续数据里给出 `b_j`，可以先填入 `outsourcingCost[j]`，再调用 `setPreprocessedHardWindows()` 和 `setPenaltyFunctions()` 重新生成窗外为 `big_M` 的 job 成本函数。粗窗计算使用 `B_j^{max}=max_i B_ij`，其中 `B_ij` 由 setup cost 矩阵给出删除 `j` 后可能获得的弧成本优势。这个 `max_i` 是保守处理，窗口只会排除所有前驱下都明显无效的完成时间，不替代 pricing 中更窄的 `H_ij`。

2026-05-16 继续补了两个实现边界。第一，数据读取允许在 `SETUP` 和 `SETUP_COST` 后追加 `OUTSOURCING_COST` 向量，长度可以是 `n` 或 `n+1`；长度为 `n` 时按任务 `1..n` 填入，虚拟点 0 仍为 0。这样后续如果数据生成器给出 baseline outsourcing cost，粗硬窗可以在 `Data` 构造阶段自动生效。第二，零长度粗窗不再额外打标记，也不再因为外包成本是否为 0 走特殊分支。外包成本为 0 本身不是当前数据里的常见情形，把它单独引入 `outsourceOnly` 标记会让数据层承担尚未实现的外包集合逻辑，反而容易把问题复杂化。

当前采用的统一处理是：`Data` 只记录计算出来的 `hardWindowStart[j]` 和 `hardWindowEnd[j]`，哪怕它们退化成 `[d,d]`；真正把窗口作用到分段函数时，由三参数 `setDomain(start,end,true)` 处理。如果窗口没有正长度交集，该函数会先把单点窗口扩成 `[d-\epsilon,d+\epsilon]`，再把窗口外写成 `big_M` 段。当前时间数据基本是整数，先取 `\epsilon=0.1`，目的不是改变模型含义，而是避免把 `[d,d]` 这种长度为 0 的点函数继续传播到 `add/merge/normalize` 中。这样既保留了“硬窗外不可行”的语义，也维持了函数定义域端点稳定；后续 pricing 里如果 `H_ij` 因 dual 和外包阈值退化成单点，也可以复用同一套窗口化逻辑。

`B_ij` 也已从“每个 job 临时扫一遍求 `max_i`”改为矩阵预处理。现在 `Data.setupCostAdvantage[i][j]` 存储论文中的 `B_ij=max_k [kappa_ik-kappa_ij-kappa_jk]^+`，`maxSetupCostAdvantage[j]` 则由该矩阵按列取最大值得到。这样当前粗硬窗仍然使用 `max_i B_ij`，后续 pricing 扩展 `i -> j` 时可以直接读取 `setupCostAdvantage[i][j]`，不用每次扩展再扫描所有后继 `k`。当前矩阵只按任务点 `1..n` 扫后继；如果论文里的终点 `n+1` 的 setup cost 设为 0，它只会贡献非正项，不影响 `max(.,0)`，因此暂时不需要额外扩展数据维度。

关于“惩罚函数是否只保留硬时间窗内定义域”，当前结论仍是不建议。单个 job 函数如果物理只保留 `[a,b]`，与累计 label 函数做 `add()` 时会因为公共定义域被截断而导致输出右端不再到 `T`，这会破坏后续 `merge/dominance/normalize` 的输入约定。更稳的做法仍然是：函数对象保留 `[0,CmaxH]` 或 label 所需的固定端点，硬窗外用 `big_M` 表达不可行。这样多处理一些 `M` 段，但换来函数结构稳定；后续如果要加速，应从 pricing 扩展层提前跳过不可行窗口，而不是物理裁短底层函数。

后续 TODO 是把这套窗口逻辑真正接到 pricing 迭代里。每次 RMP 求解后 dual 都会更新，因此 `H_ij` 也应该按当前迭代重新计算，而不是复用数据层的粗硬窗。具体流程应是：定价开始时读取本轮覆盖约束 dual `pi_j`；扩展 label 的每一条弧 `i -> j` 时，从 `setupCostAdvantage[i][j]` 取 `B_ij`，结合外包 baseline `b_j` 计算 `Gamma_ij = B_ij + min{pi_j,b_j}`，再由 job 的 earliness/tardiness 斜率反推出本轮、本前驱下的动态 completion window `H_ij=[h_ij,\bar h_ij]`。如果当前 label 的最早完成时间已经超过 `\bar h_ij`，直接跳过扩展；否则只对本次扩展使用的 job 函数副本调用 `setDomain(max(ell,h_ij), \bar h_ij, true)`。这个窗口是每轮、每条前驱弧动态变化的，不能写回 `Data.penaltyFunction[j]`，否则下一轮 dual 变化后会污染原始函数。

## 10. 硬时间窗对启发式剪枝的可能用法

2026-05-16 查了 Vidal 等人在 VRP time-window 启发式里的 route-evaluation 思路。其核心不是在局部搜索里反复完整重算时间窗，而是给每个连续子序列维护可拼接属性，然后通过 `Init/Forw/Back/Eval2/EvalN` 评价 move。对于硬时间窗 VRP，典型子序列属性包括总 travel/service 时间、最早完成时间、最晚可行开始时间和可行性标记；两个子序列拼接时，只要用桥接时间检查前一段最早完成是否能落到后一段最晚开始之前，就能常数时间判断可行性。Vidal 2014 的 unified local search 也是这个结构：局部搜索 move 被看成有限个子序列重组，预处理 forward/backward 子序列信息，修改路线后再更新这些信息。

当前项目其实已经有一部分同构结构：`Solution` 里维护 forward/backward 分段函数、一维/二维/三维子序列函数，`Move` 里的 `merge2Segments/merge3Segments` 就是在做 Eval2/EvalN 式拼接评价。硬时间窗已经通过 `Data.penaltyFunction[j].setDomain(..., true)` 写进 job 成本函数，窗外为 `big_M`，所以当前所有基于分段函数的 move 评价都会感知硬窗；若拼接后没有有效完成时间，`add/findMinimal` 最终会返回 `curUpperBound` 或 `big_M`。这说明硬窗现在已经能影响启发式评价，但它主要是在“算完函数以后发现不可行或很差”，不是在拼接前做强剪枝。

如果要让硬窗真正帮助启发式剪枝，可以加一层轻量摘要，而不是替换现有分段函数。对每个机器子序列，额外维护类似 Vidal 的时间可行区间信息，例如该子序列在当前粗硬窗下的 earliest feasible completion、latest feasible start、duration 和 feasible 标记；对 reverse 子序列也维护同样信息。局部搜索评价两段或三段拼接时，先用这些摘要和桥接 setup time 做快速判断，如果明显无可行完成时间，直接跳过 `merge2Segments/merge3Segments` 的分段函数计算。只有摘要判断可能可行时，再调用现有分段函数得到精确 TWET 成本。这样做的好处是剪枝成本很低，且不会改变现有目标函数口径。

需要注意，这里的摘要只能作为安全剪枝，不能替代分段函数成本。原因是当前问题不是单纯 VRPTW 可行性问题，而是带 earliness/tardiness 分段惩罚、setup time、setup cost，以及后续可能的外包和 pricing dual。Vidal 硬时间窗公式里只要判断可行即可；我们还要找到最优完成时间及对应惩罚，所以最终成本仍应以 `PiecewiseLinearFunction` 为准。比较稳的实现顺序是：先在 `Data` 或 `Solution` 中把每个 job 的粗硬窗 `[hardWindowStart, hardWindowEnd]` 暴露为只读信息；再给一维/二维/三维子序列预处理增加 `earliest/latest/feasible` 数组；最后在各个 move 的 `evalDelta` 里，在进入分段函数 merge 前先做一次摘要可行性检查。

关于“全 M 函数是否会被删除”，当前代码已有部分处理，但不是完整的 all-M 统一判定。`normalizeForward()` 会删除左侧连续 `big_M` 段，如果整条函数都是 `big_M`，会把函数置空；`normalizeBackward()` 会找最后一个非 `big_M` 段，如果找不到也会置空。`merge2Segments/merge3Segments` 会在输入或中间结果 `isEmpty()` 时返回 `big_M/curUpperBound`，并且会用 `curUpperBound` 做下界跳过。但普通 `add()`、`setDomain(..., true)` 和 prefix/suffix minimize 不会主动检测“所有段都是 M 就删除”。这是有意保守的，因为在 pricing 里 `M` 段可能还会受到 dual/固定成本平移影响，不能随便把 `M+a` 当普通数比较。后续如果启用 partial dominance 或大量硬窗剪枝，应增加一个明确的 M 状态标准化工具：例如把全程大于 `M/2` 的段统一压回 `big_M`，如果整条函数都是这种状态，则标记 label/子序列不可继续扩展。这个工具应由 pricing/启发式调用层显式使用，而不是悄悄塞进 `add()`，否则容易破坏当前函数操作的通用性。

2026-05-16 进一步澄清启发式中硬时间窗和 `big_M` 的关系。第一，`normalize()` 里的 M 判断如果后续要改，不应只从 `intercept >= big_M` 改成另一个裸比较，而应抽成明确的 M 状态判断，例如 `isBigMValue(v)=v>=big_M/2`，并最好按 segment 的整段取值判断。用 `big_M/2` 的好处是可以把 `M-a`、`M+a` 这类数值扰动统一视作不可行状态，避免因为固定成本相加、取小或浮点误差导致 M 状态被当成普通成本继续比较。但这类修改不宜直接塞进普通 `add()` 或所有 normalize 里；启发式阶段可以作为剪枝工具，pricing 阶段则必须结合 dual 和 reduced cost 语义谨慎使用。

第二，在不启用硬时间窗时，启发式普通递推基本不会自然产生整段 `big_M`。普通 forward/backward 扩展走的是 `shiftX + add + minimizePrefix/Suffix`，而不是 `normalize()`；`normalize()` 主要在 `mergeMinimum` 和 `updateDominatedIntervals` 这类集合操作后调用。当前启发式局部搜索主要使用预处理好的分段函数和 `merge2Segments/merge3Segments`，不走 label 合并或 partial dominance，因此“全 M 函数被 normalize 删除”不是当前普通启发式的主要风险。

第三，启用粗硬时间窗后，窗外被写成 `big_M`，某些候选拼接可能在所有完成时间上都不可行。按现在实现，这通常会表现为拼接后的函数最小值达到 `big_M/curUpperBound`，从而候选 move 不会被当成有效改进。也就是说，从正确性上看，启发式不一定必须显式判断“这个函数是不是全 M”；它最多多花一些时间做了本可以提前跳过的分段函数运算。当前若只关心解正确性和先跑通流程，可以不急着实现全 M 检测。

第四，如果后续发现硬窗明显增加函数运算开销，再考虑引入 Vidal 式拼接前可行性摘要。这个摘要不是替代 TWET 分段函数，而是在调用 `merge2Segments/merge3Segments` 前做安全过滤：若两个或三个子序列按桥接 setup time 拼起来后，硬窗摘要已经能证明没有任何可行完成时间，就直接跳过；否则仍调用现有分段函数精确评价成本。这一层优化可以减少明显不可行候选的函数运算，但需要给 normal/reverse 子序列都维护 earliest/latest/feasible 等信息，属于性能优化而不是当前正确性修复。

2026-05-16 已先完成 M 状态判断的第一步封装。代码在 `Utility` 中新增 `isBigMValue(value)`，当前阈值取 `big_M/2`，用于把 `M+a`、`M-a` 或浮点扰动后的不可行状态统一识别为 M。`PiecewiseLinearFunction.normalizeForward/normalizeBackward` 以及旧数组版 `PiecewiseLinearFunctionArray.normalize()` 中原来直接判断 `intercept >= big_M` 的地方，已改为调用该封装。这个修改只影响结构清理和剪枝判断，不改变普通目标值比较，也不把 all-M 标准化塞进 `add()`。后续如果需要更严格，可以再把判断从单个 intercept 扩展为整段函数值区间判断。

## 11. Vidal 式硬时间窗可行性检验的适配方案

2026-05-16 进一步分析如果要把 Vidal 式硬时间窗可行性检验接入当前启发式，核心应是增加一套“子序列硬窗摘要”，而不是替换现有 `PiecewiseLinearFunction`。Vidal 的 route-evaluation 框架把局部搜索 move 看成若干连续子序列重组，并为子序列预处理 `Init/Forw/Back` 信息，最后用 `Eval2/EvalN` 判断拼接后的成本和可行性。当前代码已经有相同骨架：一维 `fFunctions/bFunctions`，二维 `fFunctions_hg/bFunctions_hg/fFunctions_gh/bFunctions_gh`，以及三维 `hgl` 函数，正好对应 normal/reverse 子序列信息；`merge2Segments/merge3Segments` 则对应 Eval2/EvalN 式拼接。

如果只做硬窗可行性剪枝，可以为每个子序列额外维护一个轻量对象，例如 `TimeWindowSummary`。单个子序列至少需要记录：`duration`，即该段内部从第一个任务开始加工到最后一个任务完成的最短持续时间；`earliestCompletion`，即在满足段内所有任务粗硬窗时，这段最早能完成的时间；`latestStart` 或等价的 `latestCompletion`，即这段为了仍能满足内部所有硬窗，最晚可以从什么时候开始；`feasible`，表示该子序列本身是否存在某个整体平移后满足所有任务硬窗。normal 和 reverse 序列都要有对应摘要，因为 2-opt、path-insert、cross-exchange 都可能反转片段。

摘要的递推可以先按硬时间窗 VRP 的经典方式做。对序列 `A` 后接任务 `j`，先令 `arrival = earliestCompletion(A) + setup(last(A),j) + p_j`，再取 `completion = max(arrival, hardWindowStart[j])`，若 `completion > hardWindowEnd[j]` 则该拼接不可行，否则更新 `earliestCompletion`。反向摘要则对称地从右往左递推，用 `hardWindowStart/End` 约束前一个任务的最晚可行完成时间。这里用的是当前数据层的粗硬窗，不含 pricing dual；它只负责启发式里判断候选拼接是否明显硬窗不可行。

在 move 评价中使用时，不应改变现在的成本计算流程。两段拼接 `S1 -> S2` 之前，先用 `S1` 的最早完成时间、桥接 setup time、`S2` 的最晚可行开始/完成时间检查是否可能接上；如果已经不可能接上，直接返回 `Utility.curUpperBound` 或跳过候选，不再调用 `merge2Segments`。三段拼接也是一样，先按摘要依次拼接 `S1,S2,S3`，只要某一步摘要不可行就提前返回。摘要判断通过以后，仍然调用现有 `merge2Segments/merge3Segments` 用分段函数精确计算 TWET 成本。这样摘要只做安全过滤，不改变目标值。

需要特别注意两点。第一，当前 `PiecewiseLinearFunction` 处理的是软时间惩罚和最优完成时间，硬窗摘要只知道是否存在可行时间，不能给出 TWET 最优成本，因此不能替代分段函数。第二，摘要必须和现有子序列缓存同步更新：`updateInformationM(m)` 里现在会更新 duration、forward/backward 函数和二维函数，后续若加入摘要，也应在同一位置为该机器重建 normal/reverse 的一维、二维、必要时三维摘要。先实现二维摘要即可覆盖 `merge2Segments` 和大部分跨机器 move；三维可以在 `merge3Segments` 前按二维/一维摘要临时 EvalN 拼出来，不一定一开始就存四维数组。

推荐实现顺序是：先定义 `TimeWindowSummary` 和 `concat(summaryA, summaryB, bridgeTime)`，只做可行性判断；再在 `Solution` 里增加 normal/reverse 的二维摘要缓存，对应现在的 `fFunctions_hg/fFunctions_gh`；最后在 `Move` 的 `evalDelta` 中，在进入分段函数 merge 前调用摘要判断。先不要把这个逻辑接进 `Data.penaltyFunction` 或 `PiecewiseLinearFunction`，因为它是启发式性能剪枝层，不是函数语义层。

### Vidal 式硬窗剪枝的操作方案

后续真正实现时，先把这件事作为启发式性能优化单独推进，不要和分段函数语义、pricing 动态 `H_ij` 混在一起。第一步新增一个轻量数据对象，例如 `TimeWindowSummary`，字段只保留硬窗可行性需要的信息：`firstJob`、`lastJob`、`duration`、`earliestCompletion`、`latestStart` 或 `latestCompletion`、`feasible`。它不记录 TWET 成本，也不记录 setup cost；setup cost 不影响时间可行性，仍由现有分段函数和 bridge cost 逻辑处理。

第二步实现几个静态或工具方法：`single(job)` 负责从 `hardWindowStart[j]`、`hardWindowEnd[j]` 和 `p[j]` 生成单任务摘要；`concat(left,right,bridgeTime)` 负责判断两个子序列用一条 setup time 桥接后是否还有硬窗可行时间；必要时再提供 `reverse` 或直接为反向序列单独构造摘要。这里的 `concat` 只能做安全过滤：返回 infeasible 时必须保证这个拼接确实不可能满足粗硬窗；返回 feasible 只表示可能可行，后面仍要交给 `PiecewiseLinearFunction` 精确算成本。

第三步在 `Solution` 里增加缓存。优先做二维 normal/reverse 摘要数组，对齐现有的 `fFunctions_hg` 和 `fFunctions_gh`：`twSummary_hg[m][h][g]` 表示机器 `m` 上原顺序片段 `[h,g]` 的硬窗摘要，`twSummary_gh[m][g][h]` 表示同一片段反向后的摘要。更新入口放在 `updateFunctions2ForMachine(m)` 附近，和二维分段函数、duration 缓存一起重建。暂时不必上来就做四维 `hgl` 摘要；三段拼接可以在评价时用已有二维/一维摘要临时拼成 EvalN。

第四步接入 move 评价。先从最局部、收益最大的地方做：`merge2Segments` 相关的 `TwoOptStar`、`PathInsert` 中的两段或三段候选，在调用 `merge2Segments/merge3Segments` 之前，先取对应前段、中段、后段摘要，用桥接 setup time 依次 `concat`。如果摘要判断 infeasible，直接返回 `Utility.curUpperBound` 或不保存该候选；如果 feasible，再走现有分段函数逻辑。这样不会改变任何可行候选的成本值，只减少明显不可行候选的函数运算。

第五步验证。先构造小规模硬窗算例，用暴力枚举或现有 `Move.testSequence` 对比：摘要判定 infeasible 的候选，完整分段函数评价也应为 `big_M/curUpperBound`；摘要判定 feasible 的候选，不要求一定最终可改进，但不能因为摘要过滤丢掉一个完整评价可行且有改进的 move。然后再跑一组随机硬窗实例，对比开启/关闭摘要剪枝时最终解成本是否一致或不变差，同时记录 `M2S/M3S Total/Skip` 和新增硬窗 skip 次数。

暂时不做的内容也要明确。第一，不把 `TimeWindowSummary` 接进 `PiecewiseLinearFunction`，因为它不是成本函数语义。第二，不在第一版接 pricing 的动态 `H_ij`，启发式摘要只使用数据层粗硬窗。第三，不用摘要直接计算 TWET 成本，它只负责判断明显不可行。第四，不把全 M 标准化和 Vidal 摘要混在一起；M 状态标准化是函数/label 层问题，硬窗摘要是拼接前时间可行性过滤。

## 12. 外包接入启发式的状态设计

2026-05-16 讨论外包如何接入当前启发式。当前 `Solution` 的核心状态是 `sequences`、`cost`、forward/backward 分段函数缓存以及 duration 缓存，这些数组全部按真实机器 `0..m-1` 建立。`EngineVND` 中的 I-opt、2-opt、2-opt*、path-insert 和 cross-exchange 也都默认一台机器对应一条有顺序的加工序列，并且需要 setup time、setup cost 和分段函数缓存来评价局部搜索。因此不建议把外包直接作为第 `m` 台“虚拟机器”塞进 `sequences`。这样虽然表面上可以复用机器间交换框架，但语义会变得很乱：外包没有序列相关 setup，没有 completion-time TWET 递推，也不需要 2-opt 或 cross-exchange 改变顺序；如果强行接入，很多缓存要额外为虚拟机器开特例，后续更容易出错。

更合适的方案是把外包作为 `Solution` 里的独立任务集合维护，例如 `outsourcedJobs`。它可以继续用 `ArrayList<Integer>` 表示，但这个 list 只用于保存任务集合、稳定输出和方便 remove/insert，不把顺序解释为加工顺序。外包成本单独维护一个 `outsourcingCostTotal`，总目标值改成“真实机器成本之和 + 外包成本”。这样保留 list 的好处是后续和 ALNS 的删除、插入流程衔接比较自然：被删除任务可以来自真实机器，也可以来自外包集合；修复阶段则可以在“插入某台真实机器某个位置”和“放入外包集合”之间比较增量成本。外包 list 的顺序不影响成本，因此插入时可以统一 append，输出时如果需要可再排序。

初始解建议先做保守兼容。当前 `setInitSolution()` 是把任务随机顺序塞进满足 `CmaxH` 的真实机器；如果某个任务没有候选机器，原逻辑可能出现空候选异常。接入外包后，可以先保持“优先放真实机器”的策略，只在没有可行真实机器、或者后续明确启用外包初始策略时，把任务放进 `outsourcedJobs`。如果 `data.outsourcingCost[j]` 是 `big_M` 状态，则说明当前数据不希望这个任务外包，此时不能静默外包，应当保留不可行诊断。更激进的版本可以在初始构造时比较“最佳真实机器插入增量”和“外包成本”，但这会改变当前启发式初始解分布，建议第二步再做。

ALNS 层需要改的是 removal 和 repair，而不是直接改 VND 的机器索引。`removeJobs()` 应支持同时从真实机器序列和 `outsourcedJobs` 删除任务，并同步更新对应真实机器成本或外包成本。repair 阶段的三类插入算子在枚举真实机器插入位置时，额外增加一个外包选项，其增量就是 `data.outsourcingCost[j]`。Greedy insertion 直接在真实机器最优插入和外包之间取最小；regret insertion 则把外包作为一个候选位置参与 regret 计算；randomized insertion 也把外包作为候选 option 放进排序后的候选池。这样外包和机器插入在 ALNS repair 中是同一层决策，不需要让外包参与真实机器的分段函数拼接。

VND/LS 层不建议第一版把外包纳入现有五类序列算子。现有算子解决的是“真实机器之间或机器内部的顺序重排”，外包没有顺序收益，所以让外包参与 2-opt 或 cross-exchange 没有意义。后续如果需要局部搜索层主动利用外包，应该单独加简单算子：一类是把真实机器上的某个任务移到外包，评价为“删除该任务后的机器成本变化 + 外包成本”；一类是把某个外包任务插回真实机器最佳位置，评价为“真实机器插入增量 - 外包成本”；必要时再做“真实机器任务与外包任务交换”，但这个可以先由前两类 move 组合近似。这样外包逻辑独立、可测，也不会污染现有机器间局部搜索缓存。

因此后续实现顺序建议为：第一步在 `Solution` 增加 `outsourcedJobs`、`outsourcingCostTotal`、外包增删接口、总成本校验和 copy 支持；第二步让 `setInitSolution()`、`removeJobs()` 和 ALNS repair 支持外包候选；第三步增加外包专用 LS move，并把它作为 `EngineVND` 中独立算子接入；第四步再考虑输出、验证和 BPC 初始列解释。验证时先用 `OUTSOURCING_COST=big_M` 跑一组，确认结果与无外包版本一致；再构造少量任务低外包成本的算例，确认任务能进入外包集合，且 `curCost == sum(cost[m]) + outsourcingCostTotal`。

2026-05-16 补充文献检索后的判断。调度领域里更常见的叫法是 scheduling with rejection 或 order acceptance and scheduling，本质上就是任务要么被内部机器接受加工，要么被拒绝/外包并支付 penalty。典型并行机 rejection 模型把任务集合拆成 accepted set 和 rejected set，目标是内部调度成本加 rejected penalty；这类模型支持“外包是单独集合”的建模思路，但通常不是局部搜索里把 rejected set 当成一台普通机器。另一个相关方向是 parallel machine scheduling with outsourcing allowed，例如 Neto、Godinho Filho、da Silva 的 ACO 文章，目标是外包成本和 delay/tardiness 成本之和；这类启发式通常在解构造阶段同时决定任务是否内部加工或外包，也说明外包集合是解状态的一部分，而不是普通机器序列的简单延伸。

和我们当前“外包任务如何与机器序列交互”最接近的是 workforce scheduling and routing with outsourced tasks。该文的局部搜索使用 Swap and Relocate，从两条 route 中移除最多两个任务组成的 path 并交换，其中一个 path 可以为空，从而退化成 relocate；同时它明确把 outsourced task 扩展进这个框架：如果外包列表非空，就随机选一个外包任务，把它插入当前解的最便宜位置，再交给局部搜索修复可能的不满足约束。这给当前项目的启示是：外包确实可以参与跨机器/跨 route 的移动，但应该是“外包列表参与 relocate/swap 的适配版本”，而不是把外包伪装成普通机器后让 2-opt*、path-insert、cross-exchange 原样跑。

因此对当前代码的落地判断略作修正：外包可以参与跨机器 move，但第一版只建议接入语义清楚的部分。`outsource -> real` 对应把外包任务插入某台真实机器的最便宜位置；`real -> outsource` 对应从真实机器删除任务或片段并支付外包成本；`real segment <-> outsourced jobs` 可以看成 cross-exchange 的外包适配版，但外包侧如果一次选多个任务，进入真实机器时需要人为规定顺序，不能声称已经枚举了外包集合的最优排列。`TwoOptStar` 这类依赖两条有序尾段交换的算子不适合直接接外包，因为外包侧没有尾段顺序语义。这个结论和前面的独立 `outsourcedJobs` 设计不冲突：状态上仍然是外包列表，算子上可以写外包适配的跨集合 move。

2026-05-16 又复核了 `PathInsertOperator` 里之前把 `if(len==1) break` 改为 `if(len==0) break` 的语义。代码中 `len` 不是片段任务数，而是片段长度减 1，实际取片段时用的是 `subList(from1, from1 + len + 1)`。因此 `len=0` 才是单任务片段，此时 reverse 和 normal 完全等价，可以跳过第二个方向；`len=1` 已经是两个任务片段，反向插入会把 `[i,j]` 变成 `[j,i]`，在 sequence-dependent setup time/cost 和任务时间惩罚下通常不是同一个候选。它也不能稳定地由当前其他跨机器 move 完全覆盖：cross-exchange 需要两边都选片段，不是空片段 relocation；2-opt* 交换的是两条序列尾段，不等同于把一个两任务片段反向插入另一台机器。所以这里更像是旧代码把 `len` 当成了“任务数”或为了跳过单任务 reverse 写错了条件，不应理解为有意规避邻域重叠。当前改为 `len==0` 后，只跳过单任务反向重复，保留两任务及以上片段的反向插入，是更符合算子语义的。

基于这个复核，外包 list 保留顺序仍然是有用的，但要清楚它的顺序语义弱于真实机器。外包 list 可以作为存储顺序和扰动顺序，方便从中抽取一个任务、连续若干任务，或者把外包侧任务按当前 list 顺序插入真实机器；这样实现机器-外包之间的 relocate/swap 会比较自然。只是这个顺序不应被解释为外包加工顺序，也不保证多任务插回真实机器时已经枚举了所有排列。第一版可以接受这个启发式约束：外包侧单任务插回真实机器是精确枚举位置；外包侧多任务片段插回真实机器时，先按外包 list 当前顺序和反向两种顺序评价，后续如果发现影响大，再对小片段加局部重排或按 due-date/成本函数排序生成候选顺序。

2026-05-16 已先实现第一版外包兼容，范围控制在“单任务 relocate + ALNS repair 外包候选”。`Solution` 新增 `outsourcedJobs` 和 `outsourcingCostTotal`，外包任务仍用 list 存储，但成本只按任务集合求和；`curCost` 的语义变为真实机器成本之和加外包成本。`removeJobs()` 现在会同时从真实机器序列和外包 list 删除任务，`setInitSolution()` 仍优先把任务放到真实机器，只有没有可行机器且该任务允许外包时才放进外包。这样不会把外包当成普通机器，也不会改动现有真实机器分段函数缓存。

ALNS 的三类 repair 插入算子现在都把外包作为额外候选：对 removed 任务，除枚举所有真实机器插入位置外，也可以选择直接外包，增量成本为 `outsourcingCost[j]`。如果外包成本处于 `big_M` 状态，则该外包候选不会进入 regret/randomized 的候选池；greedy 中即使比较到 `big_M`，也不会优于正常插入。VND 中新增 `OutsourcingRelocateOperator`，它复用现有单任务删除和插入评价：机器到外包的 move 计算“删除该任务后的机器成本变化 + 外包成本”，外包到机器的 move 计算“插入真实机器的成本变化 - 外包成本”。提交时仍采用类似其他跨机器算子的贪婪非冲突提交，一轮内同一台机器最多接受一个外包 relocate，避免缓存失效。

这一版还没有实现机器片段与外包片段的 cross-exchange，也没有让外包参与 `PathInsertOperator`/`CrossExchangeOperator` 原有枚举。原因是外包侧多任务顺序只是启发式顺序，直接接入原算子容易误解为普通机器序列。后续如果要增强，可以单独新增“机器片段 <-> 外包片段”算子，先按外包 list 当前顺序和反向顺序评价，再看是否需要对小片段做内部重排。

2026-05-16 进一步根据 `parallel_machine_scheduling_with_due_window.pdf` 修正外包成本语义。文档中 `b_j` 不是任务外包成本本身，而是 baseline outsourcing cost；给定外包集合 `O` 后，先计算 `B(O)=sum_{j in O} b_j`，真正外包成本是非递减凹函数 `G(B(O))`。因此上一版直接把外包成本写成 `sum b_j` 只对应 `G(q)=q` 的特殊情形，不是完整模型。代码层已把 `OUTSOURCING_COST` 保留为 baseline 向量，并在 `Data` 中新增 `outsourcingCostFunction` 表示 `G`。如果数据文件没有给定函数，则默认 `G(q)=q`，保证旧数据兼容；如果后续需要折扣函数，可以在数据文件追加 `OUTSOURCING_TARIFF`，格式为段数加若干行 `start end slope intercept`。

在启发式解状态里，`Solution` 现在维护两个外包量：`outsourcingBaselineTotal=B(O)` 和 `outsourcingCostTotal=G(B(O))`。任务加入外包时，增量不再是 `b_j`，而是 `G(B+b_j)-G(B)`；任务从外包移回内部时，增量是 `G(B-b_j)-G(B)`。ALNS repair 的外包候选和 VND 的 `OutsourcingRelocateOperator` 都改用这个函数差分。由于 `G` 是凹函数，多个外包增删操作的 delta 依赖当前 `B(O)`，不能像普通机器 move 一样把旧状态下算好的多个 delta 批量提交；所以外包 relocate 第一版改为每轮只提交当前最优的一个 move，保证成本增量语义准确。

2026-05-16 继续扩展外包在 ALNS 和 VND 中的交互。ALNS 的 `RandomRemoval`、`WorstRemoval`、`ShawRemoval` 都把 `outsourcedJobs` 纳入删除候选；真实机器候选仍保留“每台机器至少一个任务”的约束，外包候选不占用机器删除配额。VND 中删除旧的单任务 `OutsourcingRelocateOperator`，改为接入 `OutsourcingPathInsertOperator` 和 `OutsourcingCrossExchangeOperator`。其中 path-insert 支持“真实机器连续片段 -> 外包”和“外包连续片段 -> 真实机器某位置”，cross-exchange 支持真实机器片段与外包片段交换，外包片段插入真实机器时同时尝试原顺序和反序。这里没有把外包伪装成普通机器，因为外包 list 的顺序只用于生成候选，不表示加工顺序，也不参与 setup/time 分段函数递推；受影响真实机器的成本先用完整序列重算，后续如有性能瓶颈再考虑专门缓存。由于外包成本是 `G(B(O))`，多个外包 move 的增量依赖同一个当前 baseline `B(O)`，第一版每个外包算子只提交一个当前最优 move，不使用机器-机器 move 那种批量提交和 `noImprovedSeqPair` 缓存，避免把旧 baseline 下的 delta 用到新外包集合上。验证：带 CPLEX jar 的相关类 `javac` 编译通过。

复核时补充了一个安全细节：外包片段插回真实机器、真实机器片段移到外包、以及二者交换时，先显式检查新真实机器序列的累计 `setup time + processing time` 是否超过 `CmaxH`，只有通过后才调用 `Move.testSequence()` 计算分段函数成本。这样做不是改变邻域语义，而是避免把 `testSequence()` 中“不可行时返回 `curUpperBound`”的约定继续扩散到外包增量判断里。外包 move 的核心口径仍然是：真实机器侧必须形成一条可行序列，外包侧只改变集合和 `G(B(O))`；若机器侧不可行，该候选直接跳过。

2026-05-16 继续修正外包 move 的评价方式。上一版为了先跑通语义，在外包片段插回真实机器或交换到真实机器时，直接构造 `newSeq` 并调用 `Move.testSequence(newSeq)` 重算整条机器成本。这种做法正确性上容易对拍，但放在候选枚举内层复杂度太高，等于每个候选都重新做一遍 forward/backward 递推，不符合原启发式围绕子序列预处理和 O(1) 拼接评价的设计。现在改为给外包 list 建一个临时 `OutsourcedSegmentCache`，按外包 list 当前顺序和反序预处理连续片段的 forward 函数、backward 函数和片段 duration；真实机器侧仍复用已有 `fFunctions/bFunctions/cumDuration`。这样“机器片段 -> 外包”用两段拼接评价删除后的机器成本，“外包片段 -> 机器”和“机器片段 <-> 外包片段”用三段拼接评价新机器成本，外包侧成本仍用 `G(B(O))` 的 baseline 差分 O(1) 计算。

这里没有把外包直接并入 `m+1` 台机器。原因是外包 list 只有候选生成顺序，不表示加工顺序；外包成本由集合 baseline 的凹函数 `G(B)` 决定，不由序列上的 TWET 和 setup 递推决定。如果把外包塞进 `sequences` 并把数组扩成 `m+1`，普通 I-opt、2-opt、2-opt*、path-insert 和 cross-exchange 都会把它当真实机器运行，反而需要到处写特例。当前采取的折中是：外包状态单独维护；在某段外包任务要进入真实机器时，把这段临时当作有序片段预处理，从而复用现有 `merge2Segments/merge3Segments`。这保留了原来高效数据结构，也避免污染真实机器数组语义。

关于是否还要单独做外包版 2-opt*，当前结论是先不做。纯机器之间保留 `TwoOptStarOperator` 的原因是它虽然可被 cross-exchange 视为尾段交换特例，但搜索空间小、评价只需两段拼接，作为轻量邻域有价值。外包侧技术上也能做同样的尾段交换：真实机器 `A+B` 与外包 list `C+D` 交换尾段，得到机器 `A+D`、外包 `C+B`。但这个 move 已经被当前 `OutsourcingCrossExchangeOperator` 覆盖，即机器片段取尾段、外包片段取尾段时就是外包版 2-opt*。由于外包 list 的尾段只是存储顺序上的尾段，不像真实机器尾段那样有明确加工路径语义，现阶段再单独加一个外包 2-opt* 主要是性能型特例，而不是新增邻域能力。因此先保留当前外包 path-insert 和外包 cross-exchange，等实际运行后如果发现外包 cross-exchange 候选过多或耗时明显，再考虑把外包 2-opt* 作为轻量优先邻域单独加入。

2026-05-16 对外包快速评价做了专项对拍测试。新增 `HEU.OutsourcingMoveConsistencyTest`，测试构造 60 任务、3 机器算例，手工注入随机非零 setup time 和 setup cost，并设置线性外包成本函数 `G(B)=0.8B+3`。测试先从真实机器中抽取若干任务放入外包集合，再枚举外包相关三类候选：机器片段移到外包、外包片段插入机器、机器片段与外包片段交换。每个候选都同时计算快速评价结果和慢速 `Move.testSequence(newSeq)` 重算结果，并比较 Cmax 判断是否一致、机器侧成本是否一致；外包侧则单独检查 `G(B)` baseline 差分。测试结果为 `OutsourcingMoveConsistencyTest passed, checked=10971`。这说明当前 `OutsourcingMoveEvaluator` 的 `merge2Segments/merge3Segments` 快速评价与慢速重算在该随机非零 setup 场景下保持一致。

2026-05-16 进一步确认真实机器应允许被删空。原因是外包模型下，某些极端数据里所有任务外包成本可能很低甚至为 0，此时最优解完全可能让真实机器空闲；即使不全外包，也没有数学理由要求每台真实机器至少保留一个任务。代码层因此移除“真实机器非空”的旧搜索假设：ALNS 的 `RandomRemoval`、`WorstRemoval`、`ShawRemoval` 不再强制每台真实机器至少剩一个任务，而是只避免对已经没有剩余任务的机器继续删除；外包版 path-insert 也允许把某台真实机器上的完整片段移入外包。这样 ALNS/VND 可以自然产生空机器，后续插入算子仍可通过 `pos=-1` 把任务插回空机器开头。

同时对普通跨机器算子的缓存做了边界处理。`TwoOptStarOperator` 只有两台机器都至少有两个任务时才存在合法切尾候选；`CrossExchangeOperator` 只有两台机器都非空时才存在可交换片段。因此这类“算子本身无法作用”的机器对直接跳过，不写入 `noImprovedSeqPair`，也避免多台空机器时 `Set.of(emptyList, emptyList)` 的重复元素异常。这里没有把 `Set.of` 全部替换成新 key，因为在算子可作用的前提下，两个不同机器上的任务集合不应完全相同；真正需要处理的是空机器导致的无意义 pair。

验证上，`OutsourcingMoveConsistencyTest` 增加了一个专门用例：构造只有一个内部任务的机器，并令外包成本函数 `G(B)=0`，要求 `OutsourcingPathInsertOperator` 能把这个唯一任务移入外包，使该机器变为空。更新后编译通过，运行结果为 `OutsourcingMoveConsistencyTest passed, checked=10972`。这说明当前外包快速评价、空机器成本更新和外包集合更新在该边界场景下保持一致。后续仍需关注的是：ALNS repair 若在极端情况下所有插入位置都是 `big_M/curUpperBound`，目前仍会选择一个位置作为扰动兜底；这属于之前已记录的 repair 鲁棒性问题，本次没有改变。

2026-05-16 继续给外包算子补 `noImproved` 缓存。机器-机器算子里，单机算子用单条序列缓存，`TwoOptStar/CrossExchange` 用无向 pair，`PathInsert` 用有向 pair；外包算子不能直接照搬这套 key，因为外包成本虽然只依赖外包集合的 baseline `B(O)`，但当前外包候选是按 `outsourcedJobs` 的连续片段生成的。也就是说，同一个外包集合如果 list 顺序不同，可枚举的外包片段也不同，不能只用 set 作为缓存状态。因此代码新增 `OutsourcingStateKey`，保存“机器序列 copy + 外包 list copy”，用 list 顺序保证缓存语义和候选生成完全一致。

外包 path-insert 的两个方向拆开缓存：`noImprovedMachineToOutsource` 记录“当前机器序列和当前外包 list 下，没有任何机器片段移入外包能改进”；`noImprovedOutsourceToMachine` 记录“当前外包 list 和某台机器序列下，没有任何外包片段插回该机器能改进”。这两个方向不能合并，因为 `machine -> outsource` 和 `outsource -> machine` 的候选集合、成本差分和可行性判断都不同。外包 cross-exchange 则使用单独的 `noImprovedSeqAndOutsource`，记录当前机器序列和当前外包 list 下没有可改进的机器片段/外包片段交换。所有外包缓存都会在 `Move.Recordreset()` 中清空，和原有算子缓存保持同一生命周期。

这个缓存的设计边界也要明确。第一，它保存的是外包 list，不是集合，因此如果外包集合没变但顺序变了，会重新搜索；这是必要的，因为连续片段候选变了。第二，它只缓存“没有改进”的状态；一旦某个状态存在改进候选，不写入缓存，由 commit 选择当前最优 move 后进入下一轮搜索。第三，因为 key 中包含完整外包 list，若外包 baseline `B(O)` 改变，key 也一定改变，`G(B)` 的凹函数增量不会被旧缓存误用。这个实现比直接把外包当成普通机器 pair 缓存更保守，但语义更清楚。

同日再次复核“真实机器可以为空”相关修改。当前成本层是安全的：`calCost(m)` 对空机器返回 0，`updateInformationM(m)` 在空机器下会重建空缓存并保持成本为 0，`insertJob(m,-1,job)` 可以把任务插入空机器开头。普通 VND 中，`IOpt/TwoOpt` 对空机器自然没有候选，`TwoOptStar` 要求两台机器任务数都至少为 2，`CrossExchange` 要求两台机器都非空；这些不可作用的 pair 不写入 no-improved 缓存。`PathInsert` 本来就是有向搬移，源机器空时无候选，目标机器空时可以通过 `pos=-1` 插入，因此无需额外限制。外包 VND 中，机器片段移入外包允许删除整台机器，外包片段插回机器和外包/机器片段交换仍会先做 Cmax 检查再评价成本。ALNS removal 现在也不再强制每台机器至少保留一个任务。基于当前检查和 `OutsourcingMoveConsistencyTest` 的单任务机器全外包用例，机器可为空这条主路径目前是成立的。剩余未改的是之前记录过的 ALNS repair 全部候选为 `big_M/curUpperBound` 时的兜底策略，这不是空机器本身的问题。

2026-05-16 又补了一处初始解构造的硬时间窗过滤。原来的 `setInitSolution()` 只用累计 `setup time + processing time <= CmaxH` 判断任务能否 append 到某台机器末尾；在启用预处理粗硬窗后，这可能生成一条总时长不超 `CmaxH`、但某些任务完工时间落在粗硬窗外的初始序列。现在初始构造阶段维护每台机器的当前最早可行完工时间，对候选机器 `m` 和任务 `j` 计算 `earliestCompletion = currentCompletion[m] + s[lastJob][j] + p[j]`，再取 `adjustedCompletion = max(earliestCompletion, hardWindowStart[j])`。这里的 `max` 表示如果最早完工早于硬窗左端，机器可以在加工该任务前等待，使任务完工时间推迟到粗硬窗左端。候选机器必须满足 `adjustedCompletion <= hardWindowEnd[j]` 且 `adjustedCompletion <= CmaxH`。如果所有真实机器都不满足，则沿用已有逻辑：能外包则放入外包，否则抛出初始化不可行异常。

这个处理只使用数据层的 job 级粗硬窗，不在初始构造里动态计算 pricing 的 `H_ij`。原因是初始构造不含 dual，也不对应某次具体 RMP 定价；`H_ij` 仍应留到 pricing 扩展时按前驱和 dual 动态计算。若后续 setup cost 满足三角不等式，`B_ij` 大概率为 0，粗硬窗会更接近任务自身外包成本导出的窗口。需要注意的是，随机 append 顺序下某个任务放不进机器，不代表实例本身无可行内部调度；它只说明当前随机初始构造失败。因此第一版仍保持简单策略：可外包则外包，不可外包则显式报错。若后续发现初始化失败频繁，再考虑多次 shuffle 或按 `hardWindowEnd/due window` 排序构造初始解。验证上，`OutsourcingMoveConsistencyTest` 增加了一个硬窗左端较晚的初始化用例，按同一 earliest-time 递推检查初始序列，运行结果为 `OutsourcingMoveConsistencyTest passed, checked=10973`。

## 11. 2026-05-17 未完成事项梳理

本次回看前面关于外包、硬时间窗、分段函数和 BPC pricing 的讨论后，当前已经完成的是启发式主流程中的 setup time/setup cost、粗硬时间窗、外包集合、外包 move、空机器兼容和外包 move 快速评价测试。剩下的内容可以分成三类。第一类是启发式鲁棒性问题，主要是 ALNS repair 在所有插入位置都被评为 `big_M/curUpperBound` 时仍会选一个位置插入；这在当前宽 `CmaxH` 和可外包数据下不一定触发，但严格来说还缺少“无真实可行插入”的显式兜底。第二类是性能增强问题，主要是 Vidal 式硬时间窗摘要剪枝：为 normal/reverse 子序列维护 earliest/latest/feasible 信息，在调用 `merge2Segments/merge3Segments` 前先跳过明显违反硬窗的候选。第三类是精确算法问题，包括 pricing 中动态计算 `H_ij`、接入真正的 label 扩展和 dominance、处理 partial dominance 下的 M 状态标准化、以及把 RMP/pricing/cut 的高密度输出补齐。

因此，若继续完善启发式，优先级最高的是 ALNS repair 的不可行候选兜底，其次才是 Vidal 式剪枝和批量性能测试。若转向 BPC，则应先实现 pricing label 的数据结构和动态 `H_ij`，并明确 `curUpperBound` 在 pricing 中重置为 `big_M` 的调用位置；否则前面分段函数层面的约束已经准备好，但还没有真正被定价算法使用。外包版 2-opt*、外包 list 片段内部全排列、以及更细的 profiling 目前都是可选增强，不是当前主正确性缺口。

## 12. 2026-05-17 M 状态与 partial dominance 的处理边界

关于 M 状态，本次进一步确认它和 Vidal 式硬窗剪枝不是同一类问题。Vidal 式摘要只是在进入 `merge2Segments/merge3Segments` 前判断拼接是否明显违反硬时间窗，属于性能过滤；M 状态则是分段函数和 label dominance 的语义问题，尤其会影响后续是否能安全启用 partial dominance。

如果第一版 pricing 只使用完整函数占优，M 状态的风险相对较小。此时 label 函数中的 M 主要来自 job 函数的硬窗外区间，经过扩展后的 prefix/suffix 取小，右侧或左侧很多 M 区间会被可行完成时间的真实成本闭包掉；剩下的 M 区间即使参与完整占优，也基本只表示“该时间段不可行或极差”。只要 `big_M` 远大于真实成本和 dual 修正幅度，完整占优不会因为 `M+a` 或 `M-a` 的细小数值差异产生主要正确性风险。更严格地说，第一版可以先要求完整占优只在满足统一定义域契约、没有内部 M 碎片的 label 之间使用。

真正需要警惕的是 partial dominance。partial dominance 会把一个 label 函数中被支配的局部区间写成 M 段，这会让同一个 label 内部出现不可行碎片。这个 label 后续继续扩展时，M 段会和 job 函数、setup cost、dual 项一起参与运算，可能变成 `M+a`、`M-a` 或多个 M 叠加后的数。语义上这些段仍然应视为不可行状态，但数值上已经不一定等于 `big_M`。如果后续占优、merge 或 normalize 只按普通 double 值比较，就可能把两个不可行段错误地区分优劣，或者因为 dual 把 `M` 拉成 `M-a` 而漏掉 M 状态识别。

因此，`big_M/2` 更适合作为 M 状态识别阈值，而不是直接把所有 minimize 的初始极大值改成 `M/2`。如果把 `runningMin` 直接设成 `M/2`，它会成为真实参与函数闭包的人工成本，后续很难区分“这里确实有一个很差但可行的函数值”和“这里只是不可行状态的占位”。更稳的处理是保留 `Utility.big_M` 作为不可行占位值，同时在 pricing label 层增加一个显式的 M 状态标准化步骤：扩展、加 dual、prefix/suffix 取小之后，若某段整个区间的函数值都大于 `big_M/2`，就把它统一写回水平 `big_M` 段；若整条函数都处于这种状态，则该 label 标记为不可继续扩展或直接丢弃。

基于这个判断，当前推荐的推进顺序是：第一版 BPC pricing 先不启用 partial dominance，优先使用完整 dominance 和清晰的 label 资源比较，把主流程跑通；等需要 partial dominance 压缩 label 数量时，再补 M 状态标准化和对应测试。这个标准化不建议塞进底层 `add()` 或所有 `minimize` 函数里，而应由 pricing label 层显式调用，避免影响当前启发式和普通函数操作的语义。

进一步讨论后，可以把 `M/2` 的作用再说得更准确一点。把 `minimizePrefixInPlace()` 或 `minimizeSuffixInPlace()` 里的初始 `runningMin` 直接改成 `M/2` 并不是必要条件。若某段是 `M+a`，另一段是 `M-a`，它们在语义上都应视为不可行状态；只要二者仍然大于 `M/2`，在进入 dominance 前做一次 M 状态标准化，就可以把它们统一压回水平 `big_M` 段。这样 L1 和 L2 在不可行区间上不会因为 `a` 的微小差异影响支配判断。相反，如果不做标准化而直接比较普通 double，partial dominance 可能只是在这些不可行区间上少删一些段，通常表现为剪枝变弱，而不是直接破坏正确性；真正危险的是后续把这种 `M-a` 当成真实可行成本继续传播。

因此更稳的规则是：`M/2` 只作为“足够大的不可行状态阈值”，不直接替代函数闭包里的极大初始值。pricing label 扩展完成并做完 prefix/suffix 取小后，如果某个 segment 在整个区间上的函数值都不低于 `M/2`，就把它标准化成斜率 0、截距 `big_M` 的不可行段；如果整条函数都被标准化为 M，则该 label 不再继续扩展。这里最好按 segment 的区间最小值判断，而不是只看 intercept，因为 M 段和 job 函数相加后可能带有非零 slope。这样既能解决 partial dominance 中 `M+a` 对 `M-a` 的比较问题，又不会把 `M/2` 变成一个真实参与优化的人工成本值。

本轮讨论中，用户对 M 状态的核心判断需要原样保留，后续实现 partial dominance 时优先按这个判断回看，而不是重新从泛化情况出发。原话记录如下：

> M/2到是问题不大，因为M和M/2都是一个足够大的数，远大于最优解或者过程中的数字，所以感觉这么做应该也OK？主要是为了保证partial domiannce的时候如果两个要作占优，但L1其他地方都能干掉L2，只有某一段M+a没办法干掉M-a. 欸不对，那感觉不用改成M/2作为初始的也可以？改了的话相当于此时L1可以直接干掉L2，但其实部分占优下意义不大，因为占优以后其实还是设置为M。而不设置M/2的时候，即使出现L1和L2中M的比较可能不能互相支配，但由于normalize里边用的M/2，如果L1其他地方都能干掉L2，此时其实L2里边都比M/2更大，也是可以被标准化干掉的？

> 标准化时最好不要只看 intercept，因为 M 段和 job 函数相加后可能有 slope。应该判断这个 segment 在整个区间上的最小值是否都 >= M/2，如果是，才压回水平 big_M。这个无所谓了，截距都M了，斜率完全没影响了。

> 以及，前边你的分析是啥，我的意思是感觉上好像也不需要特殊处理了，现在的基本够了，因为每次部分占优以后都会做normalize，而这个东西其实是从M/2开始的，那前边那种即使M-10没被占优，这里肯定也是可以被干掉的，即这个label的函数为空了

对这段讨论的最终整理是：当前代码中的 `M/2` 已经用于 `normalizeForward/normalizeBackward` 的 M 状态识别，因此如果 partial dominance 后函数所有段都在 M 附近，并且这些段的代表值都能被 `isBigMValue()` 识别，那么 normalize 会把这类函数清空或压成不可继续扩展的状态。这样看，第一版 partial dominance 不一定需要额外引入“把 `runningMin` 改成 `M/2`”这类操作；更合理的做法是先沿用当前 normalize 的 M 状态识别机制，等真实 pricing 中观察到 `M-a` 段大量残留并影响占优效率时，再补更显式的 M 标准化后处理。这里也接受一个工程前提：当前 `big_M` 足够大，截距在 M 量级时，斜率不会把整段函数拉回正常成本量级，因此第一版可以先按截距/代表值识别 M 状态，不必一开始就做严格的整段最小值扫描。
## 13. 2026-05-17 Vidal 式硬窗预判已接入 move

本次把前面讨论的 Vidal 式硬时间窗剪枝先接入为一个安全预判层。新增 `TimeWindowSummary`，对每个连续子序列记录首任务、尾任务、段内最短持续时间、该段内部硬窗是否自洽，以及“第一个任务最晚可在什么时候完成”。这个摘要不计算 TWET 成本，也不处理 setup cost；它只判断当前片段重组后是否已经明显违反 `hardWindowStart/hardWindowEnd`。如果摘要判断不可行，就直接跳过该 move；如果摘要判断可能可行，仍然继续调用现有的 `merge2Segments/merge3Segments`，用分段线性函数计算真实成本。因此这次修改是性能剪枝，不改变目标函数语义。

摘要缓存放在 `Solution` 里，和现有 `fFunctions_hg/fFunctions_gh` 对齐。每台机器维护 normal 和 reverse 两套二维摘要，`updateInformationM(m)` 在重建函数和累计 duration 后同步重建该机器的硬窗摘要。这样 `IOpt`、`TwoOpt`、`TwoOptStar`、`PathInsert`、`CrossExchange` 在评估候选时，都可以用若干个子序列摘要常数时间拼出候选新序列的硬窗可行性。外包相关 move 也接入同一套逻辑：外包片段在 `SegmentProfile` 里带上自己的 `TimeWindowSummary`，机器片段删除、外包片段插回机器、机器片段与外包片段交换时，先做 Cmax 检查，再做硬窗摘要检查，最后才进入分段函数成本计算。

这里的预判有意做成保守版本。它使用前一段的最早完工下界去检查后一段第一个任务的最晚可行完工时间，因此如果返回不可行，可以认为该候选确实不可能满足硬窗；但如果返回可行，只代表没有被这个轻量摘要证明不可行，仍可能在后续精确函数计算时变成 `big_M/curUpperBound`。这样做是为了避免误剪，因为当前启发式最重要的是不能把可能可行、可能改进的 move 提前删掉。

验证上，`OutsourcingMoveConsistencyTest` 已经把慢速可行性检查从单纯 Cmax 改成了 earliest-time 硬窗递推，并补了一个人为收紧 `job2` 右端硬窗的用例，检查摘要能拒绝 `1 -> 2` 这种必然不可行的拼接。当前编译通过，测试结果为 `OutsourcingMoveConsistencyTest passed, checked=10974`。剩余风险是：这套摘要目前只用于粗硬窗剪枝，不处理 pricing 里随 dual 动态变化的 `H_ij`；以后做 BPC pricing 时，仍需要在 label 扩展层按当前 dual 单独计算动态硬窗。

2026-05-17 继续优化了硬窗摘要的预处理效率。前一版 `updateTimeWindowSummariesForMachine(m)` 对每个 `[from,to]` 都复制一次 `subList`，反向摘要还要再反转一次，然后调用 `TimeWindowSummary.of(...)` 从头扫描，因此单台机器的摘要构建带有明显的重复扫描和对象创建，最坏接近 `O(n^3)`。现在改成二维递推：正向摘要固定右端 `to`，从 `to` 向左递推 `latestFirstCompletion` 和 `durationAfterFirst`；反向摘要固定左端 `from`，从 `from` 向右递推反向序列的同两个量。这样 normal/reverse 两套摘要都可以在 `O(n^2)` 时间内构建，且不再为每个区间创建临时 list。

这次同时复核了 `empty` 和 `feasible` 的语义。`empty` 仍然保留，用来统一处理 move 中不存在的 prefix/suffix；否则每个算子都要单独判断空片段。`feasible` 在“当前真实机器序列本身一定可行”的理想前提下确实大多冗余，但它对反转片段、外包片段、以及后续硬窗进一步收紧后的片段仍有保护作用，保留成本只是一个布尔值和一次递推判断，因此先不删。验证仍沿用当前外包 move 一致性测试：相关类 `javac` 编译通过，`OutsourcingMoveConsistencyTest passed, checked=10974`。

关于 `feasible=false` 片段是否还需要构造分段函数，本次先记录为后续性能优化，不立即改代码。逻辑上，如果某个 reverse 片段或外包片段自身硬窗不可行，那么它在 move 评估时会被 `TimeWindowSummary` 直接拒绝，对应的 forward/backward 分段函数不会真正用于成本计算。因此可以先算摘要，若 `feasible=false` 就跳过该片段函数构造，或者对真实机器缓存写成全 `big_M` 函数以避免旧路径误用。但这属于预处理阶段的常数优化，不改变正确性；当前函数构建本身仍是预处理的一部分，预计不是眼下主要瓶颈，所以暂不调整，只在 `TimeWindowSummary.feasible` 附近补注释，后续 profiling 后再决定是否做。

## 14. 2026-05-17 数学模型对外包的覆盖情况

本次只检查 `ArcFlowModel`、`ATIParallel` 和 `TimeIndexModel`。结论是，当前两个主要精确模型 `ArcFlowModel` 和 `ATIParallel` 都还没有把外包建进去。它们的约束仍然是“每个任务必须被内部机器加工一次”：`ArcFlowModel` 通过每个 job 的 incoming flow 等于 1、outgoing flow 等于 1 来保证任务都在内部路径中；`ATIParallel` 通过每个 job 的进入弧流量等于 1 来保证任务都被选入时间扩展网络。因此现在这两个模型已经考虑了 setup time 和 setup cost，但没有外包决策变量，也没有 `G(B(O))` 这类外包集合成本。

如果后续要让数学模型也支持外包，`ArcFlowModel` 相对更直接：可以给每个 job 增加一个外包变量 `y_j`，把任务覆盖约束改成“内部进入一次 + 外包一次 = 1”，目标函数再加入外包成本。但由于当前外包真实成本是 `G(sum b_j y_j)`，如果 `G` 是分段线性凹函数，还需要额外变量或分段线性建模来表达总外包 baseline 与成本之间的关系。`ATIParallel` 也可以做类似覆盖约束修改，但因为它是时间扩展弧流模型，需要同步处理 job 覆盖、外包变量和 `G(B)`，比单纯弧流模型更麻烦一些。

`TimeIndexModel` 当前不继续扩展。它本身是不显式区分机器和前后继的 time-indexed 基准模型，只用时间容量约束限制同时加工数量；这种结构不适合直接表达 sequence-dependent setup time，更不用说 setup cost 和外包集合成本。因此本次只在代码注释中标明它不覆盖 setup/outsource 相关实验，后续相关精确模型优先沿 `ArcFlowModel` 或新的 BPC 框架推进。

2026-05-17 已把外包变量接入 `ArcFlowModel` 和 `ATIParallel`。建模方式模仿文档中 SP2 的思想，但不是 set-partitioning 主问题，而是把同样的外包覆盖逻辑嵌入现有精确模型：每个任务增加二元变量 `y_j`，`y_j=1` 表示该任务外包，不进入内部机器序列或时间扩展网络。任务覆盖约束改为“内部加工覆盖 + y_j = 1”。在 `ArcFlowModel` 中，incoming 约束改成 `sum_i x_ij + y_j = 1`，outgoing 约束改成 `sum_k x_jk + y_j = 1`，因此外包任务既没有内部前驱，也没有内部后继；在 `ATIParallel` 中，每个任务的进入弧流量约束改成 `sum incoming arcs + y_j = 1`，外包任务不再需要时间扩展弧覆盖。

目标函数中加入 `G(sum_j b_j y_j)`。具体实现是从 `Data.outsourcingCostFunction` 读取分段函数，每段建立一个二元选择变量 `u_l` 和一个连续 baseline 变量 `q_l`，并加入 `sum_j b_j y_j = sum_l q_l`、`L_l u_l <= q_l <= U_l u_l`、`sum_l u_l = 1`。目标函数加入 `sum_l (a_l q_l + c_l u_l)`。如果某个任务的 `outsourcingCost[j]` 仍是 `big_M`，说明当前数据没有给它有效外包 baseline，模型中直接固定 `y_j=0`，避免用无意义的大数进入 `G(B)`。

同时给两个模型都补了 `extractOutsourcedJobs()`，方便后续输出和验证。当前验证包括两部分：第一，`ArcFlowModel` 和 `ATIParallel` 重新编译通过；第二，临时构造 5 个任务、外包 baseline 全为 0 且 `G(B)=0` 的小算例，两个模型都能求解到目标值 0，并且 `extractOutsourcedJobs()` 显示所有任务都被外包。这个测试说明外包变量已经真正进入覆盖约束和目标函数，而不是只加了一个无效成本项。

2026-05-17 再次复核外包链路后，当前判断是主实现基本正确。数据层由 `OUTSOURCING_COST` 读取每个任务的 baseline `b_j`，由 `OUTSOURCING_TARIFF` 读取总外包成本函数 `G(B)`；如果没有给 tariff，则默认 `G(B)=B`。启发式层单独维护 `outsourcedJobs`、`outsourcingBaselineTotal` 和 `outsourcingCostTotal`，真实机器侧仍用分段函数和 setup time/setup cost 评价，外包侧只通过 `G(B)` 的差分进入增量。模型层的 `ArcFlowModel` 和 `ATIParallel` 都已经把 `y_j` 接入覆盖约束，并用 `u_l/q_l` 表达 `G(sum b_j y_j)`。

这次复核也保留两个数据假设。第一，当前可选块是在 `loadSetupMatrices()` 里读取的，因此带外包的正式数据应按当前格式放在 setup 块之后，并以 `setup=true` 的方式构造 `Data`；如果后续希望无 setup time 的实例也能直接读取外包块，需要把 optional block 读取逻辑从 setup 分支里拆出来。第二，`OUTSOURCING_TARIFF` 的分段应覆盖所有可能的 `B=sum b_j y_j`，尤其要覆盖 `B=0`；并且如果不想“没有外包任务时也产生固定外包费用”，应保证包含 0 的分段在 `B=0` 时给出 `G(0)=0`。这两个不是当前代码的求解错误，而是数据格式和建模口径约束，后续生成外包数据时要按这个要求处理。

本次验证重新编译了 `Data/ArcFlowModel/ATIParallel/TimeIndexModel/Solution/EngineALNS/EngineVND/Move/OutsourcingMoveConsistencyTest`，编译通过；`OutsourcingMoveConsistencyTest` 通过，结果为 `checked=10974`；临时 5 任务零外包成本 smoke test 中，`ArcFlowModel` 和 `ATIParallel` 都得到目标值 0，并识别所有任务为外包。剩余未覆盖的是更一般的 tariff 数据对拍，例如非零、分段、凹折扣的 `G(B)`，后续如果正式使用这类数据，应再补一个小规模枚举或模型对拍测试。
## 15. 2026-05-17 小规模随机对拍与问题修正

本次按“带外包、带 setup time、带 setup cost”的设定随机生成小算例，用 `ArcFlowModel` 作为精确模型基准，再用当前 `EngineALNS + EngineVND` 启发式多起点求解做对拍。测试规模先控制在 7-8 个任务、2 台同质机器，外包成本采用单段线性 `G(B)=rate*B`，setup time 和 setup cost 都随机生成。这样做的目的不是证明启发式总能最优，而是用小规模精确模型尽快暴露外包接入后最容易出错的状态同步、不可行候选过滤和成本增量维护问题。

对拍过程中先后暴露了三个实现问题。第一，外包 path/cross move 修改真实机器序列以后，原来只重算了机器成本，没有同步重建该机器的二维分段函数、duration 缓存和硬时间窗摘要。后续普通 VND 算子继续使用旧缓存时，会出现机器序列长度和缓存长度不一致。现在外包 move 的机器侧提交统一改为 `updateInformationM(m)`，保证序列、成本和所有预处理缓存同步。第二，若分段函数 merge 因不可行或上界剪枝返回 `curUpperBound`，不能再把它当作真实新机器成本去减原成本；否则当原机器成本较大时，不可行候选可能被误判成负 delta 并被提交。现在 `Move.isInvalidMoveCost()` 统一识别 `big_M/curUpperBound` 伪成本，I-opt、2-opt、2-opt*、path-insert、cross-exchange 以及 ALNS 插入评估都用该逻辑过滤不可行候选。第三，`Solution.curCost` 原来在 `updateInformationM()` 中会被截断到 `curUpperBound`，这和外包成本 `G(B)` 的差分更新冲突；一旦用被截断的 `curCost` 去减真实旧外包成本，就可能产生巨大负数。现在 `curCost` 始终保存真实目标值，上界只用于分段函数剪枝和 move 过滤，不再污染解对象状态。

同时补了两个边界处理。`getNormalDuration()` 和 `getReverseDuration()` 对空片段返回 0，避免各个 move 对不存在的前缀、后缀重复写特殊判断；如果传入非空但越界的区间，则直接抛出异常，这代表调用方或缓存同步已经破坏，不应静默修正。ALNS 的 regret/randomized repair 也不再把 `big_M` 机器插入位置放入候选池，避免随机修复阶段抽到硬时间窗不可行的插入位置。

验证结果为：随机 16 组小算例，每组 20 个启发式起点，`ArcFlowModel` 与启发式最优结果全部一致，最大差距约 `3e-9`，属于浮点误差；`OutsourcingMoveConsistencyTest` 通过，结果为 `checked=10974`。另外，前 4 组中同步跑了 `ATIParallel`，其中第 3 组出现 `ArcFlowModel=21.0`、`ATIParallel=20.5` 的差异。这个差异说明两个精确模型之间仍有一个独立建模或离散化问题需要后续单独排查；本次启发式对拍先以 `ArcFlowModel` 为精确基准。

本次还尝试了全量编译 `src`，失败点在当前尚未完成的 BPC/GC 参考框架代码，例如旧 VRP 字段和当前 TWET `Label/Data` 接口不匹配。这不是本次外包和启发式修改引入的问题。因此当前可确认的是：启发式、外包 move、`ArcFlowModel/ATIParallel` 相关类的定向编译通过；完整仓库编译仍受未完成 BPC 代码影响。

## 16. 2026-05-17 ATIParallel 与 ArcFlowModel 目标差异排查

前一轮小规模随机对拍里，第 3 个算例出现 `ArcFlowModel=21.0`、`ATIParallel=20.5` 的差异。排查后确认，这不是外包目标函数或 setup cost 的差异，而是 `ATIParallel` 中时间扩展网络的弧变量原来建成了连续变量：`x[idx] = cpx.numVar(0, d.m, IloNumVarType.Float, ...)`。这样模型允许同一个任务的 1 单位覆盖流量被拆分到多条入弧上，得到的是 LP 松弛下界，而不是整数调度解。这个问题在无外包或简单算例里不一定暴露，但加入外包和分段 `G(B)` 后，更容易出现半整数或分数流让目标值比真实整数解更低。

现在把 `ATIParallel` 的弧变量改成 `[0,m]` 的整数变量，而不是 `boolVar`。原因是时间扩展网络里不只有任务转移弧，还有等待弧，等待弧可能承载多台机器的流量；如果全部改成二元变量会错误限制等待流量。使用整数变量后，第 3 个随机算例重新求解得到 `ATIParallel=21.0`，与 `ArcFlowModel=21.0` 一致；前 4 个带 ATI 对照的小算例也都与 ArcFlow 一致。这个修改会让 `ATIParallel` 从 LP 松弛变成真正的整数模型，求解可能变慢，但它作为精确模型或对拍基准时必须这样建模。

## 17. 2026-05-17 ATIParallel 来源与线性松弛语义

重新核对 Bulhoes et al. (2020) 后，当前 `ATIParallel` 基本对应论文第 2 节的 arc-time-indexed formulation `F1`：节点为 `(j,t)`，弧集合分为任务转移弧 `A1`、从虚拟源点发出的首任务弧 `A2`、返回虚拟终点的收尾弧 `A3` 和等待弧 `A4`。当前代码里的 `generateArcs()` 也是按这四类弧生成，所以可以把它理解为对论文 `F1` 的同质机简化实现，再额外加入了 setup cost 和外包变量。

论文里的 `F1` 不是一个可以直接放松成连续变量且仍保持整数等价的模型。原文在 `F1` 后面明确给出变量条件：`x >= 0` 和 `x integer`。后文提到的 linear relaxation，是先把 `F1` 用 Dantzig-Wolfe 改写成 master formulation，再“only in terms of lambda and relaxing the integrality constraints”得到 Dantzig-Wolfe master linear program。也就是说，论文允许求的是列生成主问题的 LP 松弛，用它作为下界；并没有说原始 arc-time-indexed 网络流模型把所有弧变量直接改成连续变量后还等价于整数调度。

因此，原代码里 `x[idx] = cpx.numVar(0, d.m, IloNumVarType.Float, ...)` 更像是实现时把 ATI 写成了 LP relaxation，而不是论文要求的完整整数模型。这个问题并不是加入外包以后才从“等价”变成“不等价”；没有外包时连续弧流也可能产生分数调度，只是某些小例子上未必暴露。加入外包和 `G(B)` 后，任务覆盖可以在内部弧流和外包变量之间形成更复杂的组合，分数流更容易得到低于真实整数调度的目标值，所以第 3 个随机算例出现 `20.5 < 21.0`。修正为 `[0,m]` 整数弧变量后，ATI 才能作为精确模型与 ArcFlow 对拍。
## 18. 2026-05-17 外包 move、curUpperBound 与可行 repair 的语义澄清

本轮主要澄清前面小规模对拍时修复的几个问题。第一，外包 move 修改真实机器序列以后，现在已经尽量和原有机器 move 保持同一套提交格式：只要真实机器序列发生变化，就调用 `updateInformationM(m)`，重新计算该机器的成本、一维/二维函数缓存、累计 duration 和硬时间窗摘要。原来的纯机器 move 也是在 commit 最优操作以后刷新受影响机器，区别只在于外包集合本身不是一台普通机器，不需要维护 setup/time/PWLF 的机器缓存；它只维护 `outsourcedJobs`、`outsourcingBaselineTotal` 和 `outsourcingCostTotal`，通过 `G(B)` 的差分更新。也就是说，真实机器侧按原来的机器逻辑重建，外包侧按集合成本逻辑重算，两边的职责是分开的。

第二，`curUpperBound` 不是“当前解成本一定不会超过的上界”，而是分段函数 merge 和局部搜索评估里的剪枝阈值。ALNS 扰动以后得到的当前解、某台机器的原始成本，或者某个中间候选成本，都可能比当前全局最好解更差，因此也可能大于 `curUpperBound`。这不是数学上矛盾，而是因为 `curUpperBound` 更多表示“超过这个值就没必要继续精确计算”的 incumbent 阈值。旧逻辑里，`merge2Segments/merge3Segments` 在发现下界已经不可能优于 `curUpperBound`，或者函数为空时，会直接返回 `curUpperBound` 或 `big_M`。这个返回值本质是“候选不可用/不值得继续算”的信号，不是真实的新机器成本。如果继续拿它去减 `originCost`，就会出现 `curUpperBound - originCost < 0` 的假负 delta，从而把一个被剪掉的不可行候选误当成改进 move。

第三，`curCost` 不能再被 `curUpperBound` 截断。以前 `updateInformationM()` 会把总成本压到 `curUpperBound`，这在只有真实机器且只把 `curCost` 当比较值时问题不明显；但加入外包以后，`addOutsourcedJobs/removeOutsourcedJobs` 是按真实 `G(B)` 差分更新 `curCost` 的。如果 `curCost` 的基准已经被压成一个假上界，再减旧外包成本、加新外包成本，就会把真实目标值更新坏，极端情况下出现巨大负数。因此现在 `curCost` 始终保存真实目标值；`curUpperBound` 只用于分段函数剪枝和 move 候选过滤，不再污染解对象自身的成本状态。

第四，ALNS repair 现在先不做“不可行候选也进入搜索”的策略。机器插入位置如果因为硬时间窗、`CmaxH` 或分段函数 merge 被判成 `big_M/curUpperBound`，就不再放进 regret/randomized 的候选池；外包候选仍然按真实 `G(B)` 差分进入。这样 repair 阶段得到的仍然是可行解，再交给 VND 做局部搜索。后续如果要做允许不可行解的 ALNS，就需要单独设计 penalty、接受准则和恢复机制，不能简单把 `big_M` 位置混进当前候选池，否则 VND 会在一个状态语义已经坏掉的解上继续搜索。

2026-05-17 进一步把外包集合变化后的 `curCost` 更新改成重新汇总。原来外包侧的 `addOutsourcedJobs/removeOutsourcedJobs/replaceOutsourcedSegment` 是在旧 `curCost` 上做 `G(B)` 差分，即 `curCost += newG - oldG`。如果所有前置状态都绝对准确，这个写法本身没错；但外包 move 的提交通常分两步，先改真实机器序列并刷新机器成本，再改外包集合。为了避免第二步依赖第一步留下的中间 `curCost`，现在外包集合每次变化后都重新计算 `outsourcingBaselineTotal`、`outsourcingCostTotal`，再把 `curCost` 刷新为 `sum_m cost[m] + G(B)`。这个做法和真实机器侧 `updateInformationM(m)` 后重新汇总总成本的语义一致，只是外包侧不需要维护机器序列缓存。

这里顺便确认了一个历史点：`if (totalCost >= Utility.curUpperBound) { this.curCost = Utility.curUpperBound; } else { curCost = totalCost; }` 这段截断逻辑是旧代码里已有的，不是后来为了外包补进去的。它原来的意图大概是把解成本压到当前剪枝上界，减少后续计算；但现在 `curCost` 已经不仅是比较值，还会作为外包 `G(B)` 差分、ALNS/VND 状态同步和输出验证的真实目标值使用，因此不能再被 `curUpperBound` 污染。当前代码已经去掉该截断，`curUpperBound` 只保留为分段函数 merge 和 move 候选过滤的剪枝阈值。

## 19. 2026-05-17 更多小规模精确对拍

在前一轮 16 组小算例之后，又补了一个可复跑的批量对拍入口 `HEU.SmallExactHeuristicBatchTest`。这个测试不改主流程，只在测试类里随机生成小规模实例：任务数在 6、7、8 之间轮换，机器数为 2，同时随机生成 processing time、due window、earliness/tardiness 权重、setup time、setup cost 和外包 baseline。外包总成本函数这轮先固定为单段线性 `G(B)=rate*B`，这样可以减少 tariff 非凸分段带来的额外解释，重点验证外包、setup cost 和启发式 move 的主链路是否和精确模型一致。

精确基准仍然采用 `ArcFlowModel`。每个随机实例先用 ArcFlow 求精确目标值，再用当前 `EngineALNS + EngineVND` 启发式跑 12 个随机起点取最好值。这里选择多起点不是为了证明启发式理论上必然最优，而是为了在小规模上尽量排除随机初始解影响，把测试重点放在“外包 move、setup cost、硬时间窗、curCost 更新和不可行候选过滤是否仍然一致”上。

本轮共生成 40 组随机小算例，结果为 `matched=40/40`，最大差异约 `1.59999e-8`，属于 CPLEX 和 double 运算的数值误差范围。结果明细已输出到 `test-results/exact-heuristic/2026-05-17-small-exact-heuristic.csv`，其中记录了 case id、规模、ArcFlow 精确目标值、启发式最好目标值、gap、精确解外包任务数量和启发式最优起点 seed。后续又补充了计时列 `exact_ms` 和 `heuristic_ms`：当前 40 组上，ArcFlow 精确模型平均约 `315.98 ms/例`，启发式平均约 `92.85 ms/例`。这里的启发式时间包含 12 个随机起点的完整 ALNS+VND，因此折算单起点约 `7.74 ms`。同步复跑了 `OutsourcingMoveConsistencyTest`，结果仍为 `checked=10974`。

这轮测试支持当前判断：在小规模、线性 `G(B)`、带 setup time/setup cost 和允许外包的设定下，启发式当前实现与 ArcFlow 精确模型对齐。剩余不能由本测试覆盖的内容主要有两类：一是更复杂的分段/凹折扣 `G(B)`，后续正式使用多段 tariff 时还需要单独对拍；二是更大规模下启发式不保证每次都能找到全局最优，本测试只能说明小规模随机例上主逻辑没有暴露新的错误。

## 20. 2026-05-17 补充 ATIParallel 小规模计时对拍

在前一轮 `SmallExactHeuristicBatchTest` 只比较 ArcFlow 和启发式的基础上，本次把 `ATIParallel` 也接入同一批 40 个随机小算例。测试仍然使用同一份随机生成数据：6-8 个任务、2 台同质机器、带 setup time、setup cost 和线性外包函数 `G(B)=rate*B`。每个算例先由 `ArcFlowModel` 求精确值，再由 `ATIParallel` 求 time-expanded 精确值，最后由 `EngineALNS + EngineVND` 跑 12 个随机起点取最好值。这样做的目的，是同时验证两个精确模型是否一致，以及当前启发式在同一批实例上的目标值和耗时表现。

本轮结果为：40 个算例中，`ATIParallel` 与 `ArcFlowModel` 全部一致，启发式最优值也全部一致。最大差异约 `1.59999e-8`，属于 CPLEX 与 double 运算的数值误差范围。平均时间方面，`ArcFlowModel` 约 `144.10 ms/例`，`ATIParallel` 约 `198.48 ms/例`，启发式 12 起点合计约 `61.38 ms/例`，折算单起点约 `5.12 ms`。这说明在当前很小的随机实例上三者都很快，但启发式仍明显更快；`ATIParallel` 因为是时间扩展网络，耗时受 `CmaxH` 和弧数量影响，本轮平均比 ArcFlow 慢一些。

结果明细已写入 `test-results/exact-heuristic/2026-05-17-small-exact-heuristic.csv`。CSV 现在同时记录 `arcflow`、`parallel`、`heuristic`、两类 gap、三类耗时，以及 ArcFlow/ATIParallel 识别出的外包任务数量。当前这轮对拍支持的结论是：在小规模、线性外包函数、带 setup time/setup cost 的设定下，`ArcFlowModel`、`ATIParallel` 和当前启发式主链路目标值一致；后续仍需要用更复杂的分段/凹折扣 `G(B)` 和更大规模实例继续测试。
