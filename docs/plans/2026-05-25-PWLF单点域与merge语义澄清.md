# 2026-05-25 PWLF 单点域与 merge 语义澄清

这次只补三点语义说明，不改代码。

第一，`PiecewiseLinearFunction.add()` 现在可以处理“两函数相加后的公共可行部分只剩一个点”的情况。这里不是新加的逻辑，而是原实现本来就在按 `cur <= nxt` 处理公共区间，因此当 `cur == nxt` 时仍会保留一个零长度 segment。仓库里 2025-04-23 的旧注释已经记录过这个边界：如果把这里写成严格 `<`，某些被上界裁到只剩单点的函数相加后会直接变成空函数。

第二，“函数只剩一个点”和“`domainStart/domainEnd` 也一定变成同一个点”不是一回事。当前代码里二者已经刻意分开了。`GCBidirectional.cropToInterval()` 会先 `resetDomain(start, end)`，因此如果 half-domain 裁完后只剩 `Tmid` 一个点，那么元数据也会一起收成这个单点；但普通两参数 `setDomain()` 在物理裁剪后又把元数据恢复成原来的 `this.domainStart/this.domainEnd`。因此仓库现在允许一种状态：物理 segment 只剩一个点，但 domain 元数据仍然比它更宽。这里不是错误，而是为了兼容后续 `shiftX()/trimToDomain()` 这类依赖元数据边界的操作。

第三，`mergeMinimum()` 在当前 pricing 主链里不会拿 forward label 和 backward label 混着 merge。`PaperDominanceGraph` 是分方向建图的，forward 图只 merge forward frontier/envelope，backward 图只 merge backward frontier/envelope。此次补的 `mergeDisjointMinimum()` 兜底，处理的是“同向函数经过 half-domain 或 local horizon 裁剪后，没有正长度公共区间”的边界：要么只在一个点接触，例如 `[0, Tmid]` 和 `[Tmid, Tmid]`；要么左右相邻但不重叠，例如 `[0, 900]` 和 `[905, 1200]`。旧的 overlap merge 默认假设 `start < end`，这些边界下容易留下坏链表或直接在 dominance envelope 合并里崩掉，所以才单独分出 disjoint merge 路径。

再补三点更细的过程说明。第一，`cropToInterval()` 只在 `GCBidirectional` 里使用，它不是普通“裁一裁定义域”的通用入口，而是把函数显式变成 forward half 或 backward half 的工具。当前调用点只有四类：`sourceFrontier` 初始裁成 `[0,tMid]`；root/no-cut 下的 `baseForwardHalfPenaltyByJob` 和 `baseBackwardHalfPenaltyByJob` 缓存；以及动态 profitable window 收紧后新建的 `buildForwardHalfPenalty()` / `buildBackwardHalfPenalty()`。它和普通 `setDomain()` 最大的区别是：`cropToInterval()` 会同时把 `domainStart/domainEnd` 重设成这次 half-domain 的边界，因为后续 `shiftX()->trimToDomain()` 这条链就是靠元数据边界继续维护“只能活在这个半域里”的语义。若这里模仿普通 `setDomain()` 那样把物理段裁完后又把元数据恢复成旧全局域，那么 half-domain 的约束就只停留在当前一层物理段里，后续 shift/trim 会把它重新放宽。

第二，`add()` 的旧语义一直是“只改物理 segment，不主动重写结果函数的 `domainStart/domainEnd`”。代码里结果对象直接按左操作数的元数据 `new PiecewiseLinearFunction(domainStart, domainEnd)` 创建，之后真正相加时只取两边物理段的公共部分 `[start,end] = [max(head.start), min(tail.end)]` 去扫。因此前向函数里完全可能出现：`domainStart != head.start`，甚至 `domainEnd != tail.end`。仓库里的 `Utility.debugCheckPWLFRightBound()` 也明确把这种状态当成允许存在的情况，并在注释里写了“`tail.end` 是链表真实右端，`domainEnd` 是对象元数据；`setDomain/trimToDomain` 之后可能出现前者小于后者”。所以这里不能把“前向函数的右端都应该和元数据终点一样”当成现在的全局不变量；它只是在旧的一部分 forward pricing 语义里经常成立。

第三，当前 `mergeMinimum()` 真正该看的不是“会不会出现完全没交集”，而是“在 `PaperDominanceGraph` 里，输入给它的同向 envelope/frontier 是否仍然保证有正长度公共区间”。从代码路径看，`mergeMinimum()` 只发生在 `mergeGEnvelopes()`、`addLabel()`、`recomputePredecessorEnvelope()`、`recomputeDominanceEnvelope()` 这些同向 envelope 合并场景里。理论上旧 forward label 常常满足“右端同为全局 T”，但这次 local horizon/half-domain 进来以后，更常见的边界其实是：其中一个输入已经被裁成 `[Tmid,Tmid]` 这样的单点，另一个输入从 `Tmid` 或更靠右/左开始，于是两者的公共部分最多只剩一个点。这也是为什么当前最需要兜的是“零长度交集/单点接触”，而不是把“左右完全隔开的宽区间”当成主流输入。后者在当前 pricing 主链里并不是最核心的触发场景。

最后再把 `setDomain(hStart, hEnd, true) -> cropToInterval(...)` 这条 root/no-cut dynamic window 路径单独说明一下。这里其实是两步裁剪，语义不同。第一步 `setDomain(hStart, hEnd, true)` 是“按当前 `pi_j` 的 profitable window 收缩 job 自己的完成时间窗，但不把函数对象的全局右端契约直接砍掉”。`true` 这个参数的意思不是普通物理裁剪，而是把窗口外区间写成 `big_M` 段，尽量保留原函数的元数据右端和整体 `[a,T]` 契约。第二步 `cropToInterval(..., 0, tMid)` 或 `cropToInterval(..., tMid, pricingHorizon)` 才是在双向语义里把它明确改写成 forward half 或 backward half 函数，也就是把“当前这个 job penalty 只允许在对应半域上参与 label 递推”写进函数元数据。

2026-05-25 晚些时候重新按日志和代码路径复核后，需要把上一段判断收紧。当前没有找到任何真实运行日志表明 local horizon 版本曾因 `copy()` 抛出 `Invalid domain: start >= end`；`pricingHorizon/2` 那次 `wet021_001_2m` 的问题是 backward 标签爆涨到 `54053/625941`、耗时升到 `67.971s`，不是异常崩溃。进一步沿当前代码链检查后，`setDomain(hStart,hEnd,true)` 若窗口退化，会先尝试用 `WINDOW_EPSILON` 扩成正长度小区间，仍不行就直接返回整段 `big_M`；而 root local horizon 这条主路径里的 `cropToInterval(...,0,tMid)` 和 `cropToInterval(...,tMid,pricingHorizon)` 传入的 `start/end` 也都是正长度区间。因此“这条主路径自然生成单点元数据定义域，再把旧 `copy()` 触发炸掉”这一点，目前证据并不成立。更合理的结论是：`copy()` 这次修改更像预防性稳健补丁，用来允许今后显式构造出的单点域函数被安全复制；而本轮 local horizon 真正能从代码契约上解释得通的风险点，仍然是 `mergeMinimum()` 原先默认 `start < end` 的正长度公共区间假设被打破，也就是同向 envelope/frontier 若只在单点上接触，旧 overlap merge 不够稳。换句话说，当前日志能证明的是“midpoint 取半导致状态爆炸”，以及“`mergeMinimum()` 的旧输入契约过强值得兜底”；不能证明“这次确实是 `copy()` 炸过”。

基于这个复核结论，2026-05-25 又把 `PiecewiseLinearFunction.copy()` 的单点域兜底改动撤回，只保留 `mergeMinimum()` 的零长度公共区间兜底。原因不是否认单点函数永远不可能出现，而是当前这条 root local horizon 主路径没有提供足够证据证明 `copy()` 改动是必要的；在缺少实际异常日志的情况下，优先回到更简单、证据更强的实现更稳。

2026-05-25 继续沿 `Tmid` 半域切分本身复核后，当前更准确的根因应写成这样。问题不在于 `cropToInterval(...,0,tMid)` 会把前向 job penalty 的元数据直接裁成单点；前向这条路径的元数据仍是 `[0,tMid]`，而 `setDomain(hStart,hEnd,true)` 在窗口左侧通常还保留 `big_M` 段。真正会退化的是 label/frontier 的“当前半域内有效物理支持”。对前向来说，`shiftX(delay) + jobPenalty` 之后再做 `normalize(FORWARD)`，若该 label 在 `[0,tMid]` 里唯一还能保持有限成本的完成时刻就是 `Tmid`，那么这个 label 在语义上就等价于“最早可行完成时间已经顶到 `Tmid`”；由于 forward 经过 prefix-min 后整体非增，只要还能在 `Tmid` 左边留出任何正长度有限区间，就不会退化成这种边界。对 backward 则是对称的：若在 `[Tmid,pricingHorizon]` 里唯一还能保持有限成本的完成时刻是 `Tmid`，就等价于“最晚可行完成时间已经压到 `Tmid`”。因此，`Tmid` 取得过左或过右，真正改变的是两侧 label 在各自半域内还能保留多少正长度有效支持，而不是简单地把 job penalty 元数据裁成一个点。

基于这点，当前还剩的风险也应收紧到两条。第一，`pricingHorizon/2` 那次性能崩溃的根因仍然是 midpoint 位置导致 backward 状态空间爆炸，不是 `copy()` 崩溃；因此主问题仍是 split 位置，而不是对象复制。第二，当前 `mergeMinimum()` 的 `start >= end -> mergeDisjointMinimum()` 兜底写得比主链真实需求更宽。按现有同向 envelope/frontier 语义，最应重点支持的是 `start == end` 的单点公共区间；若出现大量 `start > end` 的严格不相交，更像是上游 half-domain 或 envelope 契约被破坏，不能完全把它当成正常输入吞掉。后续若继续收紧这块，优先应该检查 `mergeMinimum()` 是否只需要保守兜住单点接触，而不是先回头重做 `copy()`。

2026-05-25 继续沿“把单点 label 从 dominance graph 主路径里旁路出去”这条思路分析后，可以把可行性判断再收紧一层。这个方向总体上是合理的，而且和 `mergeMinimum()` 旁边原注释“单点 label 不应进入 mergeMinimum、应在 pricing 层单独处理”的设计意图一致。但真正应当检测的不是预估量本身，而是 `normalize(FORWARD/BACKWARD)` 之后 frontier 的结构：前向若 `head.start == tail.end == Tmid`，就等价于“最早可行完成时间已经顶到 `Tmid`”；后向若 `head.start == tail.end == Tmid`，就等价于“最晚可行完成时间已经压到 `Tmid`”。这类 label 在各自方向上都不可能再继续扩展，因为再加任何正 processing/setup 都会越过 `Tmid`。因此，把它们从 `PaperDominanceGraph` 的 envelope merge / propagate 链路中剥离出来，在结构上是干净的。

但这个方案里有两处需要特别保守，不能直接按“每个 reachable set 只保留一个单点 label”去做。第一，双向 join 明确还要看 `visitedSet` 的相交关系；即使两个单点 label 终点 job 相同、`reachableSet` 也相同，它们的 `visitedSet` 仍可能不同，而这会直接影响后续能否和某个 backward/forward 半边拼起来。因此，“每个集合 S 下最多一个单点 label”在 bidirectional 里并不自动安全。只有在更强的条件下，例如候选 label 既满足 `reachableSet` 支配、又满足 `visitedSet` 更小（或至少不更大）、并且 `Tmid` 处 reduced cost 不差，才可以把另一个单点 label 彻底删掉。第二，若想让 graph 里的普通 node/envelope 去判掉新来的单点 label，按当前架构只能说“这与现有 reachable-set 主导的 dominance 语义一致”，但并不能单独强化成严格的 join-safe 证明；因为 graph envelope 本身不记录 `visitedSet`。因此，更稳的做法是：单点 label 不参与 graph 更新；单点 label 之间单独维护一份旁路结构，并在这份结构里使用更强的判定（至少把 `visitedSet` 也纳入比较）；至于 graph 是否还能先粗剪掉一部分单点 label，应视作和现有 dominance 模型保持一致的工程优化，而不是新的安全性来源。

如果后续真要实现，最稳的第一版不是搞复杂索引，而是按 terminal job 和方向各维护一份单点 label 列表。因为单点 label 本来就是边界情形，数量预期不会像普通 label 那么大；先暴力扫描这份小集合，比继续把它们塞回 `PaperDominanceGraph` 再依赖 `mergeDisjointMinimum()` 更可控。可以后续再按 `reachableSet` 基数做分桶，减少明显不可能的 superset/subset 比较。另一方面，join 常数延拓这条路当前本身是兼容单点 label 的：`buildForwardJoinExtension()` / `buildBackwardJoinExtension()` 都是新建 full-domain 函数，再把半域 frontier 和常数延拓段拼进去，不会回写原 frontier 的 domain 元数据；因此单点 label 只要还保留在 `activeForwardByLastJob` 或对应 backward 活跃结构里，join 这一层不需要额外改 domain 逻辑。

2026-05-25 再次按当前代码语义复核后，需要把上面那段再收紧。第一，当前 `normalize` 对单点是能处理的，不需要先怀疑这一层：`add()` 本身已经保留单点公共域；`minimizePrefixInPlace()` 和 `minimizeSuffixInPlace()` 里也各有“整条结果只剩一个点”时的补段逻辑。因此，只要扩展后的 frontier 真退化成 `[Tmid,Tmid]`，当前对象层是能把它留住的。第二，在现在这套非 partial-dominance、单窗口 `setDomain(...,true)` 的实现下，单点判定确实可以直接用“前向最早完成时间 / 后向最晚完成时间是否等于 `Tmid`”来做，不必等构造完整 frontier 后再看 `head.start==tail.end`。原因是：前向经 `normalize(FORWARD)` 后，左侧连续 `big_M` 会被删掉，右侧 `big_M` 会通过 prefix-min 传播成常数，因此一旦最早可行完成时间已经到 `Tmid`，后面不可能再出现“`Tmid` 之后或之前还有一段新的 `big_M` 缝隙”；同理，后向经 `normalize(BACKWARD)` 后，右侧连续 `big_M` 会被删掉，左侧 `big_M` 会通过 suffix-min 传播成常数，因此若最晚可行完成时间已压到 `Tmid`，也就等价于整个 half-domain 有效支持只剩这个点。也就是说，对当前实现，`earliestCompletion==Tmid` 和 `rhoPrime==Tmid` 基本就是 single-point frontier 的 O(1) 判据。

第三，关于 `visitedSet` 需要明确分成两层。用户指出得对：按“当前已经写进代码的 dominance 语义”，`visitedSet` 和占优本身没有关系，当前 `PaperDominanceGraph` 的 node key、候选搜索和 `dominates()` 检查都只看 `reachableSet + frontier`。因此，如果只是要让普通 dominance graph 或普通单点 label 结构去判一个新单点 label 是否被已有 label 干掉，就不该额外把 `visitedSet` 引进来，我前一版把它放进“当前占优条件”里是错的。`visitedSet` 真正重新变成风险的地方，不是现有 dominance，而是额外再加一条更强压缩：“每个 reachable set 只留一个 single-point label”。这条压缩比当前 dominance 更强，而 join 又显式检查 `visitedSet.intersects(...)`，因此它不能只按 `reachableSet` 和 `Tmid` 处 reduced cost 做唯一化。更稳的边界应是：单点 label 的“被已有普通/单点 label 占优并删除”仍然沿用当前 dominance 语义，不引入 `visitedSet`；但若想进一步把多个单点 label 压成“每个 S 一个”，那就必须另行证明 join 不会被破坏，否则默认不能这么压。

2026-05-25 晚些时候再纠正一个细节：上面把前向的 `earliestCompletion==Tmid` 写成 single-point 判据仍然过粗，因为当前代码里的 `earliestCompletion = frontier.head.start + delay` 只是 `isDirectForwardExtensionTimeFeasible(...)` 用的便宜可行性下界，它没有把新增 job 自己的左侧可行窗 `HStart` 算进去。也就是说，这个量只能说明“child 最早不可能早于这里完成”，不能直接等价成扩展后 frontier 的最左有限点。真正正确的 O(1) 判据应当是“扩展后最早有限完成时间是否等于 `Tmid`”——如果后续要在不构造完整 frontier 的前提下做这个判断，就得显式补前向 `HStart` 或者从新增 job penalty 里取第一个非 `big_M` 点；否则最稳妥的实现仍然是先构造并 normalize child frontier，再看它是否退化成 `[Tmid,Tmid]`。不过一旦 label 已经确认是 single-point，那么后续比较就不该再走 `dominates()` 或 `evaluate()`：因为所有 single-point label 的完成时间都固定在 `Tmid`，对同方向、同 terminal job、同 reachable set 的候选，直接比较 `Tmid` 处的 reduced cost 标量即可，forward 用 `tail.getValue(tail.end)`，backward 用 `head.getValue(head.start)`。

2026-05-25 又补一个容易说反的点：前向从 `j` 扩到 `i` 时，child 的“最早有限完成时间”不是取两个来源的较小值，而是取较大值。因为它既不能早于父 frontier 经过 `delay = s_{ji}+p_i` 平移后的最早完成时刻，也不能早于新增 job 自己窗口左端允许出现有限值的最早点。因此真正的语义是 `max(parentEarliest + delay, HStart(i))`。当前 `isDirectForwardExtensionTimeFeasible(...)` 里写的 `frontier.head.start + delay` 只是这两个约束里的前一半，用来和 `hEnd/Tmid` 做 O(1) 便宜过滤；它没有把 `HStart(i)` 纳入，所以不能直接拿来当 single-point 判据，更不能把它误写成“取两个来源的较小值”。

2026-05-25 后续实现先收在低风险范围内。第一，单向 `GC` 和双向 `GCBidirectional` 都给前向动态窗口补了 `HStart` 缓存，并把前向 direct feasibility 从 `parentEarliest + delay <= HEnd` 收紧为 `max(parentEarliest + delay, HStart) <= HEnd`，双向还同时要求不超过 `Tmid`。这不会改变可行列语义，只是把原本要到 `shift/add/normalize` 后才发现的明显无效扩展提前剪掉。第二，`PaperDominanceGraph` 和旧 `DominanceGraph` 在调用 `dominates()` 前增加方向相关的定义域覆盖必要条件：forward 下候选 envelope 的 `head.start` 不能晚于目标 label，backward 下候选 envelope 的 `tail.end` 不能早于目标 label。这个检查只跳过必不可能支配的函数扫描，不改变 dominance 语义。第三，`mergeMinimum()` 暂时不继续收紧，因为当前代码里还没有真正落地“单点 label 单独存储并完全绕开 dominance graph”的结构；现有的 `zeroDualSingletonOnlyJobs` 只是 `pi_j=0` 任务不参与多任务列的剪枝，不等同于 single-point frontier 旁路。因此在单点旁路实现前，`mergeMinimum()` 的零长度边界兜底仍保留。
