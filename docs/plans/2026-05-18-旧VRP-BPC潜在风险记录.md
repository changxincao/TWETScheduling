# 旧 VRP BPC 流程中的潜在风险记录

这份记录只讨论旧 VRP 参考代码中已经被我们反复对照过的几个边界问题，不评价代码整体好坏。旧代码在常规算例上能跑通，很多处理也是典型的工程折中；这里记录的是后续迁移到 TWET-BPC 时不能无脑照搬的地方，尤其是分支子节点列集修复、slack、列筛选和 `GCNGBB` 动态 ng-set 的状态管理。

## 1. 分支修复中的列数上限不是严格不可行证明

旧代码位置主要在 `BPC/BP/BranchA.java`、`BranchB.java`、`BranchC.java`、`BranchD.java` 的 `UpdateRouteSet(...)`。以 `BranchD.UpdateRouteSet()` 为例，子节点 route set 不足以满足当前分支约束时，会先 `lp.AddSlack(...)`，然后进入修复循环：

```text
old_col_number = col_number
col_number = hgc.FindFeasible(...)
如果 Tabu 新增列，则 gc.Reset()
col_number = gc.FindFeasible(...)
如果 old_col_number == col_number 或 col_number > m_branch_col_number，则停止
```

这里的 `m_branch_col_number=500` 是工程上限，不是数学意义上的不可行证明。也就是说，若达到 500 条修复列但 slack 仍未归零，旧代码会停止并认为该分支空间修复失败；严格地说，这更像“repair limit reached”，而不是“已经证明子节点不可行”。当前 TWET-BPC 里 `maxBranchRepairColumns=500` 是复刻这个参数，因此后续状态输出最好区分“无新列可加”和“达到修复列数上限”。

## 2. `GCTabu.FindFeasible()` 和 `GCNGBB.FindFeasible()` 的信息流不同

旧 `GCTabu.FindFeasible()` 不是只返回候选列。它内部会循环执行：

```text
解带 slack 的 LP
如果 IsNoSlack()，返回 -1
读取 dual/value
执行 Tabu 扩展
将新列加入 lp.pool
调用 lp.AddColumn(gn_index) 直接接入当前 CPLEX 模型
继续下一轮
```

因此旧外层再调用 `GCNGBB.FindFeasible()` 时，`GCNGBB` 看到的可能已经是 Tabu 加列并重解后的 LP。当前 TWET-BPC 的 repair 流程不同：`HeuristicPricingEngine.findFeasible(lp)` 和 exact engine 的 `findFeasible(lp)` 先基于同一轮 LP dual 返回候选列，再由 `PC` 统一 `addColumns()` 和 `resolveCurrentModel()`。这更干净，但不完全等价于旧代码的信息流。若后续要更贴近旧 VRP，应考虑 repair mode 下每个 engine 生成列后立即加列并重解，再决定是否继续调用下一个 engine。

## 3. `GCNGBB.Reset()` 的状态管理是经验性的

旧 `GCNGBB.Reset()` 只清空：

```java
Arrays.fill(m_low_ng_set, 0);
Arrays.fill(m_high_ng_set, 0);
```

它不会清 LP、不会清 route pool，也不是清 dominance graph。`GCNGBB.Solve(lp)` 正常 pricing 开头会无条件 reset；但 `GCNGBB.FindFeasible(lp,n,col_number)` 本身不 reset，只调用一次 `Extend(lp,n)` 并检查 slack。

`Extend()` 内部会在一次 pricing 调用中反复执行 `FWExtend/BWExtend/Join/UpdateNGSet`。如果没有生成 elementary route，但发现 duplicate cycle，`UpdateNGSet()` 会把 cycle 中的 customer 加入相关节点的 ng-set，然后下一轮用更紧的 ng-memory 重新 label。这就是旧代码注释里说的 decremental state-space refinement on ng-route。它思想上接近 DSSR，但实际调用的不是 `GCDSS` 类，而是 `GCNGBB`。

旧 `UpdateRouteSet()` 只有在前面的 `GCTabu.FindFeasible()` 新增列时才调用 `gc.Reset()`。如果上一次进入 `GCNGBB` 后仍不可行，下一次再次进入 `GCNGBB`，只要中间 Tabu 没新增列，就会保留前一次 exact pricing 中动态扩充的 ng-set。这个设计可以理解为：`GCNGBB` 自己发现的重复环信息继续保留，而外部 Tabu 改变 LP 和 dual 后让 exact pricing 从松状态重来。它不是唯一合理方案，因为 `GCNGBB` 自己新增列也会改变 LP；但旧代码就是这样管理状态。

## 4. 修复成功后的 route set 筛选不是严格保留所有基列

旧 `UpdateRouteSet()` 在 slack 修复成功后会 `lp.GetVaule()`，但真正返回给 child 的 route set 不是注释掉的 `route_active_set`，而是遍历 `lp.route_index` 中所有当前列，按如下规则筛：

```text
reduced cost < m_addin_red_cost
且不违反 child 的 feasible_arc
若超过 m_initial_col_number，则按 reduced cost 排序取前若干条
```

旧参数为 `m_addin_red_cost=5000`、`m_initial_col_number=1000`。通常 LP 正值列 reduced cost 为 0，会被这个宽阈值保留；因此常规情况下 child route set 仍然可行。但这不是严格保证。若低 reduced-cost 列数量太多，或者某些正值列在 child 中被 `feasible_arc` 判断为不兼容，维持 coverage 或分支可行性的关键列仍可能被删掉。

因此旧代码这里存在边界风险：repair 成功时当前 LP 可行，不等于重筛后的 child route set 一定可行。它主要依赖“大阈值 + 大保留数量 + 后续节点继续 AddSlack/FindFeasible 修复”来降低风险。

## 5. 旧代码只给当前分支约束加 slack，不修 coverage

旧 `LP.AddSlack(cid,sid)` 是给一条 `branch2rng` 对应的分支约束加人工变量。以 arc 分支为例，`BranchD.UpdateRouteSet()` 调用：

```java
lp.AddSlack((cid1+2)*data.m_customer_number+cid2, sid);
```

这只修当前新增的分支约束，不给 coverage 行加 slack。对于纯 VRP set partitioning，这在大多数情况下能工作，因为父节点 route set 本来覆盖所有 customer，child 通常只是新增分支约束。但如上所述，后续 reduced-cost 筛选和 `feasible_arc` 过滤仍可能删掉维持 coverage 的列，旧代码没有专门防这个边界。

TWET-BPC 当前保留 coverage slack 的原因正在这里：TWET 有不能外包任务、外包变量、tariff segment、机器数下界和 required arc 等结构，child restricted column set 暂时缺覆盖列的可能性更明显。如果后续要减少 slack 数量，优先方向不是合并成一个公共 slack，而是增强 child seed selection：兼容 child 的正值列优先保留、不能外包 job 至少保留覆盖列、required arc 相关列优先保留或构造，最后再按 reduced cost 补足列数。

## 6. 为什么旧代码通常仍然可用

上述问题都是边界风险，不代表旧代码在常规数据上一定出错。它能工作的原因主要有三点。第一，筛列阈值很宽，`m_addin_red_cost=5000` 通常足以保留 LP 基列和近似有用列。第二，`m_initial_col_number=1000` 对很多 VRP 算例来说足够大，不太容易把关键列挤掉。第三，即使某个 child route set 暂时不够，后续进入该节点时仍会再次通过 `AddSlack + GCTabu/GCNGBB FindFeasible` 修复。

迁移到 TWET-BPC 时，不能只复制“看起来能跑”的部分，而要把这些隐含前提写清楚。当前更稳的方向是：保留旧代码的低 reduced-cost 继承和 FindFeasible 修复思想，同时在状态输出上区分 repair limit 与真正不可行，在 seed selection 上显式保护关键覆盖列，在 slack 上按需要的约束行独立添加，而不是使用公共 slack。

## 2026-05-18：FindFeasible 与 AddSlack 的边界风险补充

本轮进一步明确旧 VRP `UpdateRouteSet()` 中 `FindFeasible` 和 `AddSlack` 的两个边界风险。

第一，`FindFeasible` 本身是有限补列过程。无论是 Tabu 还是 NGBB，旧代码都会受到每轮加入列数、总补列数和时间等工程上限影响。如果在这些限制内仍然没有把 slack 压回 0，`UpdateRouteSet()` 就会返回 null，child 随后被丢弃。严格来说，这不能证明完整列空间下该 child 不可行，只能说明当前修复过程没有在有限预算内找到足够的列。

第二，`AddSlack` 只加在当前分支约束行上，但 child LP 不可行的原因不一定只来自当前新分支约束。由于 child route set 是从父节点当前 LP 中按 reduced cost、数量上限和分支兼容性筛出来的，筛选可能删除一些列。这样一来，不可行也可能来自 coverage 约束、前面已经存在的分支约束，或者其他被当前列集合暂时缺失触发的约束。旧代码之所以只对当前分支约束加 slack，是基于一个工程判断：筛选出来的列通常已经满足之前所有分支状态，且 reduced-cost 阈值和保留数量足够大时，coverage 大概率仍然能满足；因此最常见的新不可行来源是当前刚加入的分支约束。

所以旧实现是实用策略，不是严格证明。它在列保留足够宽时通常能工作，但边界情况下可能把一个完整列空间下可行的 child 当作不可行。当前 TWET 保留 coverage slack 和 required-arc slack 的设计，就是为了降低这类 restricted-column 可行性误判风险。

## 2026-05-18：UpdateRouteSet 首次 LP 与分支约束位置

本轮确认了旧 VRP `UpdateRouteSet()` 中第一次 `lp.cplex.solve()` 的位置。分支约束在进入 `UpdateRouteSet()` 之前已经写入当前 LP。以 arc 分支为例，左分支会先调用 `ForceArcValue(i,j,0,0)` 新增或设置聚合弧约束，右分支则把同一条 `branch2rng` 的 bounds 改成 `1,1`，然后才调用 `UpdateRouteSet()`。车辆数分支也是先修改 `cus2rng[0]` 的上下界，再进入 `UpdateRouteSet()` 求解。

因此，`UpdateRouteSet()` 里第一次 LP 求解如果可行，说明在当前父节点列集合下，加入该 child 分支约束后已经存在一个 restricted LP 可行解，不需要立刻 `AddSlack`。如果第一次 LP 不可行，原因不能简单说成“只由当前分支约束引起”。当前分支约束是触发条件，但 restricted column set 也参与了这个结果：父节点已有列可能不足以同时满足 coverage、历史分支约束和当前新分支约束。arc 左分支 `x_ij=0` 时，原 LP 中含该 arc 的列仍在模型里，只是被聚合行压到 0；如果 coverage 很依赖这些列，restricted LP 就可能因为缺少替代列不可行。arc 右分支 `x_ij=1` 时，父 LP 虽然在该 arc 上有分数流，但现有含 arc 的列未必能和 coverage、其他历史分支约束一起组成可行解。车辆数分支也类似，单条 route 不存在“自动满足车辆数分支”的说法；车辆数是所有 route 变量之和的全局约束，列只是贡献系数 1，是否满足上下界取决于 LP 组合。

旧代码后续只对当前分支约束行加 slack，是因为它把当前新分支当作最主要的新冲突来源，同时依赖后续 `FindFeasible()` 补列来恢复 coverage 和其他约束的可行性。严格说，如果 restricted columns 本身缺覆盖列或缺满足历史分支的列，当前分支行 slack 不一定足够解释不可行性，这也是前面记录的边界风险。

## 2026-05-18：车辆数分支的列筛选方式

旧 VRP 的车辆数分支对应 `BranchA`。分支时不是筛选某类 route，而是直接修改车辆数约束行 `cus2rng[0]` 的上下界：左分支把 `max_vehicle_number` 设为 `floor(vehicle_number)`，右分支把 `min_vehicle_number` 设为 `floor(vehicle_number)+1`。这个约束行在 `LP` 初始化时就是 `cplex.addRange(node.min_vehicle_number, node.max_vehicle_number, "c,0")`，每条 route 作为一辆车，在该行中的系数都是 1。

因此，车辆数分支下单条列没有“是否满足车辆数分支”的过滤条件。是否满足车辆数上下界，取决于 LP 中 route 变量的组合数量，而不是某一条 route 自身。`UpdateRouteSet()` 首次求解时已经带着新的车辆数上下界；如果成功，后续返回 child route set 时只按 `reduced cost < m_addin_red_cost` 筛选，再按 `m_initial_col_number` 控制数量，没有像 arc 分支那样检查 `feasible_arc`。这和 arc 分支不同：arc 分支会在返回 route set 时过滤违反 child `feasible_arc` 的 route，因为某条 route 本身可以含或不含某条弧；车辆数分支则只能由列组合整体满足。
