# 启发式局部搜索与参考文献对应关系

## 当前结论

这次只对已经实现的启发式局部搜索操作做对照，不把论文里有但代码里没有的内容算进来。当前 `HEU` 包里的主体邻域算子，和 Ibaraki et al. 2008 的 `iopt、2-opt、2-opt*、path insertion、cross exchange` 五类标准邻域基本是一一对应的；其中 `iopt` 在 Ibaraki et al. 2008 里是 Or-opt 的变体，在 Ibaraki et al. 2005 里对应 intraroute neighborhood。当前代码也继承了这类方法最关键的评价思路：先为原序列、反转子序列、前缀和后缀预处理分段线性函数，然后在候选 move 中用两段或三段拼接快速计算变化后的最优惩罚值。

但外层搜索框架不是两篇文章的原样实现。2008 文章使用的是 ILS，扰动主要是随机 cross exchange，然后再按固定顺序搜索五类邻域；2005 文章的重点还包括 cyclic-exchange、MLS/AMLS 和一般时间窗惩罚函数。当前项目中真正使用的是 `EngineALNS + EngineVND`：ALNS 负责 destroy/repair 产生新解，VND 再调用局部搜索算子细化。因此可以说当前代码借用了论文里的局部搜索算子和函数拼接评价方式，但外层元启发式已经改成了适配本项目的 ALNS/VND 框架。

## 算子对应关系

`IOptOperator` 对应 2008 文章里的 `iopt`，也对应 2005 文章里的 intraroute/Or-opt 类操作。论文里的做法是从同一条 route 中取出一段连续 path，再插入到同 route 的另一个位置，并允许普通插入和反向插入；当前实现也是同一机器内部移动一段连续任务，`allowReverseIOpt=true` 时会同时考虑反转。差异在于参数不完全一致：当前代码里 `LPATH=10`、`LINS=6`，而 2008 文章实验参数中 `Liopt_path` 更小、插入距离限制更大；代码还加入了 `noImprovedSeq` 这类历史无改进缓存，这不是论文中的 distance-oriented/time-oriented neighbor-list。

`TwoOptOperator` 对应 2008 文章里的单 route `2-opt`。论文中它是在同一路径上切断两条边并反转中间段；当前代码也是对同一机器序列 `[a,b]` 反转，并通过 `merge3Segments(...)` 评价“前缀 + 反转中段 + 后缀”的新成本。2005 文章的标准邻域列表主要写 intraroute、2-opt* 和 cross-exchange，没有把单 route 2-opt 作为同等独立重点；所以这个算子主要按 2008 来理解。

`TwoOptStarOperator` 对应两篇文章里的 `2-opt*`。论文定义是从两条 route 各切一条边，把两条 route 的后半段交换；当前实现也是两台机器之间交换 cut 后的尾段，并用 `merge2Segments(...)` 分别重算两台机器的成本。当前代码额外允许 `allowReverseTwoOptStar=true`，即尾段可以反转后再拼接，这比论文中的基本 `2-opt*` 更宽一些。2008 文章也提到在某些 depot/vehicle time penalty 不对称时，`2-opt*` 需要更谨慎处理；在本项目里可以把 setup time 理解为 VRP 的 travel time，把 processing time 理解为 service time，但非零 setup 下还要继续注意 Cmax 剪枝是否安全。

`PathInsertOperator` 对应 2008 文章里的 path insertion。论文做法是从一条 route 删除长度受限的 path，插入到另一条 route 中，并可按 path 长度限制决定是否考虑反向插入；当前实现是从一台机器取出一段连续任务，插入另一台机器的某个位置，`allowReversePathInsert=true` 时考虑反转。代码中的评价方式和论文思路一致：删除段的机器可以看成两段拼接，插入段的机器是三段拼接。差异主要是当前 `LSEG=20`，并且有 `mergeCmaxValidation(...)` 这种针对本项目 makespan 上界的快速过滤。

`CrossExchangeOperator` 对应两篇文章里的 cross-exchange。论文中 cross exchange 是从两条 route 各取一段 path 互换，并在必要时考虑反向插入；当前实现就是两台机器各取一段连续任务互换，并允许两个片段分别反转。它同样通过两次三段拼接计算两台机器的新成本。这个算子和论文形式最接近，只是当前参数、剪枝条件和缓存策略是项目内实现。

## 搜索流程差异

2008 文章的 LS 明确按 `iopt -> 2-opt -> 2-opt* -> path insertion -> cross exchange` 的顺序搜索，只要某个邻域找到改进，就回到 `iopt` 重新开始；五个邻域连续没有改进才停止。当前 `EngineVND` 的结构也是“某个邻域找到改进就 commit 并重启一轮”，这一点和论文一致。但当前代码在 `buildOps()` 里对五个算子做了 `Collections.shuffle(ops, new Random(1))`，所以搜索顺序不是论文里的固定顺序，而是一个固定随机顺序。这个差异会影响实际搜索轨迹，但不改变每个算子的邻域定义。

2005 文章的局部搜索标准部分是 intraroute、2-opt*、cross-exchange，然后再接 cyclic-exchange；如果 cyclic-exchange 找到改进，再回到 intraroute。当前代码没有实现 cyclic-exchange，也没有实现文中用 improvement graph 隐式搜索 very large-scale neighborhood 的部分。因此当前实现和 2005 的关系主要体现在两个方面：一是 intraroute/2-opt*/cross-exchange 这几个标准邻域；二是基于 forward/backward minimum penalty function 的快速评价思想。2005 里更强的 cyclic-exchange 和 MLS/AMLS 外层框架，目前没有落地。

当前 `EngineALNS` 里的 random removal、worst removal、Shaw removal、greedy insertion、regret insertion 和 randomized insertion，不属于这两篇文章里的局部搜索邻域本体。它们是本项目外层大邻域搜索的 destroy/repair 组件，用来产生候选解，再交给 VND 做局部细化。也就是说，若只比较“局部搜索操作”，核心还是 `Move.java` 中的五个算子；若比较“整个启发式算法”，当前实现已经不是 Ibaraki 2008 的 ILS，也不是 Ibaraki 2005 的 MLS/AMLS。

## 评价函数实现关系

两篇文章的共同核心是：给定一条 route 的访问顺序后，用动态规划形式的 forward/backward minimum penalty function 计算该顺序下最优时间安排；在局部搜索中，候选解只改变一两条 route 的若干段，因此可以用预处理好的前缀、后缀、子路径函数进行快速拼接。当前 `Solution.merge2Segments(...)` 和 `Solution.merge3Segments(...)` 正是这个思想在 TWET 调度里的改写：`merge2Segments` 对应两段直接连接，`merge3Segments` 对应“前缀 + 中间段 + 后缀”的三段拼接。

这里的映射关系可以理解为：VRP 里的客户变成本项目的任务，route 变成机器上的任务序列，travel time 变成 setup time，service time 变成 processing time，客户时间窗惩罚变成任务 earliness/tardiness 或 due-window 惩罚。因此，当前局部搜索在数学结构上和文献方法是同源的；但由于本项目现在有 parallel machine、Cmax 上界剪枝、可能非零 setup、以及后续 BPC pricing 的需求，代码里额外加了一些论文没有直接讨论的工程约束。

## 当前判断

如果只问“现在实现了的局部搜索操作和论文是否一样”，结论是：邻域类型基本一致，尤其和 2008 文章的五类邻域高度对应；2005 文章中当前只实现了标准邻域部分，没有实现 cyclic-exchange。具体实现不是逐字复刻论文，主要差异在参数、搜索顺序、剪枝方式、历史无改进缓存、是否允许更多反转组合，以及外层元启发式框架。

后续如果要继续对齐论文，可以优先考虑两件事。第一，确认是否要把 `EngineVND` 的邻域顺序改回 2008 的固定顺序，还是保留当前固定随机顺序。第二，如果希望更接近 2005 的算法贡献，则需要单独设计 cyclic-exchange 或者至少把它作为明确未实现项记录；这不是简单改参数能得到的。

## 实现正确性补充检查

如果问题不是“是否对应论文”，而是“当前实现逻辑是否正确”，结论要更谨慎。单个候选 move 的构造和两段/三段函数拼接评价，整体是按论文思路写的，`IOptOperator`、`TwoOptOperator`、`TwoOptStarOperator`、`PathInsertOperator` 和 `CrossExchangeOperator` 的序列变换本身也基本能对应到各自邻域。但当前实现里仍有几个会影响局部搜索完整性或稳定性的点，不能直接说“完全正确”。

第一类问题在跨机器算子的 `savedOperations` 和 `noImprovedSeqPair`。如果算法设计目标就是“每轮只保留最好的 move 或少量 record-best move”，那么 `savedOperations` 只保存全局 record-best 本身可以接受，不必保存所有改进候选。真正要注意的是后面的缓存语义：`commit()` 目前只根据 `savedOperations` 更新 `improvedPair`，并把没有出现在 `savedOperations` 里的机器对加入 `noImprovedSeqPair`。这会把“有改进、但不是全局最好”的机器对误记为无改进。更稳的做法不是一定要提交这些 move，而是在搜索时单独记录 `pairHasImprovement`，只有一个机器对确实没有任何改进候选时，才加入 `noImprovedSeqPair`。

第二类问题是跨机器算子的提交语义需要明确。如果目标是“只执行全邻域最好的一个 move”，那当前 `commit()` 并不是这个语义，因为它会把 `savedOperations` 排序后，批量提交互不重叠机器上的多个 move。从“互不重叠机器、成本可加”的角度看，批量提交可以作为工程加速，并不一定错；但它不是论文里典型的一次一个 best-improvement。也就是说，这里不是候选评价错误，而是需要明确当前实现到底要“单 move best-improvement”，还是“互不重叠 move 的贪婪批处理”。

第三类问题是 `OptCost.compareTo()` 里直接 `(int)(this.deltaCost - o.deltaCost)`。如果当前输入和成本计算都保证整数，这个问题实际影响不大；但从代码语义上看仍然建议补注释或改成 `Double.compare(...)`，因为分段函数斜率、权重或后续 reduced cost 一旦变成小数，强转 int 就会导致排序不稳定。现阶段它不是最优先风险。

第四类问题是 `PathInsertOperator` 的反向枚举里有一个很像笔误的判断：`if(len==1) break;`。在这个实现里 `len=0` 表示单任务片段，`len=1` 表示两个任务片段。单任务反转没有意义，所以如果只是为了避免反向重复，应该跳过的是 `len==0` 时的 reverse；现在写成 `len==1`，会导致两个任务片段只测试普通方向，不测试反向方向，而单任务片段反而测试两次。这个判断比较确定会漏掉长度为 2 的反向 path insertion 候选，只是实际影响大小需要测试。

第五类问题是 `TwoOptStarOperator` 和 `CrossExchangeOperator` 的 pair cache 使用 `Set.of(seq1, seq2)`。如果当前初始化和实例规模能保证每台机器都有任务、且不同机器任务序列不可能完全相同，那么这个问题在当前数据下基本不会触发。它更像是代码鲁棒性风险，而不是当前主风险。后续若允许空机器或更极端初始化，再考虑改成显式 pair key。

第六类问题是非零 setup 下的 Cmax 快速过滤仍要谨慎。`PathInsertOperator.mergeCmaxValidation()` 里还保留了“m1 机器不需要验证，只会变得更小”的假设；在 setup 非零且不保证三角不等式时，这句话不严格。虽然后续函数拼接大概率还能把不可行候选评成上界或不可行，但这个剪枝/过滤注释本身不能作为正确性依据。若后续正式使用非零随机 setup，建议把这部分改成显式重算 m1 的 duration 或至少打开校验。

因此，当前判断可以写成：五类邻域的数学形态和单候选评价框架基本是对的，`IOptOperator` 和 `TwoOptOperator` 作为单机器算子风险相对低；真正需要优先修的是跨机器算子的 record-best 保存、no-improvement 缓存、double 排序、`PathInsert` 反向枚举笔误，以及 `Set.of` 对空/重复序列的稳定性问题。修完这些以后，再结合 `Move.testSequence(...)` 这种直接重算方式做一轮小规模全候选对拍，才能更有把握说当前局部搜索实现是正确的。

2026-05-15 已先修正跨机器算子的候选保存语义。`TwoOptStarOperator`、`PathInsertOperator` 和 `CrossExchangeOperator` 原来只在候选刷新全局 `bestCost` 时写入 `savedOperations`，这和后续“排序后贪婪提交互不冲突 move”的设计不一致，也会让 `noImprovedSeqPair` 误把有改进但没刷新全局 best 的机器对记成无改进。现在这三处都改为只要 `deltaCost < 0` 就写入 `savedOperations`。这样 `savedOperations` 就是完整的改进候选池，`commit()` 仍负责排序和互斥机器过滤，`noImprovedSeqPair` 的含义也恢复为“该机器对确实没有任何改进候选”。这次没有改变邻域枚举范围，也没有改变 commit 的贪婪批处理策略。

进一步检查后，除 setup time 导致的 Cmax 剪枝问题外，当前还剩两个相对明确的实现风险。第一，`PathInsertOperator` 中 `if(len==1) break` 仍然会漏掉两个任务片段的反向插入候选，合理写法大概率应是只在 `len==0` 时跳过反向重复。第二，跨机器算子在 `commit()` 末尾更新 `noImprovedSeqPair` 时，使用的是已经提交部分 move 之后的当前序列；如果某台机器刚被另一个 move 改过，那么它和第三台机器组成的新 pair 实际没有在本轮搜索过，却可能被写入 no-improved 缓存。后者的修正方向不是改变候选池，而是在写 no-improved 缓存时跳过所有已被本轮 commit 改动过的机器，只缓存“两台机器都没变、且本轮确实没有改进候选”的 pair。

剩余低优先级问题主要是代码鲁棒性，不是当前主正确性风险。`OptCost.compareTo()` 仍用 double 差值强转 int，在当前整数数据下影响很小，但后续若引入小数权重或 reduced cost，应改成 `Double.compare(...)`。`TwoOptStarOperator` 和 `CrossExchangeOperator` 的 `Set.of(seq1, seq2)` 在当前每台机器都有任务、序列不相同的前提下基本不会触发异常，后续若允许空机器再改成显式 pair key。`EngineVND` 的固定随机顺序和跨机器 move 的批量提交属于搜索策略选择，不是实现错误。

同日继续修正 `noImprovedSeqPair` 的写入时机。经过讨论后，采用更直接的做法：在跨机器算子 `commit()` 中，先根据 `savedOperations` 标记 `improvedPair`，随后立刻用当前未修改的旧序列写入 no-improved 缓存，最后才执行排序后的互斥 move。这样缓存记录的就是本轮 `searchBest()` 实际搜索过的序列 pair，不会把批量 commit 后新产生、但本轮没搜索过的 pair 错记为无改进。该调整已应用到 `TwoOptStarOperator`、`PathInsertOperator` 和 `CrossExchangeOperator`，不改变候选枚举和贪婪提交策略。
