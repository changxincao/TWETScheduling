# 2026-06-04 RMIH 上界启发式与重复覆盖修复记录

## 1. 问题背景

当前 `RestrictedMasterIntegerHeuristic` 的作用是利用当前 RMP 已生成的列池，求一个整数化的 restricted master，用来刷新 BPC incumbent。主 LP 的覆盖行使用 set covering 语义 `>=1`，这对列生成对偶是合理的；但如果整数启发式也直接用 `>=1` 更新 incumbent，就可能选出多个内部列重复覆盖同一个 job 的解。这样的解满足 restricted set covering MIP，但不满足原问题和 `BPCSolutionValidator` 的 partition 语义。

010 样例 `tmp-wet030_from040_010_2m` 正好暴露了这个问题。历史 BPC 运行中，root RMIH 得到 `17599` 并更新 incumbent，但最终 `valid=false`。后续诊断确认，该解由两条内部列 `[3321,4523]` 组成，重复覆盖 job `[19,29]`，因此不是合法原问题解。

需要注意的是，这个非法 covering 解不是完全没有信息。由于 setup time/cost 满足三角不等式，若某些 job 在多个列之间重复出现，理论上可以从部分列中删掉这些重复 job，剩余序列成本不应变差；但当前 restricted pool 是有限列池，这些删点后的列很可能还不存在。于是 `>=` MIP 可以找到一个“有潜力但还不合法”的低成本结构，而 `==` MIP 只能使用现有列，可能不可行或只能得到质量较差的合法组合。

## 2. 当前 010 root 测试事实

本轮诊断使用 `HEU.RestrictedMasterIntegerFilterDiagnostic`。它先复现 010 root LP/pricing，得到 `root_lp=16148.8`，root restricted columns 为 `5341`，LP 正值列为 `8`。随后按 reduced cost 从小到大筛列，并额外保证每个 job 保留覆盖它的前 `K=10` 条低 reduced-cost 列。正式诊断模型恢复原 RMIH 口径，包含内部列变量、外包 `y_j`、tariff segment active、baseline 变量及 tariff 约束。

`>=` covering 结果如下：

| 列筛选 | 状态 | 目标值 | 求解时间 | 首个 incumbent | 重复覆盖 |
| --- | --- | ---: | ---: | ---: | --- |
| `rc500+jobK10` | infeasible | - | `0.141s` | - | - |
| `rc1000+jobK10` | optimal | `19025` | `1.038s` | `0.684s` | `[7,29]` |
| `rc2000+jobK10` | optimal | `17599` | `2.505s` | `0.734s` | `[19,29]` |
| `rc4000+jobK10` | optimal | `17599` | `9.727s` | `1.836s` | `[19,29]` |
| `rc5341+jobK10` | optimal | `17599` | `10.005s` | `0.475s` | `[19,29]` |

因此，`>=` 下列太少时可能不可行；列数到 `1000` 后能找到 covering 解；到 `2000` 后已经找到历史 invalid incumbent `17599`；继续增加到 `4000/5341` 没有改善目标，但求解证明时间明显增加。这个结果说明 RMIH 作为辅助上界启发式时，不应无脑使用全部列求 `>=` MIP，否则可能比 pricing 还耗时。

`==` partition 结果如下：

| --- | --- | ---: | ---: | --- |
| `rc500+jobK10` | infeasible | - | `0.025s` | - |
| `rc1000+jobK10` | infeasible | - | `0.098s` | - |

`==` 的整数 MIP 在有可行组合时非常快，5000 多列仍在 1 秒内。但它的问题是列池少时容易不可行；列池足够时虽然能合法，但未必上界质量好。010 root 中，`==` 全量得到 `17723`，反而比 `>=` 结构经删点修复后的结果差。

本轮还测试了 `>=` 解后的删重复 job 修复。做法是：先求 `>=` MIP，读取选中列和重复 job；对每条包含重复 job 的选中列，枚举删除这些重复 job 的非空子集，生成新序列；用 `TWETColumnEvaluator` 重新计算真实列成本，加入列池；然后在“原筛选列 + 修复列”上求 `==` MIP。结果如下：

| 基准列筛选 | 新增修复列 | `repair==` 状态 | 目标值 | 求解时间 | 重复覆盖 |
| --- | ---: | --- | ---: | ---: | --- |
| `rc500+jobK10` | `0` | infeasible | - | `0.023s` | - |
| `rc1000+jobK10` | `6` | optimal | `16412` | `0.066s` | 无 |
| `rc2000+jobK10` | `6` | optimal | `16444` | `0.202s` | 无 |
| `rc4000+jobK10` | `6` | optimal | `16444` | `0.508s` | 无 |
| `rc5341+jobK10` | `6` | optimal | `16444` | `0.621s` | 无 |

这里出现了一个重要现象：`rc1000+repair` 的合法上界 `16412` 反而优于 `rc2000/full+repair` 的 `16444`。这说明列数和修复后上界质量不是单调关系；不同筛选列集会让 `>=` MIP 找到不同 covering 结构，删点修复后的结果也不同。RMIH 是启发式上界工具，重点不是把 restricted MIP 做到“全列最大规模”，而是在较小成本内找到能转成合法 partition 的好结构。

历史正式 BPC 日志中的 RMIH 时间也应分开理解。010 baseline-off 的两次 RMIH 分别为 node1 `23.950s`、node2 `11.885s`，合计 `35.835s`；formal-on 为 node1 `19.577s`、node2 `19.675s`，合计 `39.251s`。这不是单次 root full MIP 时间，而是两个节点各跑一次主流程 RMIH 的总时间。独立 formal diagnostic 中 root full `>=` 为 `10.005s`，它用于固定列池和覆盖语义诊断，不能直接等同主流程计时，但趋势一致：`>=` 的求解和证明会随列数变大显著变慢。

## 4. 对上述分析的补充判断

关于删点列的成本计算，当前继续使用 `TWETColumnEvaluator.evaluate(sequence)` 是合理的。删除一个 job 后，确实可以尝试复用列内部的二维函数拼接信息做增量评估；但这要求为当前列预先构建并保存前后缀函数数组，代码复杂度和内存开销都更高。RMIH 修复是低频启发式动作，当前每次只生成少量删点列，直接重算列成本更稳，也更不容易引入与 pricing 口径不一致的问题。

关于 `>=` 解是否“能当全局上界”，需要严格区分两层语义。`>=` 解本身不是原问题可行解，不能直接作为正式 incumbent 输出，也不能让 validator 通过；但它对应的删点修复结构通常可以构造出原问题可行解，并且修复后的目标值是合法上界。若没有完成修复或 validator 校验，直接把 `>=` 目标写成 BPC incumbent 会污染输出可行性；它可以作为启发式中间上界候选或内部引导，但最终更新全局 UB 应以修复后的合法解为准。

关于“如何从当前选中的重复列得到不含重复 job 的最优解”，如果只允许对当前 `>=` 解选中的列做删 job 操作，并且不允许更换列、不允许新增其他原池列，那么这个问题可以精确建成一个很小的修复 MIP。设重复 job 集合为 `D`，每条当前选中列 `r` 中包含一部分重复 job `D_r`。对每条列枚举它最终保留的重复 job 子集 `S_r ⊆ D_r`，该列删掉 `D_r \ S_r` 后重新计算成本。然后建约束：每条列选择一个 `S_r`，每个重复 job 在所有包含它的列中恰好被保留一次。目标是最小化这些修复列成本之和。这个小模型可以给出“固定当前选中列，只允许删点修复”的最优合法修复。

上面这个小模型比单纯贪婪更准，但它仍然不是全局最优上界，因为全局上界还可以选择当前 pool 里的其他列，或者选择修复列和其他候选列混合。当前诊断实际采用的是更宽一点的方式：先从 `>=` 选中列生成修复列，再把修复列加入候选池，然后解一次 `==` MIP。这允许 MIP 在原筛选列和修复列之间重新组合，因此比固定两条列删点更灵活。后续正式实现可以先沿用这个方向。

关于“能否不重建模型，直接修改原模型覆盖约束”，从 CPLEX 模型接口上是可行的：如果 RMIH 构造时保存每个 job 的 `IloRange coverRange[j]`，`>=1` 本质是下界为 1、上界为正无穷的 range；复解 `==1` 时可以把这些行的上界设为 `1.0`，再 `cplex.solve()`。修复列也可以通过 column-wise 建变量，或直接给现有模型添加新二进制变量并对 objective、coverage、branch、machine、tariff 相关行补系数。这样可以避免完全重建模型，但实现上需要把当前 `RestrictedMasterIntegerHeuristic` 的临时局部变量改成可复用模型对象。考虑到这是启发式模块，第一版可以先重建模型保证简单正确；若后续确认重建开销明显，再改成复用模型和修改约束上界。

## 5. 当前建议方案

当前不建议只采用直接 `==`。它的优点是快、合法性干净；但在列池少时不可行，在列池多时也可能因为缺少删点列而上界质量一般。010 root 的 `== full` 是 `17723`，明显差于 `>=` 修复后的 `16412/16444`。

更建议的主策略是 `>= + 一次修复 + ==复解`。具体流程如下：

1. 从当前 LP 中选择 RMIH 候选列。保留 LP 正值列、已有 incumbent 列、reduced cost 最小的前 `N` 条列，并对每个 job 保留覆盖它的前 `K` 条低 reduced-cost 列。当前可从 `K=10`、`N=1000/2000` 起测；010 root 上 `rc1000+jobK10+repair` 已经有效，`rc500` 太少。
2. 在筛选列上解正式 `>=` RMIH MIP，先求到 optimal。暂不加 time limit，继续观察耗时主要在找可行解还是证明下界。
3. 读取 `>=` 解。如果没有重复覆盖，并通过 validator，则可直接作为合法 incumbent。若存在重复覆盖，则进入删点修复。
4. 对每条包含重复 job 的选中列，若该列中重复 job 数量不超过阈值，例如 `5`，则枚举删除这些重复 job 的所有非空子集，生成修复列并重算成本。
5. 若某条列中重复 job 数量超过阈值，则暂不做全枚举。可以采用“保留删除成本损失最大的那一侧，或等价地对每个重复 job 保留删除后成本减少量最小的那条列”的迭代规则，逐步构造修复列。这个规则比简单随机/固定顺序删除更合理，但仍是启发式。
6. 把生成的修复列加入候选列集合，在同一节点分支约束和 tariff 约束下解正式 `==` RMIH MIP。第一版可以重建模型，后续再考虑复用 `IloRange` 修改覆盖行上界。
7. 只有 `==` 复解通过 validator 后，才更新正式 incumbent。

如果想在“固定当前 `>=` 选中列”这个较窄范围内做到最优修复，可以实现第 4 节提到的小型修复 MIP：每条选中列选一个保留重复 job 子集，每个重复 job 恰好保留一次，目标最小化修复后列成本。这个能精确回答“当前重复列怎么删最优”，但不能替代最后的 `==` restricted MIP，因为后者允许原候选列和修复列重新组合，启发式空间更大。

## 6. 当前结论

010 root 的证据支持以下结论。第一，直接 `>=` RMIH 会产生重复覆盖的非法 incumbent，不能直接更新正式解。第二，`>=` 不是无用的：它能找到低成本 covering 结构，删点修复后能转成更强合法上界。第三，`>=` MIP 随列数增加明显变慢，作为辅助启发式应控制候选列规模。第四，直接 `==` 快但依赖列池完整性，列少时不可行，列多时也可能因为缺失删点列而上界质量一般。第五，下一步最值得实现和测试的是 `>= + 删点修复列 + ==复解 + validator`，而不是单独押注 `>=` 或 `==`。
