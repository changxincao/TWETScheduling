# setup 三角闭包修正记录

本次问题来自同名 30 任务临时算例的 `SETUP` 矩阵仍然不满足完整有向三角不等式。之前的 `tmp-triangle-20260611` 数据虽然已经降低了一部分 setup 值，但检查时仍能找到经虚拟点 0 中转的反例，例如 `s[i][j] > s[i][0] + s[0][j]`。由于当前 pricing、completion bound 和若干启发式判断会默认使用 setup 三角不等式，这类残留违例会让历史结果和当前结果混在一起时很难判断问题来源。

本次修正采用完整有向 Floyd 闭包，节点集合包含虚拟起点 0 和所有真实任务。这样保证任意 `i,j,k` 都满足 `s[i][j] <= s[i][k] + s[k][j]`。这个处理会显著降低部分 job-to-job setup，尤其是当 `s[i][0]=0` 且 `s[0][j]` 较小时，目标值口径会随之改变。因此后续比较实验时必须同时记录数据文件路径或 hash，不能只按同名 `.dat` 判断是不是同一个算例。

代码层面，`src/Common/ETConverter.java` 在随机生成 setup 后立即调用全节点 Floyd 闭包，保证后续默认生成的数据满足该性质。数据层面，已对 `test-results/bpc` 下当前用于 30 任务测试的所有 `tmp-wet030_from040_*_2m.dat` 临时输入做同样闭包，包括根目录临时输入、`tmp-triangle-20260611` 下的 010/011，以及几个 batch 目录中的 004-008 备份输入。

验证结果为：共检查 25 个匹配的临时输入文件，闭包前每个文件都有三角违例，典型违例数在 2000 以上；闭包后统一得到 `violations=0`、`maxGap=0`。`ETConverter` 已用 `javac -encoding UTF-8 -cp target/classes -d target/classes src/Common/ETConverter.java` 编译通过。

正式 `data/` 目录随后也已按同一口径批量处理。共检查 750 个带 `SETUP` 的 `.dat` 文件，闭包前 750 个文件全部存在三角违例，最大违例 gap 为 62；闭包写回后验证 `afterBad=0`、`maxAfter=0`。这意味着当前默认数据集和临时 30 任务测试输入在 setup 三角不等式口径上已经统一。由于这会改变大量 setup 值，后续若引用旧实验结果，必须注明旧结果对应的是闭包前还是闭包后的数据。
