# 求解结果输出与验证设计

当前代码已经有求解器和若干调试输出，但没有形成统一的结果落盘工具。模型侧现在能导出 `model.lp`、打印目标值，`ATIParallel` 还能在控制台打印一份调度；BPC 侧则基本只有内存中的 `TWETSolveResult` 和若干骨架对象，没有稳定的结果文件。这样的问题是，后面一旦开始批量跑算例，求解时间、规模、上下界、incumbent 列等信息很容易散在控制台里，既不方便复查，也不方便校验目标值是否真的对。

这次的处理思路分成两条线。第一条是基于数学模型的输出工具，覆盖 `ArcFlowModel`、`TimeIndexModel` 和 `ATIParallel`。这条线统一抽成“求解摘要 + 调度提取 + 解验证 + 文件落盘”四步。摘要里保留方法名、算例名、`n/m/horizon`、求解状态、求解时间、目标值、最好界、相对 gap、变量统计和约束数量。调度提取则尽量从模型变量中重构出结构化的机器时间表，其中 `ArcFlowModel` 和 `ATIParallel` 直接沿着弧走，`TimeIndexModel` 因为没有显式机器索引，就按开始时间做一次贪心机器分配，只用于导出和验证。验证器则单独重算每个任务的完成时间成本，检查任务是否唯一覆盖、机器内时序和 setup 是否被破坏，以及目标值是否与重算结果一致。结果文件统一写到指定根目录下，以方法名建子目录，以算例名为文件名。

第二条是基于 `TWETBPC` 骨架的输出工具。这里当前真正稳定拿得到的是 `TWETSolveResult`、全局 `Pool`、`CutPool` 和 `TWETBPCContext`，所以先把最终结果导出和 incumbent 校验做实，把过程级 trace 单独留接口，不强行接到现有流程里。最终输出里保留状态、incumbent、best bound、已处理节点数、列池和 cut 池大小、最终 incumbent 列集合，并把这些列的序列、成本、来源单独导成一个 `columns.csv`。校验部分先做 incumbent 列是否覆盖全部任务、是否超出机器数、列成本和是否等于结果里的 incumbent 值。至于 `column/cut/node/pricing` 的更细过程信息，这次额外立了一个 `BPCTraceSink` 接口和一个轻量 `BPCTraceSummary` 载体，后面如果开始把 price-and-cut 真正接上，再把这些计数逐步挂进去，不用重新改结果导出层。
