# 仓库约束

## 文档写作

- 本仓库中的说明文档、研究记录、阶段总结、数据说明，默认使用正常技术中文写作，不要写成营销文，也不要写成明显的 AI 模板文。
- 允许使用少量必要的小标题帮助阅读，但不要过度分节，不要把一页内容拆成大量碎段。
- 默认采用段落式展开，不要过多使用 bullet、子 bullet 或机械罗列。只有内容天然适合列表时，才使用列表。
- 语言要直接、清楚，像正常人在认真记录和解释问题。少用空话、套话和过度铺垫。
- 默认保持简洁。一个段落能讲清的内容，不要拆成很多段重复表达。
- 对研究笔记、数据说明、实验记录，优先写清楚来源、证据、可获取性、结果幅度、影响判断和当前结论，减少重复解释。

## 实验与研究记录风格

- 记录实验或研究结论时，优先沿用用户当前的风格：先写主要结论，再写具体观察、尝试过的调整、结果大小，以及由此得到的判断。
- 保留具体数字、相对提升、gap、失败尝试和负面结果，不要把失败过程抹平。
- 当内容包含多个实验分支、调整方向或待确认问题时，优先使用 `1 / 1.1 / 1.2` 这种编号方式组织，而不是堆很多 bullet。
- 允许使用“由此”“因此”“当前主要结论为”等自然连接词，把观察和结论直接连起来。
- 如果用户已经把文档改得更紧、更短，就保持这种压缩后的风格，不要再自动扩写回长篇说明文，除非用户明确要求展开。

## 讨论沉淀

- 后续每一部分有实质内容的讨论，都要视情况沉淀成记录文档，而不是只停留在对话里。
- 记录文档默认放在 `docs/` 下的合适子目录；如果用户已经指定目录，就按用户指定的位置放。
- 记录时优先采用“问题 + 思路 + 方案 + 结论/待确认事项”这种结构。根据内容需要，也可以补“背景”“实验结果”“风险”“后续计划”等部分。
- 如果某次讨论还没有形成明确方案，也要把未决问题、当前判断和下一步打算记下来，避免上下文丢失。
- 新写记录文档时，要尽量让后续回看的人直接明白：讨论的是什么问题，为什么会有这个问题，考虑过哪些路，最后决定怎么做，哪些事情还没定。
- 每次完成实质性代码修改、测试补充或重要分析后，要在对应主题的记录文档或日志文件中补一段简要记录，写清楚本次做了什么、为什么做、验证结果如何、还剩什么风险。已有专题文档时优先追加到原文档；没有合适文档时，再在 `docs/` 下新建记录。

## 代码注释

- 新写代码或修改代码时，默认给类、核心数据对象、关键方法补充简洁中文注释，优先说明“这个东西是干什么的”，而不是逐行翻译代码。
- 注释不要过密，不要把显而易见的赋值、循环、 getter/setter 全写一遍。
- 涉及设计选择、兼容处理、历史包袱、性能取舍或临时约束时，允许在注释里加日期前缀，格式类似 `2026-04-10:`，方便后续追溯。
- 日期注释只用于关键位置，不要每个方法、每行逻辑都加时间戳。
- 修改已有核心算法时，如果改动是为了处理边界情形、历史 TODO、数值语义或性能取舍，要在对应代码位置用中文写清楚“为什么改”和“原来假设是什么”。例如分段线性函数里涉及 horizontal/vertical gap、函数不连续、前向/后向函数方向差异时，要说明连续情形下原逻辑为什么成立，以及新逻辑为什么选择先保证稳健性、后续再考虑加速。

## Git 提交

- 每次完成实质性代码修改后，默认要提交一次 git commit。提交前先运行必要测试或至少做语法/编译检查；如果因为时间、依赖或用户要求不能测试，要在最终回复里说明。
- 提交信息要简洁说明本次修改的核心目的，不要把无关文件混进同一个提交；如果工作区已有用户或 IDE 产生的无关变更，不要擅自回滚。
