同日补充了启发式 forward 函数初始化处的注释：序列首任务最早完工时间应为 `s[0][job] + p[job]`，其中 `s[0][job]` 表示从虚拟起点 0 到任务 `job` 的首任务 setup，`p[job]` 是加工时间。因此初始化 `setDomain(...)` 时裁掉更早时间点，是为了避免选择物理不可行的 completion。

同日补充了 `tail.end` 与 `domainEnd` 关系的专题记录。准确结论是：如果输入函数都满足实际右端和元数据右端同为 `T`，则 `add`、有效输入下的 `mergeMinimum/mergeMinimum2`、当前版本的 `normalize` 和正向 `shiftX` 不会主动破坏这个关系；真正会主动制造 `tail.end != domainEnd` 的，是 `setDomain(start,end)` 且 `end < 原 domainEnd`，因为它会裁短真实链表右端后再恢复原元数据定义域。

同日进一步明确了该 debug 判断的意义：它不是为了检查 `domainEnd` 字段是否变化，而是为了识别函数真实 segment 链表右端是否已经不再到全局 `T`。在 forward pricing 语义下，若 `tail.end != domainEnd`，说明实际定义域被裁短，后续仍按 `[a,T]` 做 `add/merge/dominance` 就可能出错，因此该检查能够定位这类右端定义域风险。

同日更正了对 `updateDominatedIntervals` 和 partial dominance 的判断。当前“被支配区间打成 `big_M`，再允许 `normalize()/minimizePrefixInPlace()` 做 forward 闭包”的做法，在 forward frontier 语义下可以解释通，不应再作为第一阶段主要问题反复分析。真正需要保证的是 pricing 前把 `Utility.curUpperBound` 重置为 `Utility.big_M`，避免启发式上界截断 reduced cost 函数；同时在 dominance 判定层严格控制哪些 label 有资格支配。

同日分析了 `minimizeSuffixInPlace()` 和 backward 函数定义域。结论是 backward 构造阶段基本不提前裁窄 domain，末任务直接复制原惩罚函数，递推时用负向 `shiftX`、`add` 和 `minimizeSuffixInPlace()`；当前主要风险在 suffix 最小化实现会漏掉应保留的尾部常数最小值段，而不是 backward 定义域设置本身。详细分析已补充到 `docs/plans/2026-05-13-PiecewiseLinearFunction测试分析记录.md`。

同日继续分析了 `minimizeSuffixInPlace()` 中边界判断的原始动机。该判断应是从 prefix 最小化迁移过来的保护逻辑，用来避免在函数边界插入由 `Utility.curUpperBound` 形成的人工上界段；但 suffix 版本里右侧最小值可能来自真实尾部函数，直接套用 `lastT != tail.end` 会误跳过真实尾部常数最小值段。该判断的动机和问题已补充到分段函数专题文档。

同日补充了基于问题结构的 prefix/suffix 风险判断：如果不发生 merge 和 partial dominance，单 job 凸成本在 shift、相加和 prefix/suffix minimum 后基本保持可控结构，边界误删通常不会触发；但 merge 的下包络和 partial dominance 的 `big_M` 区间会破坏普通凸性，仅靠“minimize 后单调”不足以证明下一轮扩展完全安全。因此后续若接入 BPC 中的 merge 或 partial dominance，仍建议把边界补段判断改为显式检查 `runningMin` 是否仍为 `Utility.curUpperBound`。

同日修改了 `minimizePrefixInPlace()` 和 `minimizeSuffixInPlace()` 的边界补段判断。旧逻辑用 `prevT != head.start` 或 `lastT != tail.end` 来避免补入 `curUpperBound` 人工上界段，但这种写法会把贴边界的真实最小值段也误跳过。新逻辑改为只在 `runningMin == Utility.curUpperBound` 时跳过；一旦 `runningMin` 已经来自真实函数值，即使贴着左/右边界也会补回常数最小值段。该问题在纯凸扩展链条下不常触发，但经过 merge、partial dominance 或裁剪后，再叠加 job 的凸成本函数时，累计函数开头/结尾可能正好落在 due window 外侧递增或递减部分，因此仍有可能出现。详细条件和修正理由已写入 PWLF 专题文档。

验证方面，本机没有 `mvn` 命令，因此使用 `javac -encoding UTF-8 -cp target/classes -d target/classes` 编译 `PiecewiseLinearFunction.java` 和 `PiecewiseLinearFunctionPropertyTest.java`，随后运行 PWLF 专项测试。`merge-find-contract` 模式为 `passed=5, warnings=0, failed=0`；完整属性测试为 `passed=13, warnings=1, failed=3`，剩余失败仍集中在 `dominates` 定义域覆盖和 `mergeMinimum` 完全不相交输入，不是本次 prefix/suffix 修正引入的新问题。

## 2026-05-15

补回了 `minimizePrefixInPlace()` 和 `minimizeSuffixInPlace()` 附近被上一轮修改删掉的 2025.5.10 历史注释。代码逻辑没有变化，只是在新注释旁保留原来的设计背景，避免后续追溯时丢失当时为什么要用上界段做截断的说明。

继续补充了两个 PWLF 边界回归测试，分别覆盖 prefix 左边界真实最小值延伸和 suffix 右边界真实最小值延伸。测试验证当前边界补段修改不是只修复了随机样例，而是直接覆盖旧逻辑中 `prevT == head.start` / `lastT == tail.end` 误跳过真实最小值段的场景。重新编译相关类并运行测试后，`merge-find-contract` 为 `passed=5, warnings=0, failed=0`，完整属性测试为 `passed=15, warnings=1, failed=3`，剩余失败仍是既有的 `dominates` 和无效 `mergeMinimum` 输入问题。

补充分析了 PWLF 单点段处理。当前 `add()`、prefix/suffix 最小化兜底和 `trimToDomain()` 右端裁剪可以处理部分单点退化，但这不是通用零长度 segment 支持；`evaluate()` 只额外处理 `tail.end`，内部单点不可靠，`mergeMinimum` 也要求单点 label 在 pricing 层提前收尾。因此后续仍应把单点限制在全局右端 `T` 附近作为退化兜底，避免多个内部单点进入主函数操作流程。

补充确认了 `normalize()` 和普通 label 扩展的关系：普通扩展使用 `shiftX + add + prefix/suffix minimize`，不会自动调用 `normalize()`；`normalize()` 主要在 dominance 和 merge 后使用。若扩展过程中得到 `[T,T]`，当前 add 和 prefix/suffix 兜底不会立即出错，但该 label 应在调用层直接收尾或丢弃，避免进入集合操作。另记录了 `add()` 的定义域行为：它只取两个输入实际定义域交集，若一个输入右端短于 `T`，输出也会短于 `T`，不会自动补 `big_M` 到全局右端。

重新确认了当前 `setDomain(...)` 的实际调用点。会主动把函数真实右端裁短、从而制造 `tail.end < domainEnd` 的主要是 `Move.getCost()` 里的前向回推：它为了在已知后继 completion 的情况下反推当前任务 completion，会把当前 forward 函数临时裁到后继允许的最晚完工时间。首任务初始化处也调用 `setDomain`，但右端仍是 `CmaxH/T`，只裁左端，不属于这类右端定义域风险。

确认了启发式中原始 job 惩罚函数的定义域。`Data.setPenaltyFunctions()` 会为每个任务建立 `new PiecewiseLinearFunction(0, CmaxH)`，因此对象元数据定义域是 `[0, CmaxH]`；正常权重数据下，实际 segment 链也按 early、on-time、tardy 三段覆盖到 `CmaxH`。启发式构造 forward 首任务时才用 `setDomain(s[0][job]+p[job], CmaxH)` 裁掉物理不可行的左端，后续 `Move.getCost()` 的前向回推才会临时裁右端。

结合 `parallel_machine_scheduling_with_due_window.pdf` 分析了 BPC pricing 中 job 成本函数限制定义域的实现方式。文档里的 label 函数定义域是 `[ell_L,T]`，profitable completion window 只限制本次扩展的完成时间，不应把扩展后函数右端物理裁短。因此后续不建议直接用当前 `setDomain(h,\bar h)` 后再 `add`，也不建议让底层 `add()` 自动补右端；更稳的方向是做一个显式的窗口限制接口，把窗口外区间写成 `big_M` 段，同时保持右端到 `T`。专题分析已补入 `docs/plans/2026-05-13-PiecewiseLinearFunction测试分析记录.md`。

继续明确了该窗口限制接口的命名和语义。现有 `setDomain(start,end)` 已被启发式回推依赖，语义是物理裁剪 segment 链表，不宜直接改掉；pricing 应新增一个语义明确的方法，例如 `restrictWindowWithBigM(...)`，把窗口外区间写成 `big_M` 段而不是删除，并保持真实右端到 `T`。这样可以避免把普通定义域裁剪和 pricing 窗口不可行区间表达混在一起。

按最终讨论实现了 `setDomain(start,end,fillOutsideWithBigM)` 重载。`false` 分支复用旧的物理裁剪逻辑；`true` 分支把窗口外区间写成 `Utility.big_M` 段，同时保持真实右端到原来的 `T`，供后续 BPC pricing 的 profitable window 使用。同步新增 PWLF 测试，验证窗口外为 `big_M`、窗口内保留原值、`tail.end/domainEnd` 不被裁短。编译通过后运行测试：`merge-find-contract` 为 `passed=5, warnings=0, failed=0`，完整测试为 `passed=16, warnings=1, failed=3`，剩余失败仍是既有风险。

补充说明了 `evaluate()` 与窗口右端的边界语义。当前 segment 按 `[start,end)` 处理，只有整条函数的 `tail.end` 被特判，因此 `setDomain(..., true)` 在窗口右端后接 `big_M` 段时，裸 `evaluate(end)` 会命中右侧 `big_M`。这不是 `evaluate()` 普遍错误，而是内部断点归属问题；pricing 中随后做 prefix-min 时会用左侧 `seg.end` 参与计算，通常能保留窗口右端的可行最优值。

将 BPC pricing 中最容易出错的 PWLF 使用约束集中标记到专题文档：两参数 `setDomain` 和三参数 `setDomain(...,true)` 语义不同；`add()` 不自动补右端；窗口右端裸 `evaluate` 有内部断点归属风险；pricing 前必须把 `curUpperBound` 设回 `Utility.big_M`；`[T,T]` 单点 label 应直接收尾或丢弃。后续实现 pricing 时应优先按这些约束检查。

补充分析了 `[a,T] + big_M 连续 gap + 无内部单点` 假设下 PWLF 操作的可用边界。结论是大部分底层操作可以复用，包括复制、平移、相加、prefix/suffix 取小、有效输入下的 merge 和基于 big_M 的 dominated interval 更新；但 `dominates` 仍不能直接作为严格 BPC label dominance，`mergeMinimum/mergeMinimum2` 也不能当通用下包络工具，内部断点的 `evaluate` 端点归属仍需在 pricing 层避免误用。

检查了 `PiecewiseLinearFunction.evaluate()` 的当前调用位置。普通单机成本 `calCost(m)` 和 completion 回推主要依赖 forward 函数右端值或 `findMinimal()`，不是靠 `evaluate()` 搜索最优；`evaluate()` 在线上主要用于三段拼接中对 `findMinimal()` 得到的拼接时间点回代求值，以及 `dominates()` 的右端辅助比较。因此它不是 label 扩展主流程入口，但若回代时间点落在内部断点，仍要注意 `[start,end)` 的端点归属。

确认了 `mergeMinimum2()` 的使用情况。当前正式代码没有调用它，只在 `src/Test/MergeBenchmark.java` 中作为测试/对照使用；该实现会直接修改输入 segment 的 start 并复用原节点，副作用比 `mergeMinimum()` 更难控制，公共区间边界处理也更粗。因此后续 BPC pricing 若需要下包络合并，优先使用已梳理输入契约的 `mergeMinimum()`，`mergeMinimum2()` 暂作为实验实现保留。

继续确认了剩余风险边界。当前 `dominates()` 也没有被正式求解流程调用，只在测试/demo 中出现；它不是启发式当前风险，但如果后续接 BPC label dominance，不能直接复用。现阶段 PWLF 底层操作基本已按 `[a,T] + big_M gap + 无内部单点` 契约梳理完，剩余主要工作是重新设计 label 层 dominance，而不是继续改普通函数操作。

进一步明确了 `dominates()` 和 partial dominance 的关系。在只看函数层、不要求严格改进、不考虑 label 状态时，`dominates()` 仍只是完整函数占优的 true/false 判断，不支持 partial dominance；部分占优由 `updateDominatedIntervals()` 负责，把公共定义域中被支配区间写成 `big_M`。在 `[a,T]` 右端统一契约下，`dominates()` 的右端覆盖问题被弱化，但契约外的右端短缺分支仍不是严格证明。

收紧了 `PiecewiseLinearFunction.dominates()` 的完整占优语义：右端不覆盖直接返回 `false`，不再用两个端点近似推断缺失区间；双指针扫描结束后增加 `cur >= gEnd` 兜底检查，防止未扫完整个 `g` 定义域时误判为占优。该方法仍只做函数层完整占优，不支持 partial dominance。重新编译并运行 PWLF 测试后，`merge-find-contract` 为 `passed=5, warnings=0, failed=0`，完整测试为 `passed=17, warnings=1, failed=2`，剩余失败仍是 `mergeMinimum` 无效输入。

检查了全仓库直接用 `< / <= / > / >= / == / !=` 比较 double 的位置，并整理为 `docs/plans/2026-05-15-double比较检查记录.md`。结论是 `PiecewiseLinearFunction` 主体大多已经使用 `Utility.compare...`，真正需要优先处理的是启发式里的成本/时间/bound 比较、`SchedulerForReleaseNoWait` 的 double 等号判断、少量 Data/TimeIndexModel 工具逻辑，以及 BPC/TWETBPC 框架里的 reduced cost、LP 值和分支阈值比较。本次只做检查和记录，尚未批量改代码。

同日补充分段线性函数方向语义记录。结论是 `normalize()` 明确只适配 forward label，因为它删除左侧 `big_M`、保留右侧 `big_M` 并调用 `minimizePrefixInPlace()`；因此 `updateDominatedIntervals()`、`mergeMinimum()`、`mergeMinimum2()` 这些内部调用 `normalize()` 的函数也只能按 forward 语义使用。`minimizeSuffixInPlace()` 是 backward 闭包，但当前没有对应的 backward normalize、backward merge 或 backward partial dominance 工具。

同日补充 backward pricing 的函数定义域原则：label 自身函数左端不提前裁剪，保持从 0 开始，避免负向 `shiftX` 把后段第一个任务的最早完成时间限制错误传播到前段最后一个任务。进一步明确为：forward/backward 对 job 成本函数的窗口处理差别不大，job 函数可以直接裁到窗口内；需要严格保持的是累计 label 函数的端点契约，即 forward 保持右端 `T`，backward 保持左端 0。

同日逐个检查 `PiecewiseLinearFunction` 操作函数的 backward 适用性。结论是底层函数操作大多方向无关；不适用于 backward 的主要是 `normalize()`，以及内部依赖 `normalize()` 的 `updateDominatedIntervals()`、`mergeMinimum()`、`mergeMinimum2()`。这些当前都是 forward 集合操作，后续若做 backward label 合并或 partial dominance，需要单独实现 backward 版本。

同日补充方向化接口取舍：`mergeMinimum2()` 不参与正式方向化改造，后续可注释或标记为实验函数；正式接口应集中在 `normalize(direction)`、`mergeMinimum(g,direction)` 和 `updateDominatedIntervals(g,direction)`。旧无方向接口不再作为主接口使用，pricing 中应显式传入 forward/backward 方向。

同日根据进一步讨论修正了上述风险优先级：`savedOperations` 只保留全局最优或 record-best 候选本身可以作为设计选择，真正风险是不能据此把未保存的机器对直接写入 `noImprovedSeqPair`；跨机器批量提交不是一定错误，而是需要明确它是互不重叠 move 的贪婪批处理，不是论文式单 move best-improvement；`compareTo()` 在当前整数数据下影响较小，可先补注释或后续顺手改；`Set.of(seq1, seq2)` 在当前每台机器都有任务且序列不可能相同的前提下不太会触发；比较确定需要关注的是 `PathInsertOperator` 的 `if(len==1) break` 会漏掉两个任务片段的反向插入候选，以及非零 setup 下 Cmax 剪枝仍未完善。

同日补充确认 ALNS 插入算子暂不修改的理由：当前 repair 允许较差甚至被上界截断的位置参与选择，可以理解为保留更大的扰动空间；在当前 `CmaxH` 较宽、删除比例受控、每台机器保持非空的设定下，暂不把它作为主正确性修复点。后续如果发现 repair 后频繁出现 `curUpperBound/big_M` 成本或不可行序列，再考虑改成“优先真实可行位置；全部不可行时再兜底并记录 warning”的软约束方式。另确认当前 ALNS+LS 的解更新策略：全局最优解保存在 `data.configure.bestSolution`，`EngineVND.commit()` 每次 LS move 成功后都会尝试更新全局最优；`EngineALNS.search()` 每轮 ALNS repair + VND 完成后再次比较并更新全局最优。ALNS 下一轮扰动基于 `current`，而不是直接基于全局最优；`current` 只有在 `tmpSol.curCost <= current.curCost` 时才更新为本轮局部搜索后的解。
