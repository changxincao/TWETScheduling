# 外包列化 SP1 设计分析

## 1. 论文模型与当前代码口径

下载目录下的 `parallel_machine_scheduling_with_due_window.pdf` 给了两种外包建模方式。按论文命名，`SP1` 是“内部机器序列列 + 外包子集列”，`SP2` 是“内部机器序列列 + master 中显式外包变量 `z_j` + tariff segment”。当前 `LP.java` 的实现更接近论文 `SP2`：master 中同时有内部列 `lambda`、外包变量 `y_j`、外包 tariff segment active/baseline 变量，并通过 `sum b_j y_j = sum q_l`、`sum z_l = 1` 表示整体外包成本。因此后续说的“把外包变量也转化为列”，对应论文中的 `SP1`，而不是继续沿用当前 `SP2`。

列化后的主问题有两类列。内部机器列 `s` 覆盖任务 `delta_js`，成本为该机器序列的最优内部加工成本。外包列 `o` 覆盖任务 `rho_jo`，每个外包列代表一个外包任务集合 `O(o)`，成本为 `G(sum_j b_j rho_jo)`。覆盖约束应写成等式：

`sum_s delta_js y_s + sum_o rho_jo w_o = 1`

机器数约束只作用于内部机器列：

`sum_s y_s <= m`

外包列也需要一个类似机器数量的总数约束，但它的语义是“最多选择一个总外包集合”：

`sum_o w_o <= 1`

这样就不需要显式空外包列；没有任务外包时，所有外包列变量都取 0。这个约束很关键，因为 `G(B(O))` 是对一个总外包集合计费。如果允许同时选多个外包列，就会把一个集合拆成多份分别按 `G` 计费，偏离论文模型。若未来想完全照论文 `sum_o w_o = 1` 写法，也可以加入空外包列；当前第一版采用 `<=1` 更简单。

## 2. 两类 pricing

列化外包后，每轮 LP dual 产生两类 pricing。

内部机器序列 pricing 保持当前主线逻辑。若覆盖行 dual 为 `pi_j`，机器数行 dual 为 `lambda`，内部列 reduced cost 为：

`c_s^in - sum_j pi_j delta_js - lambda`

分支 arc、SRI cut、completion bound、pricingOnly arc 等仍只影响内部机器列 pricing。现有 ng-DSSR、partial dominance、completion bound 和启发式 pricing 可以继续作为内部列定价器使用。

外包列 pricing 来自论文中的 aggregate-cost knapsack：

`max_{O subset J} sum_{j in O} pi_j - G(sum_{j in O} b_j)`

若 `mu` 是外包列总数约束 `sum_o w_o <= 1` 的 dual，则外包集合列的 reduced cost 为：

`G(B(O)) - sum_{j in O} pi_j - mu`

因此 pricing 先找使 `sum pi_j - G(B)` 最大的外包集合，再按 reduced cost 判断是否加入。论文建议按 job 编号构造 DAG：一条从 source 到 sink 的路径表示一个外包子集，label 为 `(lastIndex, accumulatedBaseline, accumulatedDualBenefit)`。同一节点上，若 `A1 <= A2` 且 `P1 >= P2`，则 label 1 支配 label 2；这是因为未来可选 jobs 相同，且 `G` 非降。`G` 的凹性用于表达折扣结构，支配规则本身只依赖非降性。

实现上建议新增 `OutsourcingPricingEngine`。若 `b_j` 是整数或可安全缩放成整数，可以按累计 baseline 做 DP；否则用论文的 label-setting DAG，并维护每个 node 上的非支配 `(A,P)` label 集。外包成本为 `big_M` 的 job 应直接从外包 pricing 候选中排除，因为当前项目语义是该 job 不允许外包。

## 3. 分支顺序与外包分支

machine arc 分支约束只作用于内部机器列，这是正确的。外包列没有内部加工顺序，也没有 `i -> j` 的 machine arc，所以 required/forbidden arc、pricingOnly arc fixing、completion-bound subtree arc elimination 都只应筛内部机器列和内部 pricing 图，不应筛外包列。

但为了 exact B&P，外包列变量本身也需要分支。定义当前解中的外包覆盖量：

`z_j = sum_o rho_jo w_o`

当内部 arc branch 找不到分数候选后，再检查是否存在分数 `z_j`。若存在，则生成两个子节点：

`z_j = 0`：job j 不能外包。已有外包列中包含 j 的列被过滤，外包 pricing 禁止选择 j。

`z_j = 1`：job j 必须外包。已有外包列中不包含 j 的非零外包列被过滤，外包 pricing 强制包含 j。因为覆盖约束是等式，内部机器列若包含 j 会自动与该分支冲突；正式 restricted set 过滤时也应直接删掉这些内部列，减少无意义列。

因此分支顺序建议为：先 internal arc branch，再 outsourcing membership branch。这样优先沿用当前机器序列分支树；只有机器 arc 信息已经不能继续切分 LP 解时，才处理外包侧分数。

`TariffSegmentBrancher` 在外包列化模式中直接禁用。此时 master 里没有 tariff segment active/baseline 变量，外包成本已经封装进外包集合列。第一版不做 segment 分支迁移，避免把 `SP1` 和当前 `SP2` 混在一起。

## 4. 子节点继承与 repair

外包列化后，节点需要维护两类 restricted pool：内部机器列和外包集合列。子节点从父节点继承列时：

内部机器列按现有逻辑处理。首次可先保留父节点列并通过新分支行判断可行性，repair 后再按 reduced cost 和分支兼容性筛选。正式 restricted set 中，违反当前 required/forbidden arc 的内部列必须过滤；若有 required-outsourced job，也应过滤包含该 job 的内部列。

外包列不受 machine arc 分支影响，应继承父节点正值列和低 reduced-cost 列；若有外包 membership 分支，则按 `z_j=0/1` 状态过滤。采用 `sum_o w_o <= 1` 后，节点初始不需要空外包列保证可行；但为了加速，应把启发式 incumbent 对应的外包集合列和父节点正值外包列传下来。

repair 阶段也需要两类 pricing。若不可行来自新增 required arc、machine lower/upper 等内部机器行，主要依赖内部 sequence pricing 修复；外包 pricing 无法满足 machine arc 分支行。若不可行来自覆盖列不足，外包 pricing 可以补外包集合列。PC 层不需要改变“一个定价器加列后立刻重解 LP 并从头开始”的节奏，只是 pricing result 要能返回两类列。

更具体地，子节点首次 RMP 应先继承父节点传下来的内部列和外包列，然后加入当前分支行直接求解。若可行，则按当前 LP reduced cost 筛选内部列和外包列，进入正常 price-and-cut。若不可行，则进入 repair 模式：只给当前新增的分支行加人工 slack，沿用当前代码的思想，不给覆盖行随便加 slack。repair pricing 顺序建议为内部启发式、外包 pricing、内部 exact pricing。每次某个 pricing engine 新增任一类列后，立即重解 repair LP；直到人工 slack 归零，再关闭 repair 模式，筛选正式 restricted sets。

外包 membership 分支的 repair 要分开看。`z_j=0` 子节点只会过滤外包列，通常不需要特殊 repair。`z_j=1` 子节点要求 job j 必须外包，因此必须保证外包 restricted set 中至少有一条包含 j 且满足其它外包 membership 状态的列；如果没有，repair 阶段应首先调用外包 pricing 的 `findFeasible`，生成一个满足 required/forbidden outsourced jobs 的外包集合列。由于覆盖等式会排斥内部列覆盖 j，内部列不负责修这个分支。

## 5. 当前代码需要新增的东西

第一版应通过配置保留当前 `SP2`，新增 `outsourcingMode=MASTER_VARIABLES/COLUMNS` 或类似开关。实现上不建议在现有 `LP` 里到处写 `if mode` 去“禁用”变量，而应把主问题结构模块化，最好新建一个列化外包主问题类，例如 `ColumnizedOutsourcingLP`，或至少抽出共同基类/接口。原因是两套 RMP 的变量、dual、solution 读取、repair 行和列添加方式都不同，硬塞在一个类里会让后续维护非常容易出错。当前 `LP` 保持现有 `SP2(y_j + tariff segment)` 语义不变；新类专门负责 `SP1(内部列 + 外包列)`。

需要新增外包列对象和外包列池，例如 `TWETOutsourcingColumn` 与 `OutsourcingPool`。外包列字段至少包括 id、jobs bitset、baseline、cost、signature。不能把外包集合硬塞进 `TWETColumn`，否则机器数约束、arc 分支、SRI cut 和列成本解释都会混乱。

`Node` 需要新增外包 membership 分支状态，例如 `outsourcingJobState[j] = FREE/FORBIDDEN/REQUIRED`。该状态只用于外包列过滤和外包 pricing；内部机器 pricing 不需要读取它。由于覆盖等式会约束内部列是否能与 required-outsourced job 同时选中，LP 层保证正确性；过滤内部 restricted set 只是为了减少冗余列。

列化外包主问题类中不再维护 `outsourceVars/outsourceSegmentActive/outsourceSegmentBaseline` 这一套变量，而是给 restricted outsourcing columns 建 `w_o` 变量。覆盖行加入外包列系数；新增 `sum_o w_o <= 1` 行并暴露其 dual `mu`；机器数行、arc 分支行、adjacency 行、SRI cut 行都只看内部机器列。读取解时从外包列变量重构 job-level outsourcing values，保证输出和 incumbent 校验还能沿用一部分旧接口。

`PricingResult/PC/Pool` 现在只认识 `TWETColumn`。要接外包 pricing，必须扩展结果对象和 PC 加列流程，让一次 pricing 可以返回内部列和外包列两类结果。更稳的做法是让 `PC.generateColumnsFromEngine()` 分别调用 `lp.addColumns(...)` 和 `lp.addOutsourcingColumns(...)`，而不是让外包 pricing 伪装成内部 sequence pricing。

`TWETBPCContext` 中 pricing engine 顺序建议为：内部启发式、外包 pricing、内部 exact pricing。外包 pricing 通常比内部 exact 便宜，放在内部 exact 前可以先处理明显负的外包集合；但只要任一定价器加列，就应按当前 PC 逻辑立即重解 LP 并从第一个 pricing engine 重新开始。

`RestrictedMasterIntegerHeuristic` 也要同步支持外包列模式：在候选内部列和候选外包列上解整数 RMP，覆盖使用等式，并保留 `sum_o w_o <= 1`。原来的 `y_j + tariff segment` RMIH 不能直接用于 `SP1` 的列化外包对照，否则上界口径会和 LP 不一致。

需要新增 `OutsourcingMembershipBrancher`。`TWETBPCContext` 在列化外包模式下禁用 `TariffSegmentBrancher`，brancher 顺序应为 `MachineCountBrancher`、可选 adjacency、`ArcBrancher`、`OutsourcingMembershipBrancher`；如果严格按“先 arc branch”执行，也可以把 `ArcBrancher` 放在外包分支之前，并保持 machine count 分支仍先处理机器数量上下界。

## 6. 正确性判断

这个方向可做，且和论文 `SP1` 一致。只要外包列集合最终包含所有允许的外包子集，外包 pricing 能精确证明没有负 reduced-cost 外包列，且外包 membership branch 能处理外包覆盖分数，整数层面就能表达“每个任务要么内部加工、要么外包，外包成本按总 baseline 走 `G`”。

第一版最容易踩的坑有三个。第一，覆盖行如果直接改成 `==1`，可以避免内部列和外包列重复覆盖同一 job，但这会改变当前 BPC 一直采用的 set covering 放松口径；后续第 8 节已经修正为主 RMP 仍保留 `>=1`，只在上界修复阶段做 `==` 确认。第二，外包列必须有 `sum_o w_o <= 1` 行，否则多个外包列会把 `G(B)` 拆开计费。第三，arc branch 只作用于机器列是对的，但 exact 搜索还需要外包 membership branch；否则可能在内部 arc 已经无法分支时仍存在外包列分数解。

总体建议是先实现 `SP1` 独立模式，不影响当前 `SP2`。先跑 root LP 和少量节点，确认两类 pricing 的 reduced cost、外包集合列成本、`sum_o w_o <= 1`、外包分支过滤和输出校验都正确，再把 repair、RMIH 和完整搜索接上。

## 7. 2026-06-20 第一版实现记录

第一版采用配置开关 `outsourcingModel` 保留旧 `masterVariables` 默认模式，只有设置为 `columns/columnized/sp1` 时才启用外包列化。实现上暂时没有新拆独立 LP 类，而是在现有 `LP` 内隔离两套建模路径：旧模式继续使用 `y_j + tariff segment`，列化模式使用内部机器列 `lambda_s` 和外包集合列 `omega_o`，覆盖约束仍保留 set covering 的 `>=1` 口径，并新增 `sum_o omega_o <= 1`。旧路径默认不变。

新增 `TWETOutsourcingColumn` 与 `OutsourcingPool` 存储外包集合列，外包列只记录任务集合、baseline 和 `G(baseline)` 成本，不混入内部机器列池。新增 `OutsourcingPricingEngine` 按外包 membership 分支状态做子集 pricing，reduced cost 口径为 `G(B)-sum pi_j-mu`，其中 `mu` 是外包列数量约束 dual。`PricingResult` 和 `PC` 已支持一次 pricing 返回内部列和外包列；列化模式下 pricing 顺序为内部启发式、外包 pricing、内部 exact pricing。

`Node` 增加 `OUTSOURCE_FREE/FORBIDDEN/REQUIRED` 状态和外包列 seed 继承。新增 `OutsourcingMembershipBrancher`，在内部 arc 分支之后根据当前外包 job 覆盖量 `z_j=sum_o rho_jo omega_o` 的分数值分支：左支禁止 job 外包，右支要求 job 外包，并预置一条包含 required outsourced jobs 的外包列。列化模式下禁用 `TariffSegmentBrancher` 和当前 SRI cut generator，因为旧 cut 行没有外包列系数。

RMIH 增加列化模式分支，当前实现先在内部列和外包列上解 `>=` covering MIP 识别重复覆盖，再生成内部删点修复列，最后用 `==` partition MIP 确认可行上界，并保留 `sum_o omega_o <= 1`。Tree/Context 已接入共享 `OutsourcingPool`，根节点会把初始解里的外包 job 集合作为 seed 外包列传入。当前第一版已通过主线源码编译；轻量 runtime smoke 未在短时间内闭合，因此后续还需要用小规模允许外包的实例专门验证外包列 reduced cost、membership branch 和 RMIH 上界行为。

## 8. 2026-06-20 覆盖约束口径修正

本节修正前文关于列化外包覆盖等式的表述。按当前主线和最新讨论，列化外包 RMP 仍应保留 set covering 口径：

`sum_s delta_js lambda_s + sum_o rho_jo omega_o >= 1`

因此，前文中“覆盖必须改为 `==1`”和“列化 RMIH 解 `==` 覆盖”的表述作废。列化外包只是在 RMP 中把原来的 job-level 外包变量替换为外包集合列，并用 `sum_o omega_o <= 1` 保证整体外包集合最多选一份；它不改变当前 BPC 主问题先用 `>=` 放松覆盖的策略。重复覆盖仍由后续 RMIH 修复、列生成修复和最终解校验处理，而不是在列化外包 LP 里强行排除。

## 9. 2026-06-20 RMIH 修复流程修正

列化外包模式下，RMIH 也应沿用旧 `coverRepair` 的思想，而不是解一次 `>=` MIP 后直接把结果当上界。当前修正后的流程为：先在筛选出的内部列和当前外包列上解 `>=` covering MIP；如果覆盖统计已经没有 missing/duplicate，则该解可直接作为上界；否则根据选中的内部列和 job-level 外包值识别重复 job，复用已有删点修复逻辑生成内部修复列；最后在“筛选内部列 + 修复内部列 + 当前外包列”上解 `==` partition MIP，只有最终覆盖统计无 missing/duplicate 时才返回可行上界。

这里的 `==` 只出现在 RMIH 的最终可行性确认阶段，不改变列化外包 RMP 的 `>=` 覆盖口径。外包列本身不做删点，重复外包 job 按旧逻辑视为虚拟保留侧，内部列通过删除该 job 生成修复列；最终 MIP 会重新选择是否保留该外包集合列。

## 10. 2026-06-20 全局审查结论

本次对列化外包第一版做了全局代码审查，主线 Java 编译通过，但当前实现还不能视为完全稳妥。最重要的风险有两个。

第一，外包 membership 右支 `z_j=1` 后，`Node.isColumnCompatible()` 会在 LP 加列入口过滤掉包含该 job 的内部机器列，但内部 pricing 过程本身还不知道这些 job 已经不能在内部列中出现。这样 exact pricing 可能把有限的 top-K 负 reduced-cost 预算消耗在后续会被 LP 拒绝的内部列上，导致真正兼容当前节点的负列没有返回。这个问题不是单纯性能问题，极端情况下会让 PC 误以为当前节点没有负列，从而给出错误下界。后续修复应把 required-outsourced job 作为内部 pricing 的不可访问任务直接接入扩展、seed 和候选返回逻辑，至少要做到候选 top-K 前先过滤并继续收集兼容列。

第二，主树当前会把 `solution.isInteger()` 的 LP 解直接作为 incumbent，但列化外包 RMP 覆盖行仍是 `>=`。因此一个变量全为 0/1 的 set-covering 解仍可能重复覆盖某个 job，例如内部列和外包列同时覆盖，或非基本列重复访问同一 job。RMIH 已经在启发式上界路径里做了覆盖统计和 `==` 修复，但 `Tree` 中整数 LP 捷径没有做同样检查。后续修复应在接受 LP integer incumbent 前执行 exact coverage 验证；若存在重复或缺失，则跳过该上界或转入 RMIH 修复，不能直接用该目标值剪枝。

补充说明：如果能够证明任意整数 covering 解都可以通过删点或取消重复外包不增成本地修成 exact-cover 解，那么该 covering 解的目标值可以作为一个数值上界使用。但当前代码在 LP integer shortcut 上既没有执行修复，也没有把修复后的可行列保存为 incumbent；因此从实现语义上仍不应直接把原始重复覆盖解作为最终 incumbent。这里对最优性风险的判断取决于“不增成本修复”是否已经成为可执行的不变量，而不仅是直觉上可修。

其它部分暂时看作第一版可接受的不变量或性能点：外包 pricing 的 reduced cost 已包含外包列数量约束 dual；`sum_o omega_o <= 1` 对保持整体外包 tariff 计费是必要的；RMIH 的列化流程已经改为“先 >= 找重复，再生成内部修复列，最后 == 确认”。当前 SRI 在列化外包模式下禁用是保守处理，因为旧 cut 行没有外包列系数。外包 pricing 的 label dominance 依赖 `G(B)` 非降，若以后允许非单调 tariff，需要单独改 dominance。诊断输出目前只报告内部 pool size，不含外包列池，后续分析日志时要注意这一点。

## 11. 2026-06-20 required-outsourced job 内部 pricing 修复

本次只修复外包 membership 右支对内部机器列生成的影响，暂不处理整数 covering 解作为 incumbent 的语义问题。新增 `PricingCompatibility` 作为 GC 包内统一判断：若当前节点要求 job 外包，则该 job 对内部机器 pricing 不可用。

修复点集中在扩展图和 seed/move 入口，而不是在候选返回或 PC 层做兜底。普通单向、双向、GCBB、GCNGBB 和 ng-DSSR pricing 在扩展入口直接拒绝扩展到 required-outsourced job；time-indexed graph pricing 则在静态 processing arc 表中把所有进入 required-outsourced job 的内部 processing arc 禁掉，等价于从内部机器图中删除该 job。启发式 pricing 保留 `isSequenceCompatible()` 检查，因为它存在 add/exchange 等直接修改序列的 move，并非所有候选都来自逐点扩展图。

曾短暂加入过“完整序列进候选堆前再检查”和“PC 写全局 Pool 前兜底过滤”两层后置检查。复核后认为这些不必要：若扩展图已经正确删除 required-outsourced job，exact pricing 恢复出的序列不应再包含该 job；PC 层兜底也会掩盖新定价器漏接节点语义的问题。因此已删除这两类后置保底，只保留真正参与生成过程的约束入口。

这样处理后，外包右支 `z_j=1` 的语义不再只依赖 LP 加列时的后过滤，而是在内部 pricing 生成阶段就生效。主线 Java 编译通过；后续仍建议用含外包右支的小算例做 runtime 验证，观察内部 pricing 返回列是否不再包含 required-outsourced job。

## 12. 2026-06-20 外包列化流程再次复核

本次按“主问题、外包 pricing、内部 pricing、分支、RMIH、输出验证”重新检查了列化外包流程。当前主语义是对齐的：列化模式下 `LP` 使用内部机器列 `lambda_s` 和外包集合列 `omega_o`，覆盖行保持 `>=1`，外包集合列计数行 `sum_o omega_o <= 1` 提供外包 pricing 的 dual；外包列 reduced cost 使用 `G(B)-sum pi_j-mu`；arc、adjacency、machine 约束只作用于内部机器列；外包 membership 分支在 arc 分支之后处理 job-level 外包覆盖量。

外包列自身的 pricing 逻辑目前是正确的一阶实现。`OutsourcingPricingEngine` 根据 `OUTSOURCE_FORBIDDEN/REQUIRED` 过滤或强制 job，遇到不可外包 job 且该 job 被 required 时直接返回无改进；候选外包列只按 job 集合、baseline 和 `G(baseline)` 建列。这里的 dominance 条件为 `baseline` 不大且 `profit` 不小，因此依赖外包 tariff `G(B)` 非降。若以后允许非单调外包成本函数，需要单独重做外包 pricing dominance。

口径 A 下，`OUTSOURCE_REQUIRED` job 不应再出现在内部机器列里。现在 `Node.isColumnCompatible()` 会在 LP restricted set 入口过滤这类内部列；各主要内部 pricing 已在扩展入口拒绝走到 required-outsourced job，time-indexed graph 则在 processing arc 表里禁掉进入该 job 的内部弧；启发式 pricing 保留 seed/move 后的序列兼容检查，因为它不是纯逐点扩展图。这样已经能防止正常 pricing 继续生成包含 required-outsourced job 的内部列。

需要继续清理的是 dominance 集合口径。若干双向定价器在 `buildReachableSet/buildExtensionSet` 中仍先把 required-outsourced job 当作“可达候选”放入集合，然后在 `canExtend` 阶段拒绝直接扩展。它通常不会导致非法内部列进入 LP，但 dominance key 表示的“未来可扩展能力”和真实内部 pricing 图不完全一致，会削弱 dominance，也会让正确性推理变得不干净。更合理的做法是把 required-outsourced job 直接从 reachable/extension set 构造阶段删除，和“该 job 已从当前内部机器图中删除”的语义一致；后续如果继续完善列化外包，应优先做这个小范围清理，而不是在候选返回或 PC 层重新加兜底过滤。

RMIH 列化路径目前符合之前讨论：先在当前内部列和外包列上解 `>=` covering MIP，若覆盖统计无重复/缺失则返回；否则根据重复 job 生成内部删点修复列，最后在“筛选内部列 + 修复内部列 + 外包列”上解 `==` partition MIP。外包列在 repair 中作为虚拟保留侧，不做删点，最终由 `==` MIP 决定是否选中。当前仍默认至少存在一条内部机器列；纯外包且内部列为空的极端节点暂不作为支持目标。

还有一个统计边界需要记录：诊断输出的 pool size 原来主要报告内部列池，不含外包列池；分析列池规模时要额外看 `OutsourcingPool`。这个统计口径已在后续第 13 节修正。

## 13. 2026-06-20 required-outsource 集合口径与列池统计修正

本次把第 12 节提到的第一个问题改成源头处理：required-outsourced job 现在直接从内部机器 pricing 的 reachable/dominance set 构造阶段删除，而不是先进入集合、再由 `canExtend` 拒绝扩展。这样 dominance key 表示的未来可扩展集合与当前节点的内部机器图一致，也减少了无意义的扩展候选。对应地，普通 `canExtend` 只保留当前直连弧、pricingOnly 弧等即时弧检查；time-indexed graph 已经通过 processing arc 表禁掉进入 required-outsourced job 的弧，不需要额外改。

列池统计也同步改成“内部列池 + 外包列池”的总规模。`Tree` 的节点 picked/master solved/heartbeat、`PC` 的 pricing trace，以及 `BPCResultWriter` 的全局列池大小都使用总列池数。这样列化外包模式下日志不会低估列池规模；旧 `masterVariables` 模式下外包列池大小为 0，统计口径不变。

不可外包 job 的处理也改成源头口径。`OutsourcingMembershipBrancher` 在选择外包 membership 分支对象时直接跳过不可外包 job，因此 big-M 外包成本的 job 不会被分到 `z_j=1` 右支。`OutsourcingPool.addColumn()` 不再静默删除不可外包 job；如果上游仍把这类 job 传入外包列池，直接抛出异常暴露调用方错误。普通外包 pricing 本身已经只枚举允许外包的 job，因此不受影响。

RMIH 内部列为空的情况暂不处理。当前算法构造和分支修复都假设至少有内部机器列参与，实际算例也基本不可能出现内部列池为空的正常节点；因此不为“纯外包节点”额外放宽入口条件，避免为了低概率场景增加模型分支。

验证上，按 TWET 主线源码范围重新编译通过。旧 `src/BPC` 目录是历史 VRP 代码，直接全量编译 `src` 会失败，这不是本次修改引入的问题。

## 14. 2026-06-20 列化外包全局正确性复查

本次再次从 LP、PC、Tree、Node、外包 pricing、内部 pricing 入口、membership branch 和 RMIH 路径复查列化外包口径。当前代码没有发现新的正确性破坏点。列化模式下主 LP 保持 `>=1` 覆盖，内部机器列覆盖系数使用 `getJobVisitCount(job)`，外包集合列覆盖系数为 0/1，并通过 `sum_o omega_o <= 1` 维持整体外包 tariff 计费。arc、adjacency、machine 和 subset-row cut 行只作用在内部机器列；列化模式下当前不启用旧 SRI cut generator，避免外包列系数口径混入旧 cut 行。

外包 membership 分支现在只会选择可外包 job。不可外包 job 不会进入 `z_j=1` 右支；如果上游仍试图构造包含不可外包 job 的外包集合列，`OutsourcingPool` 会直接抛错暴露源头问题。外包 pricing 本身也只枚举可外包 job，并按 `OUTSOURCE_FORBIDDEN/REQUIRED` 状态过滤或强制加入。这里仍有一个建模不变量：外包成本函数 `G(B)` 需要非降，否则 `OutsourcingPricingEngine` 中以 baseline 不大、profit 不小为准的 dominance 关系需要重新设计。

内部机器 pricing 对 required-outsourced job 的处理已经放到生成源头：主要双向、GCBB、GCNGBB、ng-DSSR 和 time-indexed pricing 都在 reachable/extension 或 processing arc 构造阶段移除这类 job；启发式 pricing 保留 seed/move 序列兼容检查，因为它不是单纯逐点扩展图。PC 层没有再做隐藏式过滤，避免掩盖定价器漏接节点语义的问题。

RMIH 列化路径与当前讨论一致：先解 `>=` covering MIP 找重复覆盖结构，若覆盖统计已经 exact 则直接返回上界；否则生成内部删点修复列，再用 `==` partition MIP 做最终确认。纯外包且内部列为空的极端场景仍不支持，这是当前实现的明确边界，不作为本次修复目标。另一个保留口径是 Tree 对 LP integer covering 解的 shortcut 暂未做 exact-cover 修复；按当前讨论这不会破坏下界正确性，但若要把该解作为最终可行排程输出，后续仍应接入和 RMIH 一致的覆盖校验或修复。

验证方面，重新按 TWET 主线源码范围编译通过。全量 `src` 仍不作为验证标准，因为旧 `src/BPC` 历史代码和当前 `Basic.Data` 接口不兼容。

## 18. 2026-06-20 列化外包变慢原因分析

从 `n=40,m=2,case=14` 的倍率测试看，列化外包慢的主因不是外包子集 DP 本身。`scale=5` 时列化模式总耗时 `82.238s`，其中 `OutsourcingPricing=156/12203/0.568s`，真正增加的是 `pricing_rounds=654`、内部启发式 `425` 次和 master LP 累计 `4.277s`；同一倍率的 `masterVariables` 只有 `pricing_rounds=267`、内部启发式 `196` 次。`scale=1` 下差异更极端：原变量模型 7 轮、274 列闭合，列化模型 80 轮、外包 pricing 生成 3695 条外包列、内部启发式调用 63 次。

这里的结构性原因是，当前随机算例使用线性 tariff。原 `masterVariables` 模型用 job-level `y_j` 和 tariff segment 变量直接表达所有外包组合，是一个紧凑 formulation；列化模型则用外包子集列 `omega_o` 和 `sum_o omega_o <= 1` 表达同一件事。即使语义等价，LP relaxation 中一个任意分数外包向量也需要若干外包集合列组合才能表达，因此列化模型必须通过 pricing 逐步补外包子集列。在线性 tariff 下 reduced cost 可写成 `sum_j (rate*b_j - pi_j) - mu`，很多相近子集都会是负列；当前 `OutsourcingPricingEngine` 又复用 `maxExactPricingColumns=5000`，会把大量负子集一次性放回 master，造成列池膨胀和 dual 频繁变化。

PC 调度节奏会进一步放大这个问题。列化模式的 engine 顺序是 `HeuristicPricing -> OutsourcingPricing -> exact`，且任一 engine 加列后立即重解 LP，并从第一个 engine 重新开始。外包列每轮改变覆盖 dual 后，内部启发式会被反复唤起；反过来内部列加入后，又会打断外包 pricing 的闭合过程。这种节奏对“一个重 exact pricing + 少量内部修复列”比较合适，但对“便宜且会持续产生大量外包子集列”的列化外包不够好。

因此当前列化外包效果差主要是工程策略问题，不是正确性问题。优先优化方向应是外包列返回策略：增加单独的 `maxOutsourcingPricingColumns`，先测试每轮只返回 1/10/50 条；对线性 tariff 增加 shortcut，只返回“所有负边际 job 组成的最佳集合”或少量扰动集合，而不是所有负子集；也可以尝试把 `OutsourcingPricing` 放到内部启发式之前，或者在外包列连续为负时先闭合外包 pricing，再启动内部启发式。后续若继续使用 SP1 列化外包，重点不应放在加速外包 DP，而应放在减少外包列数量、降低 dual 震荡和减少内部启发式被重复触发。

随后直接测试了“把 `OutsourcingPricing` 放到最前面”的顺序。`n=40,m=2,case=14,scale=1` 在该顺序下超过 90 秒仍未结束，而原顺序同一配置约 `3.3-3.7s` root 闭合；恢复原顺序后复跑为 `columns=3.357s`，目标和结构仍与原模型一致。说明当前问题不是“外包列补得太晚”，而是外包列一次返回太多、并且过早改变 dual 后会造成更严重的重启 churn。因此默认仍保持 `HeuristicPricing -> OutsourcingPricing -> exact`，后续应优先限制外包列返回数量，而不是简单把外包 pricing 前置。

按当前默认顺序 `HeuristicPricing -> OutsourcingPricing -> exact` 再复跑 `n=40,m=2,case=14,scale=5`，得到 `columns=82.038s`，目标 `481.75`，外包 `19/40`，与前一轮同顺序 `82.238s` 基本一致。由此确认当前默认顺序稳定，但它不是 columns 慢的解决点。

进一步增加独立的 `maxOutsourcingPricingColumns` 后测试外包列返回数量。`scale=5` 下，`outcap=10` 反而恶化为 `columns=124.329s`，pricing round 从默认的 `654` 增加到 `1383`，内部启发式调用从 `425` 增加到 `902`；`outcap=50` 为 `83.810s`，`outcap=200` 为 `81.926s`，和默认 5000 基本同量级。说明外包列一次返回太少会造成外包闭合过慢和更多重解；达到几十到几百后，主要瓶颈已经不是外包列返回上限。关闭内部启发式并用 `outcap=200` 时，150 秒仍未结束，也说明当前启发式虽然耗时高，但对内部列补充和整体收敛仍有帮助，不能简单关闭。

当前更准确的结论是：列化外包慢主要来自 formulation 层面的列生成代价。原变量模型用 `y_j` 直接表达外包分数向量，而列化模型需要外包集合列逐步逼近；这些外包列改变 job dual 后，又会触发更多内部启发式和 exact pricing 轮次。单独限制外包列数量只能在“太小会更慢”和“足够大后收益饱和”之间调参，不能从根本上消除差距。后续若继续优化，应考虑线性 tariff 专用 shortcut 或稳定化策略，而不是单纯调外包列返回上限。

## 19. 2026-06-20 Dabia/Lai/Vigo common carrier 论文对照

阅读 `An Exact Algorithm for a Rich Vehicle Routing Problem with Private Fleet and Common Carrier` 后，当前列化外包实验和论文结论基本一致。论文同样比较了两种 set-partitioning 口径：SP1 同时为 private fleet route 和 common carrier subcontracted subset 做列生成；SP2 只对 private fleet route 做列生成，把外包 customer decision 和 tariff segment 留在 master 中。论文明确指出 SP1 的线性松弛更强，但由于 private pricing problem 和 common pricing problem 都会倾向选择高 dual customer，容易在同一轮列生成中同时生成内部路线列和外包子集列，形成较强对称性；同时还要解两个 pricing subproblem，因此总体计算效果反而不如 SP2。

该结论和我们 `masterVariables` 与 `columns` 的对拍非常接近。当前 `masterVariables` 模式就是 SP2 型口径：内部机器列由 pricing 生成，外包通过 `y_j + tariff segment` 紧凑表达；`columns` 模式对应 SP1 型口径：内部机器列和外包集合列分别生成，并用 `sum_o omega_o <= 1` 保证整体 tariff 计费。测试中列化外包结果正确，但 pricing round、外包列池和内部启发式 pricing 调用显著增加；外包 DP 本身很便宜，真正慢的是外包集合列进入 master 后造成 dual 反复变化和列生成节奏被打散。论文 Table 6 也呈现同类现象：SP1 root lower bound 略强，但平均求解时间和可解实例数都不如 SP2。

因此当前阶段不宜把外包列化作为默认模型。更稳的主线仍应是保持 `masterVariables` 口径，把外包决策留在主问题中；列化外包可以保留为实验分支，用于后续测试非线性 tariff、更复杂 common carrier 约束或需要外包集合本身参与 cut/branch 的场景。若继续优化列化外包，重点不应放在加速 common pricing DP，而应放在减少外包集合列数量、降低 symmetry/churn、稳定 dual，以及避免外包列反复触发内部启发式 pricing。

## 20. 2026-06-20 root bound 复跑

为直接判断“外包列化是否让根节点下界更强”，本次给 `OutsourcingModelComparisonTest` 增加 `root_bound` 输出字段，读取 `BPCTraceSummary.getRootBound()`，并用 `maxNodes=1` 只处理根节点。测试对象为 `n=40,m=2,case=14`，外包成本倍率分别为 `scale=1/5/10/20`，其余配置沿用当前外包对拍默认口径。

复跑结果如下。`scale=1` 时，`masterVariables/columns` 的 root bound 都是 `135.350000`，root 时间分别为 `0.603s/3.150s`，pricing rounds 为 `7/80`。`scale=5` 时 root bound 都是 `477.937500`，root 时间为 `2.786s/19.985s`，pricing rounds 为 `11/112`。`scale=10` 时 root bound 都是 `749.833333`，root 时间为 `6.206s/24.868s`，pricing rounds 为 `22/83`。`scale=20` 时 root bound 都是 `1124.285714`，root 时间为 `11.428s/35.606s`，pricing rounds 为 `26/104`。

因此在当前这组线性 tariff 算例上，外包列化没有表现出更强 root bound。它和 `masterVariables` 给出的根节点下界完全一致，但需要明显更多 pricing rounds 和更长 root 时间。这和前面对 formulation 的判断一致：当前 `y_j + tariff segment` 已经是外包线性 tariff 的紧凑表达，列化外包没有带来下界收益，反而引入了外包集合列生成和 dual churn。论文中 SP1 root bound 略强的现象没有在这组 TWET 线性 tariff 随机算例上出现；后续若要继续寻找列化外包可能更强的场景，应测试非线性/凹折扣 tariff 或更复杂 common-carrier 约束，而不是当前线性 tariff 口径。

结果文件写入 `test-results/bpc/2026-06-20-outsourcing-rootbound-n40-scale1.csv`、`scale5.csv`、`scale10.csv` 和 `scale20.csv`。这些 CSV 只用于 root-bound 对照，`scale=10/20` 下 root-only incumbent 不一致是正常现象，因为搜索被限制在根节点，不代表最终最优值不一致。

## 21. 2026-06-20 root bound 未强化的原因

这次 root bound 没有因为外包列化变强，主要不是测试偶然，而是当前外包成本结构和 LP 松弛口径决定的。当前随机算例使用的是线性 tariff，旧 `masterVariables` 模型中的 `y_j + tariff segment` 已经能直接表达任意分数外包向量，并通过 baseline 变量按同一个总外包量计费。列化模式只是把“选择一个外包集合”展开成 `omega_o` 列，再用 `sum_o omega_o <= 1` 保证整体计费。对线性 tariff 来说，这个展开后的外包凸包投影回 job-level 变量，和旧模型的 `y_j` 表达能力基本一致，所以不自然产生更强 root bound。

更具体地说，列化模式下任意外包列组合可以映射为 `y_j = sum_o rho_jo omega_o`，其成本是这些外包集合成本的加权和；线性 tariff 下，这正好等于按 `sum_j b_j y_j` 计算的外包成本。反过来，旧模型里的任意分数 `y` 也可以理解成某个外包集合分布的边际概率；只要外包列定价补齐到闭合，两边 LP 松弛给出的外包部分下界就会一致。因此列化外包真正增加的是变量生成过程和 dual churn，而不是根节点 relaxation 强度。

外包列化可能更强的场景并不是当前这种线性 tariff。它需要额外结构让“外包集合列”表达的信息超过 job-level `y_j`，例如非线性且不能被现有 segment 变量紧表达的 common carrier 成本、外包集合本身参与 cut/branch、或者外包侧有容量/路线/组合约束。当前实现里外包列不参与机器 arc 分支，覆盖仍是 set covering `>=`，也没有外包集合 cut，因此根节点可利用的信息和旧模型基本相同。由此解释了本次四个倍率下 root bound 完全一致，而列化模式只是更慢。

## 22. 2026-06-20 与 VRP 论文 root bound 差异的再解释

前一节“旧 `y_j + segment` 已经足够紧”的结论只对本次测试的单段线性 tariff 成立，不能推广到论文中的折扣 tariff 场景。`OutsourcingModelComparisonTest` 在构造数据时把外包函数重建为一段 `G(B)=rate*B`，因此旧 master 中的 `y_j` 和 baseline 等式已经完整表达了外包成本：任意分数外包向量的成本就是 `rate * sum_j b_j y_j`。这种情况下，外包集合列的凸包投影回 `y` 空间不会额外加强根节点。

VRP common-carrier 论文中 SP1 root bound 更强，原因恰恰在于 SP2 的 master 变量通常不能完整刻画外包集合成本的凸包。特别是存在折扣/凹 tariff 时，简单的 segment active 变量会给出函数图像的连续松弛或下凸包近似；它只知道总 baseline 和分段选择，不知道这个总量来自哪些 customer 组合。SP1 的外包集合列则把“具体哪个集合被外包”和对应的 `G(sum b_j)` 绑定在一起，LP 松弛是在集合列凸包上做组合，因此可能比 job-level `z_j/y_j + segment` 更强。

这也解释了当前实验和论文看起来不一致的地方：实现路线是类似的，但测试函数不同。当前 root-bound 对拍用的是线性 tariff，旧 master 没有明显凸包缺口，所以 `columns` 不会更强；论文的优势来自非线性 common-carrier 折扣和集合列对组合成本的刻画。若要验证我们这里是否也能出现类似强化，应构造两段或多段凹折扣 tariff，并重新做 root-only 对拍；例如沿用 `MediumBPCBatchTest` 中 `0.8/0.25` 两段折扣函数，而不是 `OutsourcingModelComparisonTest` 当前的一段线性函数。

## 23. 2026-06-20 多段凹折扣 tariff 对拍

为验证上一节判断，本次扩展 `OutsourcingModelComparisonTest`，新增 `twet.bpc.outsourcingCompare.tariffSegments` 参数。默认仍为 1 段线性，显式设置为 3 或 5 时，会构造连续凹折扣外包函数：第 1 段斜率为 `1.0`，后续第 k 段斜率为 `1/(k+1)`，并按连续性设置截距。CSV 同步增加 `tariff_segments` 字段，避免后续混淆线性和多段折扣测试。

先在 `n=40,m=2,case=14,maxNodes=1` 上做 root-only 对拍。3 段 tariff 下，`masterVariables` root bound 为 `173.611111`，`columns` root bound 为 `206.923077`；5 段 tariff 下，`masterVariables` root bound 为 `136.970000`，`columns` root bound 为 `163.567925`。这说明一旦 tariff 变成凹折扣，旧 `y_j + segment` master 的 LP 松弛确实出现凸包缺口，外包集合列可以明显强化根节点下界。root-only incumbent 不一致是正常现象，因为只处理根节点，不代表最终最优不一致。

随后用 `n=20,m=2,cases=7,8` 做完整对拍，确认正确性。3 段 tariff 下，两例最终目标和 bound 都一致：case 7 为 `93.0`，case 8 为 `88.0`；但 root bound 分别从 `68.305556/63.500000` 提高到 `82.666667/81.759259`。5 段 tariff 下，两例最终也一致：case 7 为 `80.0`，case 8 为 `75.833333`；root bound 分别从 `55.470000/53.076667` 提高到 `68.920000/66.808333`。

性能上，列化外包仍然明显更慢。`n=20` 的 3 段测试中，`masterVariables` 总计 `0.622s`，`columns` 为 `31.515s`；5 段测试中，`masterVariables` 为 `1.491s`，`columns` 为 `70.379s`。慢的原因和前面一致：外包集合列强化了 bound，但也带来大量 pricing round、外包列和内部启发式调用。当前结论应修正为：线性 tariff 下列化外包几乎没有下界收益；多段凹折扣 tariff 下列化外包确实能强化 root bound，但计算代价很高，是否值得作为默认模型还要看更大规模和整体 BPC 收敛表现。

## 24. 2026-06-20 多段 tariff 下两种方法继续对比

继续用多段凹折扣 tariff 对 `masterVariables` 和 `columns` 做对比。`n=30,m=2,cases=12,13,maxNodes=1` 的 root-only 结果显示，列化外包的 root bound 强化比较稳定。3 段 tariff 下，case 12 从 `101.722222` 提高到 `130.685185`，case 13 从 `129.944444` 提高到 `152.444444`。5 段 tariff 下，case 12 从 `79.685000` 提高到 `104.538000`，case 13 从 `100.073333` 提高到 `121.007143`。对应 root 时间从 `0.14-0.56s` 上升到 `0.83-1.54s`，root 阶段仍可接受。

但完整搜索代价仍然很高。`n=25,m=2,case=9,3 段 tariff` 完整求解中，两种方法最终目标和 bound 都为 `121.833333`，验证结果一致；root bound 从 `86.111111` 提高到 `110.095238`。不过总时间从 `masterVariables=1.007s` 增加到 `columns=55.298s`，节点数从 `5` 增加到 `246`，pricing rounds 从 `27` 增加到 `2140`，生成列从 `813` 增加到 `24363`。其中 `OutsourcingPricing` 自身只有 `0.323s`，主要代价仍是外包列改变 dual 后反复触发内部启发式和 exact pricing。

当前判断是：多段凹折扣下，列化外包作为 SP1 确实能给更强 root bound，这点和 VRP common-carrier 论文一致；但在当前 PC 调度和外包列返回策略下，更强 root bound 并没有转化为更快完整求解，反而因为外包集合列和内部列之间的 dual churn 导致搜索更慢。后续若继续研究 SP1，应优先考虑稳定化、外包列返回策略、外包列池压缩或分阶段闭合外包 pricing，而不是只看 root bound 强度。
