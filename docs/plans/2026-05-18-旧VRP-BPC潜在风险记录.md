# 旧 VRP BPC 潜在风险记录

这份记录只保留当前已经确认的旧 VRP 参考代码边界风险。前面一度怀疑 `UpdateRouteSet()` 在首次 LP 求解前已经筛选 child 列集，因此可能导致 coverage 或历史分支约束被提前破坏；复核代码后确认这个前提不成立。旧代码的真实流程是先把 child 分支约束写入父节点当前 LP，再直接用当前 `lp.route_index` 求第一次 LP；只有首次 LP 可行，或者 `AddSlack + FindFeasible` 修复成功后，才在 `UpdateRouteSet()` 末尾筛选 child 的初始 `route_set`。因此，`AddSlack` 只加当前分支行本身不能视为主要漏洞。

## 1. UpdateRouteSet 的实际流程

旧 VRP 的 `UpdateRouteSet()` 主要出现在 `BranchA/B/C/D` 中。一个 child 生成时，代码先修改当前 LP 中对应的分支约束。例如车辆数分支会修改 `cus2rng[0]` 的上下界，arc 分支会通过 `ForceArcValue(i,j,0,0)` 或把同一条 arc 分支行 bounds 改成 `1,1`。这些约束写入以后，`UpdateRouteSet()` 才调用第一次 `lp.cplex.solve()`。

所以第一次 LP 不是“筛选后的 child 列集合 LP”，而是“父节点当前列集合 + 当前 child 分支约束”的 restricted LP。如果这一步可行，说明当前列集合已经能在新分支约束下给出一个可行 LP 解。此时末尾筛选通常会保留该可行解所需的列，因为基变量 reduced cost 通常为 0，会通过 `m_addin_red_cost` 这种宽阈值。若第一次 LP 不可行，则新增分支约束确实是最直接的新冲突来源；给当前分支行加 slack，并用其对偶引导 `FindFeasible()` 生成修复列，是合理的工程做法。

这和之前误判的情况不同。真正需要注意的是末尾筛选和有限补列预算，而不是“首次 LP 前筛列导致不可行”。

## 2. 末尾列筛选的真实含义

旧代码真正的列筛选发生在 `UpdateRouteSet()` 成功之后。成功可以来自第一次 LP 直接可行，也可以来自 `AddSlack + FindFeasible` 修复成功。随后代码遍历当前 `lp.route_index`，按 reduced cost 阈值和数量上限筛出 child 的 `route_set`。车辆数分支 `BranchA` 只按 `reduced cost < m_addin_red_cost` 和 `m_initial_col_number` 筛选，因为每条 route 在车辆数行中的系数都是 1，车辆数约束是否满足取决于 route 变量组合，而不是单条 route 自身。`BranchB/C/D` 涉及 customer、bid 或 arc 状态时，还会额外检查 `n.feasible_arc`，过滤包含 forbidden arc 的 route。

这个筛选通常是安全的，但不是严格证明。若存在大量 reduced cost 很低或同为 0 的列，而 `m_initial_col_number` 又不足以全部保留，理论上可能把某些维持 child 可行性的列挤掉。旧代码依赖较宽的 `m_addin_red_cost=5000` 和 `m_initial_col_number=1000` 降低这个风险。对常规算例来说，这更像工程边界，而不是高频错误。

## 3. FindFeasible 的列数上限是否是真风险

`FindFeasible` 的上限仍然是一个真实的边界风险，但性质是“有限预算修复”，不是逻辑漏洞。旧代码中 `m_branch_col_number=500` 同时影响外层 `UpdateRouteSet()` 修复循环和 `GCTabu.FindFeasible()` 内部循环。修复流程大致是：先加当前分支行 slack，然后反复调用 Tabu 和 NGBB 生成列；每次生成列后重新求解带 slack 的 LP，如果 `IsNoSlack()` 为真，就返回成功；如果没有新列、超过 `m_branch_col_number` 或超时，就停止。

严格意义上，如果完整列空间中存在可行 child，但 Tabu/NGBB 在 500 条修复列和时间限制内没有找到足够的列，旧代码会把该 child 当作修复失败并丢弃。因此这个上限不能作为“数学不可行证明”。不过它也不是前面那类结构性错误：修复目标是让 slack 归零，只要找到足够的列即可，不要求枚举完整列空间。对旧 VRP 常规规模来说，500 条通常是一个够宽的工程阈值；迁移到 TWET-BPC 时更稳的做法是保留上限作为防爆保护，同时在状态输出中区分“无新列可加”“达到修复列数上限”“超时”和“确认为不可行”。

## 4. GCTabu 与 GCNGBB 的状态差异

旧 `GCTabu.FindFeasible()` 不是只返回候选列。它内部会求带 slack 的 LP、读取 dual、执行 Tabu 扩展，把新列直接加入 `lp.pool` 并调用 `lp.AddColumn()` 接入当前 CPLEX 模型，然后继续下一轮。因此，外层再调用 `GCNGBB.FindFeasible()` 时，`GCNGBB` 看到的可能已经是 Tabu 加列并重新求解后的 LP。

`GCNGBB.Reset()` 只清空 `m_low_ng_set` 和 `m_high_ng_set`。正常 pricing 的 `Solve(lp)` 开头会 reset；但 `FindFeasible(lp,n,col_number)` 本身不 reset，只调用一次 `Extend(lp,n)` 并检查 slack。旧 `UpdateRouteSet()` 只有在前面的 Tabu 新增列时才调用 `gc.Reset()`。如果连续进入 `GCNGBB` 且中间 Tabu 没有新增列，前一次 exact pricing 动态扩充的 ng-set 会被保留。这可以理解为保留 DSSR/ng-route 过程中发现的重复环信息；它是经验性状态管理，不是必须照搬的接口形式。

## 5. 迁移到 TWET-BPC 时的当前判断

旧代码这部分总体是合理的工程实现。需要继承的是：先在当前 restricted LP 上尝试 child 分支约束，失败后用 slack repair 找列，成功后再为 child 准备一个较宽的初始列集。需要避免的是把有限修复失败直接写成严格不可行证明，也不要把 reduced-cost/数量筛选当成绝对安全的保可行操作。

当前 TWET-BPC 可以比旧代码更保守一些：保留 repair limit 的状态输出，必要时对 coverage、required arc 等关键行分别加 slack，并在 child 初始列筛选中显式保护关键列。这样不是因为旧代码明显错，而是因为 TWET 引入了外包、tariff segment、setup time/cost 和后续更复杂分支，restricted column set 的结构比纯 VRP 更容易出现边界情况。

## 6. 双向 labeling 中 `max_time/2` 切分的潜在偏移

旧 VRP 的 `GCNGBB` 和 `GCBDT` 在 bounded bidirectional labeling 中使用的是全局时间 horizon 的一半。代码上，构造函数里取 `max_time = data.m_customer.get(data.m_customer_number).m_late_time`，然后 forward/backward 扩展都围绕 `max_time/2` 控制半边搜索。这个规则简单，但它隐含了一个假设：有效时间窗大致分布在 `[0,max_time]` 的中部附近。

如果客户时间窗整体集中在 horizon 后半段，固定取 `max_time/2` 会左偏。此时 forward 很快被截断，backward 半边拥有更大的可回推空间，后缀 label、reachable set 和 join 负担可能集中到 backward 侧，双向搜索就不再真正平衡。旧 VRP 里这个问题未必高频暴露，因为经典 VRPTW 的 depot early 通常从 0 开始，时间窗分布也未必整体右移，而且还有 time/capacity bound 与 NG memory 共同剪枝；但从迁移角度看，`max_time/2` 不是稳健的通用 midpoint 规则。

TWET-BPC 这里已经出现了同类现象：如果直接用 `Cmax/2` 或固定全局比例，而当前 profitable/hard window 实际集中在后半段，双向会明显失衡。因此当前更合理的规则是按本轮有效窗口区间切分：`L=max(min_j H_j^start, min_j setup_0j+p_j)`，`H=max_j H_j^end`，再取 `T^mid=(L+H)/2`。这不是说旧 VRP 代码一定错，而是说明它的 midpoint 选择依赖实例时间窗分布；迁移到 TWET 这种动态窗口会变化的问题时，不能机械照搬 `max_time/2`。

## 7. subset-row cut 的加入数量和节点继承

旧 VRP 的 subset-row cut 不是每轮都无条件分离。`PC` 在列生成结束并发现当前 LP 仍然非整数后调用 `GenCut.Solve(lp)`，而 `GenCut` 会先尝试普通 cut、clique cut、bid partition 等分离；只有当前没有普通 cut，也没有 clique cut 时，才进入 `SRCut.Solve(lp)` 分离 subset-row cut。也就是说，SRI 在旧代码里更像是常规割失败后的后备加强，而不是和所有割同时批量加入。

`SRCut` 的分离对象是三客户集合。它遍历 `c1<c2<c3`，统计当前 LP 中访问该三元组至少两个客户的 route 变量值，以及覆盖该三元组至少两个客户的 bid 变量值。如果二者之和大于 1，就形成一个违反的 subset-row cut 候选。代码中的 `total` 不是 reduced cost，而是这条候选 cut 左端在当前 LP 解下的取值，即 `route_sum + bid_sum`；cut 右端是 1，所以违反量可以理解为 `total - 1`。候选会按 `total` 从大到小排序，并经过两个阈值过滤：最佳候选的 `total` 至少要达到 `m_min_sr_addim=1.1`，后续加入时单条候选的 `total` 不能低于 `m_min_sr_threshold=1.02`。如果最佳候选低于 1.1，本轮一条 SRI 都不加；如果排序列表后面的候选低于 1.02，就停止继续向后扫。因此，即使存在轻微违反，例如 `total=1.005`，旧代码也会故意跳过。

每次分离最多加入 `m_max_sr_number=10` 条 subset-row cut，这是 `Common.Configure` 里的默认值。实际加入数量可能是 0 到 10 条，还会受到 `m_max_appear_number=20` 的限制：如果某个客户已经在太多已选 SRI 中出现，本轮枚举会跳过包含它的三元组。加入时先查全局 `SRCutPool`，已有定义就复用 index；但如果该 index 已经在当前 LP 的 `sr_map` 里，就不会重复加入。

真正写入 LP 时，`LP.AddSRCut(..., false)` 会为每个三客户 SRI 建一条 `<= 1` 的行。系数规则和分离时一致：一条 route 如果覆盖三元组中至少两个客户，对该行系数为 1；一个 bid 如果覆盖三元组中至少两个客户，对该行系数也为 1。新增 cut 的 index 会追加到当前 `LP.sr_index`，而这个对象在 `LP.Construct(node, ...)` 里直接指向 `node.sr_index`，所以本节点分离出的 SRI 会写回当前节点。

子节点会继承这些 cut，但不是全树全局自动下发，也不是生成子节点以后重新判断一遍该保留哪些父节点 cut。分支时左子节点通常直接沿用当前 `lp.node`，右子节点通过 `Node.Copy()` 克隆；`Node.Copy()` 会 clone `cut_index`、`sr_index` 和 `cli_index`。因此当前节点已经加入的 subset-row cut 会原样进入它的左右子节点。后续子节点重新构建 LP 时，`LP.Construct()` 调用 `AddSRCut(sr_index, true)`，按继承的 index 从全局 `SRCutPool` 取定义并重建行。相反，其他已经存在的兄弟节点不会因为全局 pool 里有这个 SRI 就自动拥有该 cut；每个 node 真正生效的 cut 仍由自己的 index list 决定。子节点求解过程中当然还可以继续分离新的 SRI，但那是在继承父节点 cut 的基础上追加，而不是替换父节点 cut。
