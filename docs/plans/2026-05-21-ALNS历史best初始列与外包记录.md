# 2026-05-21 ALNS 历史 best 初始列与外包记录

本次把前一段讨论落到代码里。`Configure.updateBestSolution()` 现在除了维护 `bestSolution`，还会把每一次全局 best 的副本追加到 `bestSolutionHistory`。这个入口覆盖初始构造解、ALNS 刷新出来的全局 best，以及 VND commit 后刷新出来的全局 best，因此不需要在 ALNS 或 VND 内部单独加采集逻辑。

`InitialColumnBuilder` 的 root 初始列逻辑保持旧 VRP 风格：最终 seed solution 的每台真实机器完整序列仍然是 incumbent column set；如果 `TWETBPCConfig.useBestSolutionHistoryForInitialColumns=true`，则额外遍历历史 best 解，把其中真实机器上的完整序列加入 root initial column set。这里不切短子序列，也不补 singleton；新增列仍通过 `Pool.addColumn()` 的序列签名和 `LinkedHashSet` 去重。这样历史 best 只增加 RMP 起步列，不改变 incumbent 的定义，也不会把外包任务伪造成内部机器列。

外包部分单独记录。`InitialColumnBundle` 现在保存最终 seed 的外包任务集合、baseline 总量和外包成本 `G(B)`；`Tree` 初始化 incumbent 外包变量时直接使用 bundle 中的外包任务集合，而不是重新从 seed 对象里读。这样后续如果 seed 的外包信息需要独立传递，也不会丢失。结果输出方面，`BPCResultWriter` 的 `.outsourcing.csv` 现在包含一行 `TOTAL`，记录最终外包 baseline 总量和 `G(B)` 成本，同时继续逐个输出被外包任务的 baseline 贡献。

验证结果：局部 `javac` 编译通过；`HEU.SmallBPCBatchTest` 继续通过 8 个随机小算例与 ArcFlow 对拍，tariff 分支诊断例也通过。当前剩余风险主要是历史 best 数量如果后续在大规模 ALNS 中过多，root 初始列可能变大；如果出现这个问题，再加历史池大小限制或只保留最近若干个历史 best。

## 外包集合是否可以后续转成列

当前实现中，历史 best 解是按完整 `Solution` 副本记录的，因此每个历史解里的外包任务集合、baseline 总量和 `G(B)` 成本都在 `Solution` 对象里保留了。但当前 `InitialColumnBuilder` 只把这些历史解中的真实机器序列转成 root 初始列；外包集合只在最终 seed incumbent 层面记录，没有进入独立的外包列池。

如果后续要把外包也列化，思路上可以做，但不能简单照搬机器列的“多列可同时选择并累加成本”语义。真实机器列的成本是可加的，每台机器选一条序列即可；外包成本是公共函数 `G(B(S))`，其中 `S` 是整个外包任务集合。如果把多个外包子集列同时选中并把成本相加，就会变成 `sum G(B(S_r))`，这通常不等于 `G(sum B(S_r))`，除非 `G` 是线性的。

因此外包列更合理的建模方式是：每条外包列代表一个完整外包集合，主问题额外加“最多/恰好选择一条外包集合列”的约束，并用这条列覆盖其中的外包任务，成本直接取 `G(B(S))`。空外包集合也可以作为一条成本为 `G(0)` 的列。在这种建模下，历史 best 解中的外包集合可以像机器序列一样作为初始外包列加入主问题：列的覆盖系数是集合内任务为 1，列成本是该集合的 `G(B)`。

这相当于把当前 `y_j + z_s/q_s` 的外包建模替换或并行成“外包集合列”建模，需要同步调整覆盖约束、外包列选择约束、外包 pricing/补列逻辑和分支逻辑。当前阶段先不改代码；结论是数据已经足够记录，后续可以列化，但不能把外包子集列当成普通机器列那样任意多选。

## 2026-05-24：ALNS 到 VND 的脏机局部刷新

这次处理的是初始解启发式链路里一个已经确认的结构性低效点。原来的 `EngineALNS -> EngineVND` 交接方式是：ALNS 的 `removeJobs()/insertJob()` 只重算被改机器的一维 forward/backward 函数和单机成本，但一旦进入 `EngineVND`，构造函数仍然会对所有机器统一调用一次 `updateInformationM(m)`，把二维函数、duration 和硬时间窗摘要整机重刷。这样即使 ALNS 本轮只改了少数几台机器，进入 VND 前也会把所有机器的缓存全量重建一遍。

这次改成按脏机刷新。`Solution` 新增三类状态：`dirtyMachines` 记录 ALNS 真正改动过的机器；`sharedOneDimFunctions` 和 `sharedMachineState` 记录 `Solution.copy()` 之后哪些缓存仍然和源解共享。原因是 ALNS 当前每轮都是 `tmpSol = current.copy()`，而旧的 `copy()` 会直接复用机器级函数缓存。如果只改 `EngineVND` 的入口而不处理共享状态，那么在 `tmpSol` 上刷新某台机器的二维函数、duration 或硬窗摘要时，可能会把 `current` 里对应机器的缓存一起改掉，尤其是在 VND 接受劣解失败、`tmpSol` 被丢弃时会留下隐性污染。

现在的处理方式是：`copy()` 仍然不深拷整套机器缓存，但会把二维数组和时间窗摘要的最外层数组 clone 一层，并把各机器标记为“共享缓存”。随后，`calCost(m)` 在需要重算某台机器的一维函数时，先通过 `detachSharedOneDimFunctions(m)` 断开与源解共享的一维函数列表；`updateInformationM(m)` 和 `updateInformationM(m, costM)` 在真正重建机器缓存前，会先断开该机器共享的二维函数/摘要缓存，再做更新。这样保留了 copy 的轻量性，同时保证后续局部刷新不会污染源解。

在 ALNS 这条链上，`removeJobs()` 和 `insertJob()` 现在在重算完单机成本后把对应机器标成 dirty；`setInitSolution()` 和 `setSequence()` 则在整体改序列后直接把所有机器标成 dirty。`EngineVND` 构造函数不再无条件循环 `updateInformationM(m)`，而是改为调用 `refreshDirtyMachineInformation()`，只对脏机器做完整缓存重建。对于 ALNS 中被删空的机器，`calCost(m)` 现在会显式清空一维函数列表，避免旧实现里“机器序列已经空了，但 `fFunctions/bFunctions` 还残留旧内容，必须靠全机 `updateInformationM` 兜底”的隐式依赖。

这次没有继续细拆到“脏机内部再按位置局部重建二维函数”。当前先把收益最大的那一层做对：保留 VND 内部各类 move commit 现有的按受影响机器更新方式，只消掉 ALNS repair 完成后进入 VND 前那次最粗的全机重刷。

验证做了两步。第一步是定向编译：用本机现有 `src` classpath，加一个临时 `IloException` stub，重新编译了 `Solution.java`、`EngineALNS.java`、`EngineVND.java` 和 `Move.java`，编译通过。第二步是运行时验证：`OutsourcingMoveConsistencyTest` 通过，结果为 `checked=10974`；另外单独写了一个小探针，实际跑通 `setInitSolution -> ALNS -> VND` 主链，能正常完成并给出改进后的 best cost。当前剩余风险主要有两个：一是命令行环境仍然没有完整 CPLEX classpath，因此这次没有做带精确模型对拍的全链回归；二是 `refreshDirtyMachineInformation()` 目前仍按“脏机器整机更新”实现，后续如果这里继续成为热点，再考虑把 `updateInformationM(m)` 本身继续拆成更细的局部刷新。
