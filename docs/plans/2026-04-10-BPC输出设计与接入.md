# BPC 输出设计与接入

这次调整的目标很明确，不是简单多打几行 `println`，而是把当前 `TWETBPC` 骨架里的过程输出和最终结果输出先搭成一套能复用的结构。前面的旧参考代码里其实已经有比较完整的输出思路，只是分散在 `Tree`、`Configure.PrintTimeInformation()`、`Utility.OutputSummaryResult()` 和若干调试函数里，没有整理成统一模块。当前新框架如果继续沿那种写法堆输出，后面 pricing、cut、branch 真接进来以后会很乱，所以这次先把结构收住。

这次采用的思路是把输出拆成三层。第一层是过程事件接口，也就是 `BPCTraceSink`。`Tree` 和 `PC` 只负责在关键节点上报事件，例如初始列准备完成、节点开始处理、主问题求完、加列、加 cut、incumbent 更新、分支发生、求解结束。这样求解主流程不直接依赖具体输出格式。第二层是统计和追踪层，主要是 `BPCTraceSummary`。它负责累计节点数、root bound、pricing/cut 轮数、新增列和 cuts 数、分支次数、incumbent 更新次数、队列峰值，并把节点级摘要和过程日志都收起来。第三层是展示层，分别是 `BPCConsoleReporter` 和 `BPCResultWriter`。前者负责运行时 console 输出，后者负责把最终结果写到指定目录。

这次对齐旧参考代码时，保留的关键信息主要有三类。第一类是节点级过程信息，旧 `Tree` 在每个节点后都会输出当前节点目标值、是否得到可行整数解、当前 best cost、gap、队列大小、列池和 cut 池规模。现在这些字段已经收进 `BPCNodeRecord`，并写到 `.nodes.csv`。第二类是最终 summary，旧代码会记录初始解、root node lower bound、最终最优值、最终 lower bound、总时间、节点数和 cut 数。现在这些字段已经写进 summary markdown。第三类是过程日志，旧代码既有 console 输出，也会单独写 log。现在同目录下会额外生成一份 `.log`，保存节点处理、加列、加 cut、分支和最终总结。

为了让大部分东西后续还能复用，这次没有把输出逻辑硬写死在 `Tree` 或 `PC` 里，而是把格式化也单独提到 `BPCOutputFormatters`。这样 console 和文件导出共用一套格式化逻辑，后面如果你要把模型法也接一部分共用输出风格，改动范围会比较小。当前 `TWETBPCContext` 里已经把 `traceSummary`、`consoleReporter` 和 `traceSink` 统一装配好，`TWETBPCSolver.solve()` 在求解结束后会负责打印最终总结，并在配置允许时自动把结果写到 `results/<methodName>/` 下。

在信息密度上，这一轮又往前补了一层，尽量把当前骨架已经拿得到的真数据都接进来了。除了总节点数、root bound、最终上下界之外，现在还会额外记录整数节点数、被 incumbent 剪枝的节点数、无法继续分支而关闭的节点数、求解结束时剩余队列大小、各个 pricing engine 的调用次数/成功次数/累计新增列数、各个 cut generator 的调用次数/成功次数/累计新增 cut 数，以及各个 brancher 的尝试次数和成功次数。换句话说，旧版那种“节点表 + 过程日志 + 汇总表”的密度已经往现在这套结构里迁了一大块，而且这些都是真实由当前骨架产生的，不是占位字段。

接不了的部分也已经在对应代码里写了 `TODO`。核心是 `LP.solveRelaxation()` 和 `PC.solve(...)` 里那几类统计：active column values、dual、reduced cost、每轮 cut 的违反度、真正进入模型的 branch/cut 约束条数。这些不是忘了接，而是当前骨架还没有真实的 RMP、pricing 和 cut separation 数学内核，所以现在只能把接口和提示位置先留好。等后面把精确主问题和子问题真正填进去，这批字段可以顺着现在这套输出框架继续往里补，不需要重写输出层。

当前这套输出已经覆盖了骨架阶段最需要的内容，包括 console 过程输出、最终 summary、节点级 csv、incumbent columns 和当前 cut 池导出。还没有做的是更细粒度的 dual、active column value、具体 cut violation 这类调试信息。这部分旧参考代码里主要靠 `LP.DebugValue()` 和 `DebugDual()` 做，等后面真正把 RMP、pricing、cuts 填实以后，再把它们作为更细一级的调试接口接进来会更合适。
